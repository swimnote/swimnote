[최종 통합 지시서 - 슈퍼관리자용 DB / 수영장 운영용 DB 이원화 + Supabase 실제 연결 + 용량 분리 관리 + 앱 기능 보완]

지금부터 스윔노트 플랫폼의 데이터 구조를
“슈퍼관리자 운영 데이터”와 “수영장 실제 운영 데이터”로 완전히 분리하고,
Replit 서버에서 이를 실제로 연결/관리 가능하게 정리한다.

이번 작업은 단순 DB 연결이 아니다.
아래 4가지를 한 번에 끝내는 작업이다.

1. 슈퍼관리자용 DB와 수영장 운영용 DB를 실제로 분리 등록
2. 각 DB를 다른 서버/다른 데이터 영역처럼 독립적으로 관리 가능하게 구성
3. 용량 관리도 각각 따로 하게 만들고 슈퍼관리자가 둘 다 확인 가능하게 구현
4. 현재 앱 구조에 이 기능이 없으면 먼저 앱 기능/화면/API/DB 구조를 구현하고 그 다음 연결

중요:
- “연결된 척” 금지
- DB 이름만 나눈 척 금지
- 실제로 어떤 기능이 어느 DB를 보는지 명확해야 함
- 슈퍼관리자와 수영장 운영 데이터를 절대 섞지 말 것
- 구현이 안 되어 있으면 앱 기능부터 먼저 만들고 연결하라
- 거짓 완료 보고 절대 금지

==================================================
0. 최종 목표
==================================================

최종적으로 아래 구조가 되어야 한다.

A. 슈퍼관리자용 데이터 영역
- 플랫폼 운영
- 수영장 목록
- 구독 상태
- 결제 상태
- 매출/정산
- 위험도
- 삭제 예약
- 감사 로그
- 백업 메타
- 각 운영 DB 서버 용량 요약

B. 수영장 운영용 데이터 영역
- 회원
- 선생님
- 학부모
- 반
- 반배정
- 출결
- 일지
- 보강
- 레벨
- 공지
- 사진/앨범 메타
- 수영장 내부 운영 데이터 전부
- 해당 수영장 DB/스토리지 사용량

이 둘이 구조적으로 분리되어 있어야 하며,
슈퍼관리자는 각 운영 DB의 상태/용량/위험을 볼 수 있어야 한다.

==================================================
1. DB 분리 방식
==================================================

우선 Supabase에서 아래 중 실현 가능한 가장 안정적인 방식으로 구현한다.

우선순위 1:
- 서로 다른 2개 DB/서버처럼 관리 가능한 구조
- 예: 슈퍼관리자용 DB / 수영장운영용 DB를 분리 연결

우선순위 2:
- 물리 분리가 당장 어렵다면
  Supabase 1개 프로젝트 안에서 스키마 2개로 명확히 분리
  (super_admin / pool_ops)
- 단, 앱/서버/API/용량관리에서 각각 다른 데이터 서버처럼 관리 가능해야 함

즉:
- 리플릿은 실제 구현 가능 범위에서
  가장 안전하고 현실적인 방식으로 분리하되
  “분리된 것처럼 보이는 척” 하지 말고
  실제 운영상 독립적으로 관리 가능한 구조를 만들어야 한다

슈퍼관리자용 영역에는 아래만 둔다.

필수 테이블/영역 예시:
- operators
- subscriptions
- revenue_logs
- settlement_logs
- audit_logs
- risk_summary
- deletion_queue
- backup_meta
- policy_consents
- db_server_status
- storage_server_status

필수 포함 데이터:
- 수영장 ID
- 수영장명
- 소유자/운영자
- 구독 플랜
- 결제 상태
- 환불 상태
- 읽기모드 상태
- 삭제 예정 상태
- 매출/수수료/실수익
- 각 수영장 운영 DB 사용량
- 각 수영장 파일 저장소 사용량
- 위험 경고 상태

중요:
- 슈퍼관리자는 수업 내부 상세 데이터 전체를 직접 들고 있지 않음
- 요약/정산/운영관리/리스크 중심

==================================================
4. 두 영역 연결 키
==================================================

슈퍼관리자 영역과 수영장 운영 영역은
최소 키만 공유한다.

공통 키 예시:
- pool_id
- operator_id
- subscription_id

중요:
- 슈퍼관리자는 pool_id 기준으로 수영장 요약을 본다
- 상세 운영 데이터는 수영장 운영 DB/스키마에서만 조회
- 양쪽 데이터가 한 쿼리에서 무분별하게 섞이지 않게 할 것

==================================================
5. 각각 다른 서버/DB처럼 관리되게 만들 것
==================================================

리플릿은 아래를 실제로 가능하게 만들어야 한다.

반드시 구현:
- 슈퍼관리자용 DB 연결 정보
- 수영장 운영용 DB 연결 정보
- 서버 코드에서 어떤 기능이 어느 DB를 보는지 명확히 분리
- 데이터 조회/저장 시 잘못된 DB를 보지 않도록 서비스 레이어 정리

가능한 구현 방식:
- DATABASE_URL_SUPER
- DATABASE_URL_POOL
또는
- 동일 프로젝트 내 super_admin / pool_ops 스키마 분리 후 명시적 schema 접근

중요:
- 어떤 API가 어떤 DB/스키마를 보는지 표로 보고할 것
- 연결만 해놓고 실제 코드가 한쪽 DB만 보고 있으면 실패

==================================================
6. 용량 관리 분리
==================================================

이 부분이 핵심이다.

용량은 아래처럼 반드시 분리 관리되어야 한다.

A. 슈퍼관리자 운영 데이터 용량
- 구독/정산/로그/감사/백업 메타 용량

B. 수영장 운영 데이터 용량
- 회원/출결/일지/반/보강/레벨 등 DB 용량
- 사진/앨범 메타 용량
- 실제 파일 저장소(R2 또는 기타) 사용량

필수 기능:
- 슈퍼관리자가 각 수영장의 운영 DB 사용량 확인 가능
- 슈퍼관리자가 슈퍼관리자 DB 자체 사용량도 확인 가능
- 둘을 따로 보이게 할 것
- 합산표시가 아니라 분리표시
- storage 위험 경고도 각각 가능하게

==================================================
7. 슈퍼관리자 화면 구현/보완
==================================================

슈퍼관리자 화면에 아래가 실제로 보여야 한다.

필수 항목:
- 플랫폼 운영 DB 사용량
- 수영장 운영 DB 사용량
- 수영장별 저장소 사용량
- 위험 임계치(예: 80% / 90% / 100%)
- 어떤 서버/DB가 위험한지 표시
- 새로고침/재조회 기능
- 마지막 확인 시각

예시 화면:
- DB 서버 상태
- 파일 저장소 상태
- pool_id별 사용량
- 전체 합계
- 위험 배지

중요:
- 지금 화면이 없으면 앱에 먼저 만들어라
- 단순 숫자만 던지지 말고 관리자가 이해 가능한 구조로 표시

==================================================
8. 구현이 안 되어 있으면 먼저 만들어야 할 기능
==================================================

만약 현재 앱/서버에 아래 기능이 없다면
연결 전에 먼저 구현해야 한다.

필수 보완 기능:
1. super_admin DB 상태 조회 API
2. pool_ops DB 상태 조회 API
3. 수영장별 운영 DB 용량 계산 API
4. 수영장별 파일 저장소 용량 계산 API
5. 슈퍼관리자 대시보드/리스크 화면 표시 UI
6. DB/스토리지 상태를 저장하는 테이블 또는 계산 로직

중요:
- “나중에 연결 예정” 금지
- 기능이 없으면 먼저 만들고 연결하라

==================================================
9. Replit에서 반드시 확인할 것
==================================================

다음 항목을 실제로 검증하라.

1. 현재 연결된 Supabase 프로젝트/DB 확인
2. 슈퍼관리자용 연결 정보
3. 수영장 운영용 연결 정보
4. 실제 쿼리 실행 테스트 (select now)
5. 각 테이블이 올바른 DB/스키마에 있는지 확인
6. 잘못된 DB에 저장되는 테이블이 없는지 확인

중요:
- 실제 쿼리 결과로만 판단
- 추측 금지

==================================================
10. 기존 코드와 합쳐서 정리할 것
==================================================

기존에 이미 만든 아래 구조와 반드시 충돌 없이 합쳐라.

- 구독 플랜 구조
- 무료/유료/엔터프라이즈 플랜
- revenue_logs / settlement_logs
- 읽기모드 / 결제실패 / 삭제예약
- 미배정 / 미연결 분리
- 반배정 / 반 제외 / 변경 이력
- 사진/용량 정책
- 슈퍼관리자 리스크 화면

즉:
기존 시스템을 깨지 말고
DB 분리 구조 위에 자연스럽게 얹어야 한다.

==================================================
11. 실연동 테스트 필수
==================================================

반드시 아래 테스트를 실행하라.

A. 수영장 관리자에서 회원 등록
기대 결과:
- pool_ops 영역에 저장
- super_admin에는 상세 회원 데이터가 직접 중복 저장되지 않음

B. 수영장 구독 변경/결제
기대 결과:
- super_admin 영역에 subscription / revenue_logs 반영
- pool_ops에는 운영 제한값(member_limit, storage_limit 등) 반영

C. 수영장 운영 데이터 증가
기대 결과:
- 운영 DB 사용량 증가
- 슈퍼관리자에서 해당 용량 확인 가능

D. 결제 실패 → 읽기모드
기대 결과:
- super_admin 상태 변경
- pool_ops 운영 제한 반영

E. 파일/사진 저장소 사용량
기대 결과:
- pool별 사용량 계산 가능
- 슈퍼관리자 화면에서 확인 가능

==================================================
12. 의심 항목 전수 점검
==================================================

조금이라도 의심되면 바로 확인:
- 사실상 한 DB만 쓰고 있는지
- super_admin / pool_ops 분리된 척만 하는지
- revenue_logs가 운영 DB에 들어가는지
- students/attendance/class_journals가 슈퍼관리자 DB에 들어가는지
- 용량 계산이 총합만 있고 분리 안 되어 있는지
- 파일 스토리지와 DB 사용량이 섞여 있는지
- 화면은 있는데 API가 없는지
- API는 있는데 실제 쿼리가 wrong schema를 보는지

==================================================
13. 완료 기준
==================================================

다음 조건을 모두 만족해야 완료다.

- 슈퍼관리자용 / 수영장운영용 데이터 영역 분리 완료
- 각각 다른 DB/서버처럼 관리 가능한 구조 완료
- 어떤 기능이 어느 DB/스키마를 보는지 명확
- 용량 관리 분리 완료
- 슈퍼관리자가 각 DB/스토리지 사용량 확인 가능
- 구현 안 된 기능은 먼저 앱에 구현 후 연결 완료
- 기존 결제/정산/운영 기능과 충돌 없음

==================================================
14. 완료 보고 형식
==================================================

반드시 아래 형식으로 보고하라.

1. 실제 사용한 분리 방식
   - 서로 다른 DB 2개 / 같은 프로젝트 내 스키마 2개 중 무엇인지
2. 연결된 Supabase 프로젝트 정보
3. 생성한 DB/스키마 목록
4. super_admin에 들어간 테이블 목록
5. pool_ops에 들어간 테이블 목록
6. 어떤 API가 어느 DB/스키마를 보는지 표
7. 슈퍼관리자 화면에 추가/보완한 용량관리 기능
8. 실제 쿼리 테스트 결과
9. 실데이터 저장/조회 테스트 결과
10. 아직 분리 안 된 잔존 테이블/기능
11. 내가 Supabase에서 직접 확인해야 할 위치
12. 지금 구조가 안전한지 한 줄 요약

중요:
- “분리 완료” 한 줄 보고 금지
- 실제 연결/쿼리/저장/조회 결과를 적어라
- 미확인 항목은 미확인이라고 적어라
