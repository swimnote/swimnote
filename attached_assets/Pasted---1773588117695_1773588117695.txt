수영일지 시스템을 운영형 구조로 전면 구현해줘.
이번 작업은 기존 일지 기능을 단순 저장형이 아니라
"반 공통 일지 + 학생별 추가 일지 + 수정/삭제 감사기록" 구조로 다시 만드는 작업이다.

핵심 목표:
1. 선생님은 반 기준 공통 일지를 1번만 작성
2. 필요할 때만 학생별 추가 일지를 작성
3. 학부모는 자기 자녀 기준으로 공통 일지 + 개별 추가 일지를 합쳐서 읽게 해줘
4. 선생님은 본인이 작성한 일지를 수정/삭제 가능하게 해줘
5. 관리자는 수정되거나 삭제된 일지의 이전 내용까지 확인 가능하게 해줘
6. 관리자 화면에서 수정/삭제된 일지는 글박스 왼쪽 상단에 "수정" 또는 "삭제" 표시가 남게 해줘
7. 실제 DB 저장 구조로 구현하고 앱 재실행 후에도 유지되게 해줘

핵심 운영 구조:
- 일지 작성 단위는 "반 기준"
- 선생님은 오늘 수업한 반에 대해 공통 일지 1건 작성
- 필요하면 해당 반 학생 중 일부에게만 개별 추가 일지를 작성
- 학부모는 자기 자녀가 속한 반의 공통 일지를 보고
- 자기 자녀에게만 추가 일지가 있으면 그 내용을 이어서 보게 해줘
- 학부모 입장에서는 "우리 아이에게 온 일지"처럼 자연스럽게 읽히게 구성

반드시 구현할 데이터 구조:

1. 공통 일지 테이블
table: class_diaries
필드:
- id
- class_group_id
- teacher_id
- teacher_name
- swimming_pool_id
- lesson_date
- common_content
- is_edited boolean default false
- edited_at nullable
- edited_by nullable
- is_deleted boolean default false
- deleted_at nullable
- deleted_by nullable
- created_at
- updated_at

2. 학생별 추가 일지 테이블
table: class_diary_student_notes
필드:
- id
- diary_id
- student_id
- note_content
- is_edited boolean default false
- edited_at nullable
- edited_by nullable
- is_deleted boolean default false
- deleted_at nullable
- deleted_by nullable
- created_at
- updated_at

3. 일지 감사기록 테이블
table: class_diary_audit_logs
필드:
- id
- diary_id nullable
- student_note_id nullable
- target_type ('common' | 'student_note')
- action_type ('create' | 'update' | 'delete')
- before_content nullable
- after_content nullable
- actor_id
- actor_name
- actor_role
- swimming_pool_id
- created_at

반드시 구현할 권한 규칙:
1. 선생님은 자기 반(class_group_id) 일지만 작성 가능
2. 선생님은 본인이 작성한 공통 일지 수정 가능
3. 선생님은 본인이 작성한 학생별 추가 일지 수정 가능
4. 선생님은 본인이 작성한 공통 일지 삭제 가능
5. 선생님은 본인이 작성한 학생별 추가 일지 삭제 가능
6. 관리자는 자기 수영장 내 모든 일지 조회 가능
7. 관리자는 수정/삭제 감사기록 조회 가능
8. 학부모는 자기 자녀 기준으로만 조회 가능
9. 학부모는 절대 수정/삭제 불가

삭제 정책:
- 실제 hard delete 하지 말고 soft delete 로 처리
- is_deleted = true 로 저장
- 관리자만 삭제된 일지도 볼 수 있게 해줘
- 선생님 화면에서는 삭제된 일지는 기본적으로 숨겨줘
- 학부모 화면에서는 삭제된 일지는 절대 보이지 않게 해줘

수정 정책:
- 선생님은 수정 가능
- 수정 시 class_diaries 또는 class_diary_student_notes 의 본문을 업데이트
- 수정 전 before_content 와 수정 후 after_content 를 class_diary_audit_logs 에 저장
- 수정 후 is_edited = true, edited_at, edited_by 반영
- 학부모 화면에는 "수정됨" 표시만 보이게 해줘
- 관리자 화면에는 수정 전/후 기록도 볼 수 있게 해줘

작성 흐름:
1. 선생님이 반 선택
2. 공통 일지 작성
3. 학생별 추가 일지 섹션에서 필요한 학생만 선택
4. 해당 학생에게만 추가 코멘트 작성
5. 저장하면 공통 일지 1건 + 학생별 추가 일지 N건 저장
6. 감사기록에도 create 로그 저장

학부모 조회 구조:
학부모가 자기 자녀 화면에서 일지를 조회하면
서버는 아래 구조로 응답하게 해줘:

- class diary common_content
- 해당 자녀 student_id 에 매칭되는 student_note
- student_note 가 없으면 null
- 학부모 화면에서는 공통 일지를 기본으로 보여주고
- student_note 가 있으면 아래에 "추가 안내" 섹션으로 붙여서 보여줘
- "반 전체 공통 일지" 같은 표현은 쓰지 말고
- 학부모가 자기 자녀에게 온 일지처럼 읽히게 문구를 구성

선생님 일지 작성 UI 요구:
1. 일지 작성 단위는 반 기준
2. 화면 상단에 반 이름 / 날짜 표시
3. 공통 일지 작성 영역
4. 학생별 추가 일지 영역
5. 학생 리스트에서 필요한 학생만 눌러서 추가 작성 가능
6. 공통 일지와 학생별 추가 일지는 분리된 카드로 보여줘
7. 저장 버튼은 크게 하단 고정 또는 눈에 잘 띄게 배치

일지 작성 효율화 기능:
1. 공통 일지는 선택형 문장 템플릿 + 직접 작성 둘 다 가능하게 해줘
2. 관리자가 미리 템플릿 문장을 등록할 수 있는 구조를 만들어줘
3. 선생님은 문장을 선택해서 공통 일지를 빠르게 완성할 수 있게 해줘
4. 필요하면 직접 입력도 가능하게 해줘
5. 학생별 추가 일지도 짧게 추가할 수 있게 해줘

관리자 조회 UI 요구:

- 정상 일지
- 수정된 일지
- 삭제된 일지
- 학생별 추가 일지 포함 여부

관리자 화면 카드 규칙:
1. 일지 카드 글박스 왼쪽 상단에 작은 상태 배지 표시
   - 수정됨
   - 삭제됨
2. 수정된 일지는 "수정" 배지
3. 삭제된 일지는 "삭제" 배지
4. 카드 안에는 현재 내용 표시
5. 필요하면 "변경기록 보기" 버튼으로 감사기록 모달 또는 상세 화면 열기
6. 변경기록 보기에는 아래 항목 표시
   - 수정 전 내용
   - 수정 후 내용
   - 삭제 전 내용
   - 수정/삭제한 사람
   - 시간

학부모 화면 UI 요구:
1. 제목은 "오늘 수업 내용" 또는 날짜 기준 일지 제목
2. 공통 일지 표시
3. student_note 가 있으면 "추가 안내" 섹션 표시
4. 수정된 경우 "수정됨" 작은 표시만 보여줘
5. 삭제된 일지는 안 보여줌
6. 절대 "반 전체 일지" 같은 표현 쓰지 말 것

API 요구:
다음 API를 구현해줘.

공통 일지
- POST /diaries
- GET /diaries?class_group_id=&lesson_date=
- PUT /diaries/:id
- DELETE /diaries/:id

학생별 추가 일지
- POST /diaries/:id/student-notes
- PUT /diaries/student-notes/:noteId
- DELETE /diaries/student-notes/:noteId

관리자 감사기록
- GET /diaries/:id/audit-logs

학부모 조회
- GET /parent/diaries
- GET /parent/students/:studentId/diaries

조회 응답 규칙:
- teacher/admin 조회는 common + student notes 모두 가능
- parent 조회는 common + 자기 자녀 student note 만 가능
- 삭제된 데이터는 admin 전용 조회만 가능
- teacher 조회에서는 본인 수업 기준만 가능

DB/ORM 요구:
1. PostgreSQL 테이블 실제 생성
2. Drizzle ORM 스키마 추가
3. schema index export 반영
4. 기존 diary 라우트가 있으면 새 구조로 교체 또는 확장

감사기록 저장 규칙:
1. create 시 action_type='create'
2. update 시 before_content, after_content 저장
3. delete 시 before_content 에 삭제 전 내용 저장
4. actor_id / actor_name / actor_role 저장
5. common 일지와 student note 둘 다 로그 남김

추가 요구:
1. 수정/삭제 버튼은 선생님 본인 작성 일지에만 노출
2. 관리자에게는 수정/삭제 버튼 대신 감사기록 조회 버튼 노출
3. 삭제는 확인 모달 후 실행
4. 수정은 별도 편집 모드 또는 모달로 처리
5. 카드 크기와 레이아웃은 기존 공통 디자인 시스템에 맞춰 통일
6. 상단 고정 / 하단 목록 스크롤 구조 유지

연동 요구:
1. 오늘 스케쥴 탭에서 일지 작성 버튼 누르면 해당 반/날짜 기준으로 이동
2. 일지 저장 완료 시 오늘 스케쥴 카드 상태가 즉시 갱신
3. 학부모 앱에서 저장 후 바로 조회 가능
4. 관리자 앱에서도 즉시 확인 가능
5. 출결/반/회원 구조와 자연스럽게 연결

최종 점검:
1. 선생님이 반 공통 일지를 저장하면 DB에 실제 INSERT 되는지
2. 선생님이 학생별 추가 일지를 저장하면 DB에 실제 INSERT 되는지
3. 선생님이 수정하면 audit log 가 남는지
4. 선생님이 삭제하면 soft delete 처리되는지
5. 관리자가 수정/삭제 일지와 이전 내용을 볼 수 있는지
6. 학부모가 자기 자녀 기준으로 common + student_note 를 자연스럽게 보는지
7. 앱 재실행 후에도 일지가 유지되는지
8. 권한 분리가 API 레벨에서 안전하게 되는지

- 학부모는 자기 아이에게 온 일지처럼 읽음
- 선생님은 수정/삭제 가능
- 관리자는 수정/삭제 이력까지 확인 가능
- 운영 효율성과 기록 안정성을 동시에 만족하는 구조로 구현
