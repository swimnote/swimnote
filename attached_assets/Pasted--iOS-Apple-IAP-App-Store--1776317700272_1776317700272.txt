[스윔노트 iOS 심사 리젝 해결 — Apple 로그인 + IAP 근본 해결 지시문]

지금부터 할 일은 “대충 막는 수정”이 아니라, App Store 심사에서 한 번에 통과할 수 있게 Apple 로그인과 인앱구독(IAP) 문제를 근본적으로 해결하는 것이다.

절대 금지:
- 프론트 문구만 바꾸고 끝내기
- 에러를 숨기거나 뭉뚱그린 메시지로 덮기
- 추측으로 수정하고 테스트 없이 종료하기
- 기존 코드가 꼬여 있으면 억지로 이어붙이기
- “아마 될 것 같습니다” 같은 보고 금지

원칙:
- 문제되는 구현이 있으면 삭제하고 새로 깔끔하게 다시 만들어도 된다
- 목표는 “작동하는 코드”가 아니라 “심사 통과 가능한 안정 구조”
- 수정 후 반드시 실기기/시뮬레이터 기준 재현 테스트와 증빙 스크린샷까지 포함한 보고서를 제출한다

현재 Apple 리젝 이슈:
1) Sign in with Apple 실패
- Apple 로그인 UI는 뜨지만 최종적으로 “Sign-Up Not Completed” 발생
- 심사 기기: iPad Air 11-inch (M3), iPhone 17 Pro Max
- 신규 설치와 업데이트 설치 모두에서 정상 완료되어야 함

2) IAP 심사 제출 누락
- 앱 내 구독 참조는 있으나 관련 IAP 상품이 심사 제출 상태가 아님
- App Review screenshot 포함 제출 필요

3) IAP 구매 에러
- 구독관리 화면에서 “구독 상품 로드 실패”
- 상품 fetch 또는 StoreKit/RevenueCat/App Store Connect 설정 중 하나 이상이 근본적으로 깨져 있음

해야 할 작업 범위:
A. Apple 로그인 근본 해결
B. IAP 상품 로드/구매 실패 근본 해결
C. App Store Connect 제출 전 체크리스트 정리
D. 최종 검증 보고서 + 스크린샷 첨부

━━━━━━━━━━━━━━━━━━
1단계 — Apple 로그인 근본 원인 추적
━━━━━━━━━━━━━━━━━━

다음 순서로 진행:
1. Apple 로그인 시작부터 완료까지 전체 흐름을 코드 기준으로 추적
   - 클라이언트 Apple auth 요청
   - Apple credential 응답 수신
   - 서버 전달 payload
   - 서버 계정 조회/생성/업데이트
   - 세션 생성
   - 온보딩/라우팅

2. “Sign-Up Not Completed”가 정확히 어디서 발생하는지 찾기
   - 프론트 catch에서만 뭉개지고 있는지
   - 서버 DB insert/update 실패인지
   - Apple 최초 로그인/재로그인 분기 문제인지
   - email/name 최초 1회 제공 특성 처리 누락인지
   - nullable 처리 안 된 컬럼 때문에 실패하는지
   - unique 충돌/apple_id 매핑 꼬임인지
   - swimming_pool_id / phone / onboarding 관련 필수값 문제인지

3. 로그 강화
   - Apple 로그인 관련 모든 단계에 구조화 로그 추가
   - “실패”만 찍지 말고 어느 단계에서 어떤 필드 때문에 실패했는지 찍기
   - 단, 민감정보 원문은 마스킹 처리
   - 프론트에도 개발 빌드에서만 상세 에러를 볼 수 있게 하고, 프로덕션 사용자 메시지는 짧게 유지

4. 해결 방식
   - 기존 로직이 꼬였으면 부분 패치하지 말고 Apple 로그인 신규가입/기존가입 플로우를 다시 설계
   - 기준:
     a. Apple의 sub 값을 유일 식별자로 안전하게 저장
     b. 최초 1회만 오는 email/name이 없어도 재로그인 정상 처리
     c. 신규 계정 생성 시 필수값이 비어도 가입이 막히지 않게 설계
     d. 이후 온보딩에서 필요한 값은 별도 단계에서 입력받도록 분리
     e. 이미 존재하는 일반계정/카카오계정과의 충돌 정책 명확화
     f. iPad/iPhone 모두 모달/레이아웃/안전영역 문제 없이 동작

5. 꼭 해야 할 테스트
   - 완전 신규 설치 후 Apple 로그인 성공
   - 기존 버전 설치 후 업데이트 설치 후 Apple 로그인 성공
   - 최초 로그인 성공
   - 두 번째 로그인(재로그인) 성공
   - Apple 계정의 email 숨기기 옵션 사용 시도
   - iPad / iPhone 각각 테스트
   - 실패 시 어떤 단계에서 실패했는지 로그로 확인 가능해야 함

2단계 — IAP 구독 로드/구매 실패 근본 해결
━━━━━━━━━━━━━━━━━━

다음 순서로 진행:
1. 현재 구독 구조 전수 점검
   - 앱 코드에 쓰는 product id 목록
   - RevenueCat entitlement / offering / package / product mapping
   - StoreKit fetch 코드
   - 구독관리 화면의 상품 로드 시점
   - 실패 시 에러 처리 방식

2. “구독 상품 로드 실패” 원인 분해
   - product id 오타/불일치
   - iOS 번들/앱 ID 불일치
   - RevenueCat 프로젝트 연결 오류
   - App Store Connect 상품 미생성 또는 Draft 상태
   - Paid Apps Agreement 미수락
   - Sandbox 계정 미설정
   - iPad에서만 상품 fetch가 실패하는 UI/초기화 타이밍 문제
   - 네트워크 요청 실패인데 프론트에서 일반 에러로 뭉개는 문제

3. 해결 방식
   - 상품 로드 로직이 불안정하면 삭제 후 다시 구성
   - 기준:
     a. 화면 진입 시 offerings/product fetch 안정적으로 수행
     b. 로딩/성공/빈값/실패 상태를 분리
     c. 실패 시 실제 원인을 로그에 남김
     d. fetch 실패와 purchase 실패를 서로 다른 에러로 처리
     e. iPhone/iPad 동일하게 동작
     f. Apple 심사 sandbox 환경에서도 로드 및 구매 가능한 구조

4. 반드시 확인할 콘솔/설정
   - App Store Connect 구독 상품 상태
   - product id 정확히 일치 여부
   - RevenueCat entitlement 연결
   - Paid Apps Agreement 활성 여부
   - Banking/Tax/Business 섹션 누락 여부
   - App Review screenshot 준비 여부
   - 첫 제출 구독이면 새 바이너리와 함께 심사 제출 준비

5. 구매 테스트
   - sandbox 계정으로 실제 구독 버튼 탭
   - 결제 시트 표시
   - 결제 성공
   - entitlement 반영
   - UI 즉시 갱신
   - 재실행 후 권한 유지 확인

다음 기준으로 정리:
- Apple 로그인 관련 파일/함수 흐름 단순화
- IAP 관련 파일/함수 흐름 단순화
- 임시 분기 / 죽은 코드 / 중복 분기 제거

- iPad 레이아웃 최적화
