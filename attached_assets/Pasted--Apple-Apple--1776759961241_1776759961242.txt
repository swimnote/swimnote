[스윔노트 Apple 로그인 간헐 실패 — 근본 원인 특정 지시문]

지금부터는 빌드만 반복하지 말고, Apple 로그인 실패 위치를 정확히 특정하는 작업만 하세요.
임시 패치 금지, 추측 금지, “다시 빌드해보자” 금지.
목표는 수정이 아니라 “어디서 죽는지 증명”입니다.

현재 현상:
- Apple 로그인 버튼 누르면 멈추는 현상이 반복됨
- Build 101에서는 한 번 성공했음
- 이후 다시 실패
- 어제 되다가 오늘 아침에 다시 안 됨
- Render 재배포 후에도 해결 안 됨
- 즉, 정적 코드 문제 하나보다 환경/서버/세션/후처리 쪽 간헐 문제 가능성이 큼

이 상황에서는 아래 순서대로만 점검하세요.

━━━━━━━━━━━━━━━━━━
1. 실패 위치 100% 특정
━━━━━━━━━━━━━━━━━━

Apple 로그인 전체 흐름을 아래 단계로 쪼개고, 각 단계마다 고유 로그 태그를 넣으세요.

STEP1
- Apple 로그인 버튼 탭
- handleAppleLogin 진입 확인

STEP2
- signInAsync 호출 시작
- signInAsync 반환 성공/실패
- 성공 시 credential 존재 여부
- identityToken 존재 여부

STEP3
- 서버 요청 시작 직전 로그
- 요청 URL, method, timeout, request id
- 서버로 실제 fetch가 나갔는지

STEP4
- 서버 응답 수신 성공/실패
- HTTP status
- response body
- timeout / abort / network error 여부

STEP5
- setSession / setUser / SecureStore 저장 여부
- 저장 성공/실패

STEP6
- 라우팅 실행 여부
- 홈/온보딩/대시보드 중 어디로 분기되는지
- 분기 조건값 함께 로그

중요:
각 단계 로그는 아래 형식으로 통일하세요.
[AppleLogin][STEP1]
[AppleLogin][STEP2 OK]
[AppleLogin][STEP2 FAIL]
...
한 번의 로그인 시도마다 requestId 또는 traceId를 생성해서 모든 로그에 같이 붙이세요.
그래야 한 시도 흐름을 끝까지 추적할 수 있습니다.

━━━━━━━━━━━━━━━━━━
2. 간헐 실패 원인 후보를 “증거”로 분리
━━━━━━━━━━━━━━━━━━

아래 가능성 중 무엇인지 증거로 분리하세요.

A. Apple 시트 단계 실패
- signInAsync 자체가 반환 안 되거나 에러
- 이 경우 서버 문제 아님

B. 서버 요청 미도달
- credential은 왔는데 fetch 전/직후 멈춤
- 클라이언트 비동기/Promise 문제 가능

C. 서버 도달 후 응답 실패
- Render/API 문제
- 환경변수, Apple secret, JWT 검증, DB, timeout 문제 가능

D. 응답 성공 후 세션 저장 실패
- SecureStore/상태 업데이트/Abort 뒤엉킴 가능

E. 세션 저장 성공 후 라우팅 실패
- useEffect 조건, onboarding 분기, stale state 가능

이 다섯 가지 중 어디인지 확정해서 보고하세요.

━━━━━━━━━━━━━━━━━━
3. 서버 측 필수 점검
━━━━━━━━━━━━━━━━━━

서버 로그도 같이 보세요. 단순 “요청 안 옴” 수준 말고, 아래를 확인하세요.

1) Apple 로그인 엔드포인트 요청이 실제로 들어오는지
2) 들어오면 request body에서 무엇이 비어 있는지
3) Apple token decode/verify 단계 성공 여부
4) DB 조회/생성 단계 성공 여부
5) 응답 직전까지 갔는지
6) Render 재배포 전후 차이
7) 환경변수 누락/변경 여부
8) Apple 관련 secret/key/team/client 설정값이 현재 살아 있는지

특히 확인:
- Render 환경변수 변경 이력
- 재배포 직후와 실패 시점 서버 로그 차이
- timeout 또는 cold start 영향 여부

━━━━━━━━━━━━━━━━━━
4. “101에서 됐다가 다시 안 되는” 현상 기준으로 우선 의심할 것
━━━━━━━━━━━━━━━━━━

이 현상은 아래 순서로 의심하세요.

우선순위 1
- 서버 환경변수/Apple secret/키 설정/Render 상태 변화

우선순위 2
- 서버 응답은 성공하지만 세션 저장 또는 상태 반영 실패

우선순위 3
- Apple signInAsync 결과 후 비동기 흐름이 꼬여서 fetch 전 멈춤

우선순위 4
- 라우팅/useEffect 분기 문제

정적 코드 구조나 entitlement 문제만으로는
“어제 됐다가 오늘 안 되는” 패턴 설명이 약하므로,
환경/서버/세션 계층을 먼저 보세요.

━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━
