[학부모 모드 학생 자동 연결 실패 — 원인 추적 + 단일 서버 로직으로 재구축 지시]

현재 문제:
관리자 모드에는 학생이 등록되어 있는데, 학부모 모드 회원가입 시
학생 이름 + 학부모 전화번호가 일치해도 학생 연결이 되지 않는다.

기존 승인 버튼 / 승인 대기 / 수동 승인 과정은 모두 제거했고,
의도한 구조는 아래와 같다.

목표 구조:
- 관리자 모드에서 학생 등록 시 학생 이름과 학부모 전화번호를 저장
- 학부모 모드 회원가입 시 아이 이름 + 전화번호를 입력
- 두 값이 일치하면 즉시 자동 승인
- 관리자 추가 승인 없이 바로 학생 연결 완료
- 연결 성공 시 학부모 홈에서 학생 정보가 즉시 보여야 함

지금부터 UI 수정이 아니라
실제 DB 저장 구조 / 조회 쿼리 / 연결 로직을 끝까지 추적해서 원인을 잡고 수정해줘.

절대 원칙:
- 화면 문구만 바꾸지 말 것
- 프론트 임시 우회 금지
- 서버에서 조회 + 연결까지 한 번에 처리할 것
- 기존 승인 관련 잔여 로직이 있으면 전부 찾아낼 것
- 어떤 파일, 어떤 테이블, 어떤 쿼리가 문제인지 보고할 것
- 수정 전 현재 흐름을 먼저 분석할 것

━━━━━━━━━━━━━━━━━━
1. 먼저 해야 할 분석
━━━━━━━━━━━━━━━━━━

아래를 먼저 전수 조사해서 보고해줘.

1) 관리자 모드에서 학생 등록 시 실제 저장되는 테이블
- 테이블명
- 학생 이름 컬럼명
- 학부모 전화번호 컬럼명
- 수영장 ID 컬럼명
- 승인 상태 관련 컬럼 존재 여부

2) 학부모 모드 회원가입 후 학생 연결 시 호출되는 실제 코드 위치
- 화면 파일
- API 함수
- 서버 함수
- DB 조회 함수
- 연결 저장 함수

3) 현재 매칭 조건
- 이름으로 찾는지
- 전화번호로 찾는지
- 둘 다 쓰는지
- pool_id 조건이 있는지
- status / approved / pending 조건이 남아 있는지

4) 연결 성공 시 어디에 저장되는지
- parent_profiles
- student_parent_links
- students 테이블 직접 업데이트
- 기타 연결 테이블
정확히 어느 방식인지 보고

2. 무조건 추가해야 하는 디버그 로그
━━━━━━━━━━━━━━━━━━

학부모 회원가입 시 아래 로그를 서버에 반드시 남겨줘.

- 입력 childName 원본
- 입력 parentPhone 원본
- 정규화된 childName
- 정규화된 parentPhone
- 사용된 pool_id
- 실행된 조회 조건
- 조회된 학생 수
- 조회된 학생 ID 목록
- 연결 insert/update 성공 여부
- 실패 에러 원문

주의:
로그는 민감정보 마스킹해서 남기되,
비교가 가능한 수준으로는 보여야 한다.

예:
phone=010****5678
normalized=01012345678

━━━━━━━━━━━━━━━━━━
3. 데이터 정규화 규칙 강제
━━━━━━━━━━━━━━━━━━

전화번호 비교 실패 가능성이 높으므로 아래 규칙을 강제 적용해줘.

A. 학부모 전화번호
- 비교 전 숫자만 남기기
- 하이픈, 공백, 국가코드 제거/정리
- 예: 010-1234-5678 → 01012345678

B. 학생 이름
- trim 처리
- 양끝 공백 제거
- 연속 공백 정리
- 필요 시 비교용 normalized 값 생성

관리자 등록 시 저장값도 동일 규칙 적용
학부모 회원가입 입력값도 동일 규칙 적용

가능하면 DB에 아래 비교용 컬럼 추가 검토:
- student_name_normalized
- parent_phone_normalized

비교는 normalized 컬럼 기준으로 수행

━━━━━━━━━━━━━━━━━━
4. 자동 연결 로직 단일화
━━━━━━━━━━━━━━━━━━

현재 연결 로직이 여러 화면/여러 함수로 흩어져 있으면
하나의 서버 함수로 통합해줘.

예시 목표 함수:
linkParentToStudentAfterSignup({
  userId,
  poolId,
  childName,
  parentPhone
})

이 함수에서 아래를 한 번에 처리:
1) 입력값 정규화
2) 같은 수영장(pool_id) 내 학생 조회
3) childName + parentPhone 일치 학생 검색
4) 승인 관련 과거 조건 제거
5) 일치 시 연결 테이블 insert/update
6) 중복 연결 방지
7) 결과 반환

반환 형식 예시:
- success: true/false
- reason: MATCHED / NOT_FOUND / DUPLICATE / ERROR
- matchedStudentIds: []

━━━━━━━━━━━━━━━━━━
5. 반드시 점검할 문제 후보
━━━━━━━━━━━━━━━━━━

아래 항목을 하나도 빠짐없이 점검해줘.

- 관리자 학생 정보가 실제로 pool DB에 저장되는데 학부모 쪽은 다른 DB를 조회하고 있지 않은지
- 전화번호 형식 불일치로 조회가 0건이 되는지
- 학생 이름 공백/trim 문제로 조회가 실패하는지
- 기존 approval / pending / approved 조건이 아직 서버 쿼리에 남아 있는지
- 연결 저장 시 RLS / FK / unique constraint 때문에 막히는지
- 학부모 계정 생성 직후 user id가 아직 준비되기 전에 연결 로직이 먼저 돌아가는 race condition이 있는지
- 연결 후 조회 캐시가 남아 학부모 홈에서 즉시 안 보이는지
- pool_id가 잘못 들어가서 타 수영장 범위로 조회되거나 아예 0건 되는지

━━━━━━━━━━━━━━━━━━
6. 수정 목표
━━━━━━━━━━━━━━━━━━

수정 후 아래 시나리오가 반드시 성공해야 한다.

시나리오 A
- 관리자 모드에서 학생 등록
- 학생 이름: 홍길동
- 학부모 전화번호: 010-1234-5678

학부모 회원가입
- 아이 이름: 홍길동
- 전화번호: 01012345678

결과:
- 자동 연결 성공
- 승인 버튼 없이 바로 연결
- 학부모 홈에서 학생 정보 즉시 표시

시나리오 B
- 전화번호만 다르면 연결 실패
- 이름만 다르면 연결 실패
- 실패 사유를 사용자에게 명확하게 표시

시나리오 C
- 이미 연결된 학생이면 중복 연결 방지

━━━━━━━━━━━━━━━━━━
7. 최종 결과물
━━━━━━━━━━━━━━━━━━

최종적으로 아래를 보고해줘.

1) 실제 원인
- 정확히 어느 파일/어느 쿼리/어느 조건이 문제였는지

2) 수정 내용
- 어떤 파일 수정
- 어떤 함수 수정/추가
- 어떤 테이블/컬럼 사용

3) 테스트 결과
- 위 시나리오 A/B/C 통과 여부

4) 남은 리스크
- 형제자매 동일 번호 처리
- 동명이인 처리
- 향후 자녀 추가 연결 방식 제안

중요:
임시 우회 말고
실제 자동 연결이 동작하도록 끝까지 수정해줘.
먼저 현재 연결 흐름 분석 결과부터 보고하고,
그 다음 수정안을 적용해줘.
