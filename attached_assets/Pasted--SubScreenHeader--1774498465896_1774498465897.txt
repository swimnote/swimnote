[스윔노트 — 공통 헤더/공통 컴포넌트 인증값 누수 전수점검 지시문]

현재 SubScreenHeader의 홈 버튼 버그 원인은 해결 방향이 맞다.
하지만 이번 문제는 한 파일만의 문제가 아니라,
공통 컴포넌트가 실제 인증 Provider가 아닌 null context 또는 잘못된 인증 소스를 읽는 패턴이 다른 곳에도 숨어 있을 가능성이 높다.

이번 작업은 SubScreenHeader 수정 내용을 유지하면서,
공통 컴포넌트 전반의 인증값/역할값 참조 방식을 전수점검하고,
권한 라우팅 누수를 완전히 막는 작업이다.

━━━━━━━━━━━━━━━━━━
1. 현재 수정 유지
━━━━━━━━━━━━━━━━━━

아래 수정은 유지한다.

- SubScreenHeader.tsx
  - useContext(AuthContext) 제거
  - useAuth() 사용
  - sessionKind → kind 수정
  - router.navigate() → router.replace()
  - 알 수 없는 역할 fallback → "/" 로 변경

- (admin)/_layout.tsx
  - super_admin / platform_admin / super_manager 가 admin 진입 시
    /(super)/dashboard 로 즉시 리다이렉트

이 변경을 되돌리지 말 것

━━━━━━━━━━━━━━━━━━
2. 전수점검 대상
━━━━━━━━━━━━━━━━━━

다음 패턴을 프로젝트 전체에서 검색해서 전수 확인:

- createContext(...null) 로 만든 인증/세션 컨텍스트 직접 사용
- useContext(AuthContext)
- useContext(SessionContext)
- role fallback이 admin으로 가는 코드
- unknown role일 때 /(admin)/dashboard 로 보내는 코드
- router.navigate("/(admin)/dashboard")
- 공통 Header / Tab / TopBar / SubHeader / BackButton / EmptyState / ModalHeader 내부 role 분기
- localStorage / AsyncStorage의 role 값을 직접 읽는 UI 컴포넌트
- 실제 Provider 없이 context를 읽는 컴포넌트

목표:
👉 실제 인증값은 반드시 useAuth()/useSession() 같은 실제 Provider 연결 훅으로만 읽게 할 것

━━━━━━━━━━━━━━━━━━
3. 강제 규칙
━━━━━━━━━━━━━━━━━━

[인증값 읽기 규칙]
- UI 컴포넌트는 null context 직접 참조 금지
- 인증/역할값은 실제 Provider가 연결된 훅으로만 읽기
- auth가 없으면 admin fallback 금지
- auth가 불명확하면 안전한 경로 "/" 또는 로그인 화면으로 보낼 것

[역할 라우팅 규칙]
- unknown role → admin 금지
- super role → super 영역
- pool_admin → admin 영역
- teacher → teacher 영역
- parent → parent 영역

━━━━━━━━━━━━━━━━━━
4. fallback 안전화
━━━━━━━━━━━━━━━━━━

프로젝트 전체에서 아래 잘못된 패턴 제거:

- default route = /(admin)/dashboard
- fallback route = /(admin)/dashboard
- role 없을 때 admin으로 이동

안전 기준:
- 인증 없음 → 로그인/루트
- role 불명확 → 루트
- super 계열은 super 우선
- admin/teacher/parent는 본인 영역만

━━━━━━━━━━━━━━━━━━
5. 공통 컴포넌트 점검 항목
━━━━━━━━━━━━━━━━━━

반드시 확인:

- SubScreenHeader
- 공통 Header
- 공통 뒤로가기 버튼
- 탭 헤더
- 카드에서 상세 이동 버튼
- 홈 버튼
- 설정 버튼
- 로그아웃 후 첫 진입
- 새로고침 후 첫 진입
- 딥링크/직접 URL 진입

중요:
- “한 군데만 수정” 금지
- 같은 패턴 발견 시 전부 고칠 것

━━━━━━━━━━━━━━━━━━
6. role 누수 차단
━━━━━━━━━━━━━━━━━━

아래도 같이 확인:

- super 계정이 admin route 직접 접근 시 차단
- admin 계정이 super route 직접 접근 시 차단
- teacher, parent도 타 영역 직접 접근 차단
- 화면 렌더링 전에 role 검사
- 반쪽 차단 금지 (화면은 보이고 API만 막히는 구조 금지)

━━━━━━━━━━━━━━━━━━
7. 실제 테스트
━━━━━━━━━━━━━━━━━━

반드시 실제로 확인:

1. super_admin 로그인 후 공통 홈 버튼
→ /(super)/dashboard

2. pool_admin 로그인 후 공통 홈 버튼
→ /(admin)/dashboard

3. teacher 로그인 후 공통 홈 버튼
→ teacher 홈

4. parent 로그인 후 공통 홈 버튼
→ parent 홈

5. role 불명확 상태 / auth null 상태
→ "/" 또는 로그인 화면

6. super 계정으로 admin route 직접 접근
→ super 홈으로 리다이렉트

7. localStorage/AsyncStorage에 잘못된 role 값 주입
→ 공통 헤더가 영향받지 않는지 확인

━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━
