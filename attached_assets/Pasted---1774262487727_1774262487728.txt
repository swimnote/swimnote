[스윔노트 전체 플랫폼 자동 테스트 + 대규모 시뮬레이션 + 기능 검증 통합 지시서]

지금부터 이 프로젝트 전체에 대해
리플릿이 수행할 수 있는 범위 안에서 가능한 테스트를 전부 실행하고,
기능별 작동 여부, 데이터 연결, 상태 변화, 화면 진입, 예약 동작, 대규모 데이터 시뮬레이션까지 포함한
“실전형 자동 테스트 + 시뮬레이션 점검”을 수행하라.

중요:
- 이번 작업은 단순 코드 에러 확인이 아니다
- 실제 운영처럼 시뮬레이션을 돌려서 기능이 끝까지 이어지는지 점검해야 한다
- 테스트 중 발견한 오류는 숨기지 말고 전부 기록
- 가능한 항목은 자동으로 직접 테스트하고
- 자동으로 확인이 불가능한 항목은 무엇이 왜 불가능한지 분명히 적어라
- 테스트를 했다고 말만 하지 말고 실제로 어떤 데이터를 넣고 어떤 흐름을 돌렸는지 결과를 보고하라
- “완료” 한 줄 보고 금지
- 테스트 중 기능이 깨지면 원인 추적 후 수정하고 다시 테스트하라
- 시간이 걸려도 괜찮으니 빠짐없이 하라

==================================================
1. 테스트 목표
==================================================

목표는 아래 4가지다.

1. 전체 모드 기능이 실제로 연결되어 작동하는지 확인
2. 대규모 운영 상황에서도 구조가 버티는지 시뮬레이션
3. 결제/구독/회원수/용량/사진/일지/푸시/권한/복구 흐름이 꼬이지 않는지 확인
4. 지금 상태가 실제 MVP 운영 가능한 수준인지 검증

==================================================
2. 테스트 대상 모드
==================================================

아래 4개 모드를 모두 포함한다.

- 슈퍼관리자
- 수영장 관리자
- 선생님
- 학부모

각 모드별:
- 화면 진입
- 버튼 동작
- 상태 변경
- 데이터 반영
- 다른 모드와의 연결
을 전부 점검한다.

==================================================
3. 기본 테스트 방식
==================================================

테스트는 아래 순서로 반드시 반복 수행한다.

1. 정적 점검
- import/export 오류
- 라우팅 오류
- undefined component
- 렌더링 실패
- 콘솔 에러
- 타입 충돌
- 상태값 누락

2. 화면/버튼 점검
- 각 메뉴 진입 가능 여부
- 각 카드/탭/버튼 작동 여부
- 잘못 연결된 메뉴가 없는지
- 빈 화면/깨진 화면 여부

3. 기능 흐름 점검
- 생성 / 수정 / 저장 / 삭제 / 조회 / 전환 / 필터 / 탭 / 팝업
- 역할별 권한 차단 여부

4. 대규모 데이터 시뮬레이션
- 회원 대량 생성
- 수영장 다중 생성
- 사진/일지/일정/결제/로그/푸시 발생 시뮬레이션
- 성능과 구조 점검

5. 오류 수정 후 재테스트
- 발견한 오류를 수정하고
- 같은 흐름을 다시 돌려서 회귀 테스트

==================================================
4. 대규모 시뮬레이션 규모 (반드시 수행)
==================================================

아래 규모로 시뮬레이션 데이터를 생성해서 테스트하라.

기준 규모:
- 수영장 10개
- 각 수영장 회원 최소 120명
- 총 회원 1,200명 이상

추가 생성 대상:
- 각 수영장 관리자 계정
- 각 수영장 선생님 계정 여러 명
- 학부모 계정
- 다자녀 계정 일부 포함
- 반/수업 스케줄
- 출결 기록
- 결석/보강 데이터
- 일지 데이터
- 사진 업로드 데이터
- 푸시 이벤트
- 결제/구독/용량 데이터

중요:
더미 데이터만 대충 넣지 말고
실제 운영처럼 관계가 맞게 생성하라.

예:
- 학생 ↔ 학부모 연결
- 학생 ↔ 반 배정
- 반 ↔ 선생님
- 수영장 ↔ 구독 플랜
- 회원수 ↔ 과금 집계
- 사진 ↔ 학생 또는 반
- 일지 ↔ 반/학생
- 보강 ↔ 결석 원본 연결

==================================================
5. 반드시 시뮬레이션해야 하는 항목
==================================================

--------------------------------------------------
5-1. 회원/학생/학부모 구조
--------------------------------------------------

반드시 테스트:
- 학부모 가입
- 자동 승인 조건
- 추가 자녀 연결
- 다자녀 탭 전환
- 학생 미배정 상태
- 반배정
- 학생 프로필
- 퇴원 / 탈퇴 / 삭제 처리
- 삭제 후 조회 차단
- 삭제 로그 남는지

대규모 시뮬레이션:
- 수영장당 회원 120명 이상 생성
- 일부는 연기회원
- 일부는 퇴원회원
- 일부는 다자녀 계정
- 상태값별 집계 확인

--------------------------------------------------
5-2. 수업/출결/보강
--------------------------------------------------

반드시 테스트:
- 수업 스케줄 생성
- 출결 체크
- 결석 처리
- 자동 보강 생성
- 보강 가능/완료/기간 만료 분기
- 보강 정책 기간 계산
- 보강 횟수 제한
- 보강 출석 완료 표시
- 관리자 읽기 전용 유지
- 선생님 모드에서만 수업 실무 수정 가능 여부

대규모 시뮬레이션:
- 각 수영장 여러 반 운영
- 결석 다수 발생
- 보강 다수 생성
- 기간 만료 회원 생성
- 탭별 리스트 정합성 확인

--------------------------------------------------
5-3. 일지/피드백/사진
--------------------------------------------------

반드시 테스트:
- 반 공통 일지 작성
- 개별 피드백 추가
- 학부모에서 자기 아이 기준 표시
- 일지 저장 시 푸시
- 사진 단독 업로드 시 사진 푸시
- 일지 + 사진 동시 저장 시 일지 푸시만 발송
- 사진/일지/출결/레벨이 다른 자녀와 섞이지 않는지

대규모 시뮬레이션:
- 각 수영장 다수 일지 생성
- 사진 업로드 반복
- 학생별/반별 데이터 연결 정상 여부
- 저장 후 리스트/상세/푸시 반영 여부

--------------------------------------------------
5-4. 레벨 시스템
--------------------------------------------------

반드시 테스트:
- 기본 1~10 fallback
- 관리자 레벨 설정
- 사용안함 처리
- 뱃지 반영
- 선생님이 학생 레벨 변경
- 관리자/학부모/선생님 프로필 동시 반영
- 학부모 수영정보에 레벨 설명/학습 내용/승급 기준 노출
- 비활성 레벨 선택 차단

대규모 시뮬레이션:
- 여러 수영장별 다른 레벨 체계
- 6개만 쓰는 수영장 / 10개 다 쓰는 수영장
- 학생별 레벨 분포
- 레벨 변경 후 전체 반영 확인

--------------------------------------------------
5-5. 푸시 시스템
--------------------------------------------------

반드시 테스트:
[학부모]
- 공지 즉시
- 전날 수업 알림
- 당일 수업 알림
- 일지 푸시
- 사진 푸시
- ON/OFF 작동

[선생님]
- 오늘 수업 일정 알림
- 학부모 쪽지 알림
- 메신저 알림
- ON/OFF 작동

[수영장 관리자]
- 메신저 알림
- 운영 경고 알림 ON/OFF
- 구독/결제 알림 무조건 발송

대규모 시뮬레이션:
- 다량 푸시 이벤트 발생
- 중복 발송 여부
- 역할별 발송 대상 정합성
- 설정 꺼진 사용자에게 발송 안 되는지
- 예약 푸시 시간 계산 정상 여부

--------------------------------------------------
5-6. 공지/메신저/쪽지
--------------------------------------------------

반드시 테스트:
- 슈퍼 공지
- 수영장 공지
- 학부모 쪽지
- 메신저 새 메시지
- 알림/푸시 연결
- 최신 공지 규칙
- 쪽지 누적 구조

대규모 시뮬레이션:
- 다수 공지 발송
- 메신저 대량 이벤트
- 관리자/선생님/학부모 수신 정합성

--------------------------------------------------
5-7. 결제/구독/매출/지출
--------------------------------------------------

반드시 테스트:
- 실제 결제 시뮬레이션
- 구독 생성
- 구독 갱신
- 결제 실패
- 환불
- 부분 환불
- 중도 해지 수수료 10%
- 결제일 기준 매출 집계
- 해당 월 환불은 해당 월 마이너스 반영
- 연기회원 과금 포함
- 회원수 집계와 과금 일치
- 용량 업그레이드
- 추가 용량 구매
- 지출 계산
- 순이익 계산

대규모 시뮬레이션:
- 수영장 10개 각각 다른 플랜
- 회원수 증가에 따른 과금
- 용량 상승에 따른 지출 변화
- 결제/구독 알림 발생
- 매출/지출/로그 수치 일치 여부

중요:
“실제로 결제하고 구독하고 용량 올리고 회원수 넣고 사진 올리고”
이 흐름을 가능한 범위에서 반드시 시뮬레이션하라.
테스트 결제/가상 결제라도 흐름상 전부 돌려라.

--------------------------------------------------
5-8. 지점관리 / 멀티 수영장
--------------------------------------------------

반드시 테스트:
- 운영자 1명이 여러 수영장 보유
- 수영장 등록
- 수영장 목록 진입
- 수영장 전환
- 전환 후 재진입처럼 상태 새로 로드
- 데이터 분리
- 선택 복사 (가격/레벨/결제정보/피드백문구)
- 비선택 항목 미복사

대규모 시뮬레이션:
- 동일 운영자 다수 수영장 관리
- pool_id 데이터 혼합 없는지 확인

--------------------------------------------------
5-9. 보안/외부서비스/로그
--------------------------------------------------

반드시 테스트:
- 보안설정 진입
- 외부 서비스 연결 상태 화면
- 상세 보기
- 새로고침 버튼
- 상태값 표시
- 로그 필드 누락 여부
- Invalid Date / NaN / undefined 금지
- 고위험 작업 재인증 흐름

대규모 시뮬레이션:
- 다양한 로그 누적
- 서비스 상태 refresh 반복
- 로그/감사 화면 성능 점검

==================================================
6. 실제 운영 시나리오 테스트 (끝까지 돌릴 것)
==================================================

반드시 아래 시나리오를 처음부터 끝까지 여러 번 돌려라.

시나리오 A:
- 수영장 생성
- 관리자 로그인
- 선생님 승인
- 학생 등록
- 학부모 승인/자동연결
- 반배정
- 수업 출결
- 일지 작성
- 학부모 확인
- 푸시 수신

시나리오 B:
- 결석
- 자동 보강 생성
- 보강 기간 계산
- 보강 출석 완료
- 만료 처리
- 학부모 확인

시나리오 C:
- 구독 시작
- 결제 완료
- 회원수 증가
- 용량 증가
- 매출/지출 계산
- 구독/결제 알림

시나리오 D:
- 퇴원 처리
- 탈퇴 처리
- 삭제
- 삭제 후 조회 차단
- 슈퍼관리자 삭제 로그 확인

시나리오 E:
- 다자녀 계정
- 자녀 탭 전환
- 자녀별 일지/사진/레벨/수업 분리 확인

시나리오 F:
- 지점 추가
- 복사 항목 일부 선택
- 새로운 수영장 진입
- 기존 수영장과 데이터 분리 확인

==================================================
8. 자동 수정 원칙
==================================================

테스트 중 발견한 오류는 아래 우선순위로 수정하라.

1. 데이터 오류
2. 기능 오류
3. 권한 오류
4. 예약/시간 계산 오류
5. UI 오류

수정 후 반드시 같은 시나리오를 다시 돌려 회귀 테스트하라.

==================================================
9. 테스트 불가능 항목 처리
==================================================

리플릿이 자동으로 완전 검증할 수 없는 항목은 숨기지 말고 분리해서 적어라.

예:
- 실제 iOS 푸시 수신
- 실제 PG 실결제
- 실제 App Store / Play Store 연동
- 실제 기기 문자앱 호출
- 실제 외부 서비스 실환경 응답

이 경우에는
- 현재 어떤 수준까지 시뮬레이션했는지
- 무엇이 가상 데이터/목 처리인지
- 실제 기기 또는 실환경에서 무엇을 추가 점검해야 하는지
명확히 적어라.

10. 완료 기준
==================================================

아래를 만족하기 전까지 완료 처리 금지:

- 수영장 10개 / 회원 1,200명 이상 시뮬레이션 수행
- 핵심 기능 흐름 전부 테스트
- 결제/구독/용량/회원수/사진/일지/푸시 시뮬레이션 수행
- 발견 오류 수정 후 재테스트 완료
- 각 모드별 치명 오류 없음
- 상태값/집계/권한 꼬임 없음
- 남은 제한사항 명확히 보고

12. 최종 목적
==================================================

이번 작업의 목적은
스윔노트를 단순 기능 묶음이 아니라
“실제 수영장 운영을 버틸 수 있는 MVP”인지 검증하는 것이다.

핵심:
- 전체 모드 자동 점검
- 대규모 운영 시뮬레이션
- 결제/구독/용량/회원/사진/일지/푸시 전체 흐름 검증
- 오류 수정 및 회귀 테스트
- 운영 가능 수준 판단

시간이 오래 걸려도 된다.
가능한 범위의 테스트를 전부 수행하고,
빠짐없이 보고하라.
