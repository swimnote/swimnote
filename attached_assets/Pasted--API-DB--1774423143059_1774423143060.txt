지금부터 “변경이 왜 반영되지 않는지” 근본 원인 추적만 해라.
기능 수정은 잠시 중단하고, 실제 실행 경로/번들/API/DB를 증거로 확인하는 디버깅 작업을 해라.

중요:
- “캐시 같아요” 식 추측 금지
- 실제 증거 제출
- 웹과 Expo Go 둘 다 기준으로 확인
- 수정했다고 말만 하지 말고, 화면에 보이는 증거/응답 증거/로그 증거를 제출해라

━━━━━━━━━━━━━━━━━━
1. 목표
━━━━━━━━━━━━━━━━━━

현재 문제:
- 수정한 내용이 웹과 Expo Go에 둘 다 반영되지 않음

이번 작업 목표:
1. 내가 수정한 파일이 실제 실행 중인 화면 파일이 맞는지 확인
2. 현재 앱이 어떤 번들을 보고 있는지 확인
3. 어떤 API/DB를 실제로 보고 있는지 확인
4. 왜 변경이 반영되지 않는지 정확한 원인을 특정

━━━━━━━━━━━━━━━━━━
2. 1차 확인: 화면 파일이 진짜 맞는지
━━━━━━━━━━━━━━━━━━

이번에 수정했다고 주장하는 화면 파일에
아주 눈에 띄는 임시 디버그 문구를 직접 넣어라.

예:
- 화면 최상단 빨간 텍스트
- "DEBUG_ATTENDANCE_0325_V1"
- "DEBUG_CLASS_ASSIGN_0325_V1"

규칙:
- 웹과 Expo Go에서 둘 다 눈으로 바로 보이게 넣을 것
- 스타일 강하게 넣을 것
- 작은 console.log 말고 실제 화면 텍스트로 넣을 것

확인 결과:
- 보이면 → 해당 파일이 실제 실행 파일 맞음
- 안 보이면 → 다른 파일/다른 라우트를 보고 있는 것

반드시 제출:
- 수정한 파일 경로
- 삽입한 디버그 문구
- 웹에서 보이는지
- Expo Go에서 보이는지

━━━━━━━━━━━━━━━━━━
3. 2차 확인: 실제 라우트가 어느 파일을 쓰는지
━━━━━━━━━━━━━━━━━━

반드시 찾아라:
- 현재 문제 화면의 실제 route
- 그 route가 import하는 컴포넌트 파일
- 중간 wrapper/layout 파일
- 같은 이름의 중복 파일 존재 여부

특히 확인:
- app/ 와 artifacts/swim-app/app/ 중 무엇이 실제 실행 기준인지
- (teacher), (admin), 공용 route 중 어느 경로를 실제 쓰는지
- 같은 기능 화면이 2개 이상 존재하는지

반드시 제출:
- 실제 route 경로
- 실제 렌더링 파일 경로
- 관련 import 체인

━━━━━━━━━━━━━━━━━━
4. 3차 확인: 캐시/번들 문제인지
━━━━━━━━━━━━━━━━━━

반드시 아래를 실행하고 결과를 제출해라.

- 앱/서버 완전 중지
- API 서버 완전 재기동
- Expo 캐시 초기화 모드로 시작
- 웹 preview 새로 열기
- 시크릿창에서도 확인
- Expo Go 앱 완전 종료 후 새 QR로 재접속

그리고 다시 디버그 문구가 보이는지 확인해라.

반드시 제출:
- 재기동 명령 또는 절차
- 캐시 초기화 여부
- 웹/Expo Go 각각 문구 표시 여부

━━━━━━━━━━━━━━━━━━
5. 4차 확인: API가 진짜 바뀌었는지
━━━━━━━━━━━━━━━━━━

문제 화면에서 호출하는 API를 찾아라.
그리고 그 API 응답에 “수정 후 값”이 들어가는지 직접 확인해라.

예:
- attendance 화면이면 /api/attendance 또는 관련 overview API
- class-assign면 /api/teachers/unassigned 또는 관련 API

방법:
- 서버 로그
- 네트워크 응답
- 필요하면 응답 JSON에 임시 debug 필드 추가

예:
{
  "debug_version": "ATTENDANCE_0325_V1"
}

확인:
- 응답에 debug_version이 보이는지
- 웹/Expo Go가 그 새 응답을 실제 받는지

반드시 제출:
- 호출 API 경로
- 현재 응답 JSON 일부
- debug_version 표시 여부

━━━━━━━━━━━━━━━━━━
6. 5차 확인: DB 마이그레이션/테이블 반영 여부
━━━━━━━━━━━━━━━━━━

이번 수정이 DB 구조에 의존하면 반드시 실제 DB까지 확인해라.

확인 항목:
- 추가/수정한 테이블 존재 여부
- 추가/수정한 컬럼 존재 여부
- enum 값 존재 여부
- migration 로그
- 서버 시작 시 migration 실행 여부

Supabase에서 직접 확인:
- super DB 프로젝트
- pool DB 프로젝트
- 실제 테이블/컬럼

반드시 제출:
- 확인한 프로젝트명
- 확인한 테이블명
- 존재 여부
- migration 적용 결과

━━━━━━━━━━━━━━━━━━
7. 6차 확인: 조건문 때문에 안 보이는지
━━━━━━━━━━━━━━━━━━

수정한 코드가 있어도 아래 이유로 화면에 안 보일 수 있다.

- 특정 role일 때만 렌더
- 특정 pool_id일 때만 렌더
- feature flag로 가림
- stale state 때문에 옛 상태 유지
- useMemo/useEffect 조건으로 렌더 안 됨
- modal 조건문 때문에 안 뜸

반드시 확인:
- 현재 계정 role
- 현재 pool_id
- 렌더 조건식
- disabled 조건
- early return 조건
- hide/show 조건

필요하면 조건문 옆에도 임시 debug 텍스트 넣어라.
예:
- "ROLE_OK"
- "POOL_OK"
- "FEATURE_ON"

━━━━━━━━━━━━━━━━━━
8. 최종 보고 형식
━━━━━━━━━━━━━━━━━━

반드시 아래 형식으로 제출해라.

1. 수정하려던 기능명
2. 수정한 파일 경로
3. 실제 실행 파일 경로
4. 화면 디버그 문구 표시 여부
   - 웹
   - Expo Go
5. 호출 API 경로
6. API 응답 debug_version 표시 여부
7. DB 반영 여부
8. 캐시/번들 문제 여부
9. 실제 근본 원인
   - 잘못된 파일 수정인지
   - 캐시인지
   - 다른 route인지
   - API 미반영인지
   - DB 미적용인지
   - 조건문 문제인지
10. 그 원인을 어떻게 고칠지

중요:
- “다시 해보겠습니다” 금지
- 반드시 근본 원인 하나 이상 특정해서 제출
