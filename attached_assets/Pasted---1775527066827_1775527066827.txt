[스윔노트 구독 기능 전면 점검 및 재구축 지시문]

현재 스윔노트의 구독 기능이 슈퍼관리자 / 수영장관리자 양쪽에서 서로 다르게 동작하고 있으며,
실제 플랜 상태, 화면 표시, 이동 경로, RevenueCat 연동, 수동 플랜 부여, 회원 수 제한, 저장용량 반영이 일치하지 않는 문제가 발생하고 있다.

이번 작업은 임시 패치가 아니라
“구독 상태의 원천 데이터 정리 → 연결 코드 전수 점검 → 불필요한 구독 찌꺼기 코드 제거 → 단일 로직으로 재구축 → 실제 동작 검증”
순서로 진행한다.

절대 UI만 수정하지 말고,
반드시 DB / API / 앱 라우팅 / RevenueCat 연동 / 슈퍼관리자 수동 부여 / 플랜 자동 반영까지 전체 구조를 점검하고 수정해라.

━━━━━━━━━━━━━━━━━━
1. 최우선 목표
━━━━━━━━━━━━━━━━━━

아래 6가지를 전부 만족하도록 수정한다.

1) 수영장관리자에서 “구독” 버튼 클릭 시
   올바른 구독/플랜 관리 페이지로 이동해야 한다.
   현재 이상한 페이지로 이동하는 문제를 원인까지 찾아 수정한다.

2) RevenueCat 연동 여부를 코드 기준으로 실제 확인한다.
   SDK 설치만 되어 있고 실제 동작 안 하는 상태인지,
   entitlement / customer info / product / offering / webhook / restore 흐름이 누락된 것인지
   정확히 분석한다.

3) 슈퍼관리자가 결제 없이도 특정 수영장에 플랜을 직접 부여할 수 있어야 한다.
   예: 체험중 / 무료 / Premium100 / Premium300 / 화이트라벨 등
   이 수동 부여가 저장되고 즉시 반영되어야 한다.

4) 플랜이 바뀌면 회원 한도, 저장 용량, 영상 기능 잠금 여부, 상태 배지, 표시 문구가
   수영장관리자 화면에 자동 반영되어야 한다.

5) 결제 기반 플랜과 슈퍼관리자 수동 부여 플랜이 충돌하지 않도록
   우선순위와 상태 계산 로직을 명확히 정리한다.

6) 수정 후 반드시
   “어떤 코드가 문제였는지 / 무엇을 제거했는지 / 무엇을 새로 만들었는지 / 실제 테스트 결과”
   보고서를 작성해서 제출한다.

━━━━━━━━━━━━━━━━━━
2. 반드시 먼저 할 일: 현재 구조 전수 조사
━━━━━━━━━━━━━━━━━━

수정하기 전에 아래를 먼저 전수 조사해서 보고하라.

A. 구독 관련 화면/라우트
- 슈퍼관리자 구독 관리 화면
- 수영장관리자 구독/결제 화면
- 수영장 상세 화면 내 구독 정보 카드
- 구독 변경 버튼 클릭 시 연결되는 route / screen / link
- 잘못 연결된 페이지가 있으면 현재 연결 구조를 표로 정리

B. DB / 테이블 / 컬럼 조사
현재 구독 관련 값이 어디에 저장되는지 모두 찾는다.
예시:
- swimming_pools
- subscriptions
- payment_logs
- revenue_logs
- users
- pool_settings
- storage_settings
- any revenuecat cache table
- any entitlement mapping table
- any admin override table

다음 항목이 실제 어디에 저장되는지 전부 찾는다.
- 현재 플랜명
- 구독 상태
- 체험 여부 / 체험 종료일
- 결제 플랫폼
- 회원 한도
- 저장 용량
- 영상 기능 활성화 여부
- 화이트라벨 여부
- 결제/수동부여 구분
- 만료일
- 크레딧 또는 부가 과금 관련 값

C. 코드 전수 탐색
아래 키워드 전부 검색해서 관련 파일 목록을 먼저 뽑아라.
- subscription
- plan
- pricing
- revenuecat
- purchaser
- entitlement
- offering
- package
- premium
- free
- trial
- memberLimit
- storageLimit
- videoAddon
- whiteLabel
- billing
- purchase
- restore
- customerInfo
- pool subscription
- admin override

D. 실제 데이터 흐름 추적
아래 흐름을 코드 기준으로 추적하라.
- 수영장관리자가 구독 버튼 누름
- 결제 페이지 진입
- 결제 완료 후 상태 갱신
- 앱 재실행 시 구독 상태 불러오기
- 슈퍼관리자가 플랜 직접 변경
- 변경 직후 수영장관리자 화면 반영
- 회원 한도 / 저장용량 자동 반영

이 과정에서
“어느 값이 기준이고 어느 값이 화면용 캐시인지”
정확히 밝혀라.

━━━━━━━━━━━━━━━━━━
3. 구조 원칙: 구독 상태 단일 기준값으로 재구성
━━━━━━━━━━━━━━━━━━

현재 여러 곳에서 플랜명을 따로 들고 있거나,
화면에 임시 문자열을 직접 쓰는 구조면 전부 제거한다.

반드시 구독 상태의 단일 기준값(single source of truth)을 만든다.

권장 구조:
- subscriptions 또는 pool_subscription_state 같은 단일 상태 테이블/서비스를 기준으로 사용
- 모든 화면은 이 단일 상태를 조회해서 표시
- 회원 한도 / 저장용량 / 영상기능 / 화이트라벨 여부는
  플랜 정의값(plan definition)에서 파생 계산되게 만들 것
- 화면에서 문자열 하드코딩 금지
- 중복된 local state / 임시 매핑 / 예전 플랜 분기 제거

플랜 정의는 코드 한 군데에서만 관리한다.
예시 필드:
- planCode
- planName
- memberLimit
- storageMb 또는 storageGb
- videoEnabled
- whiteLabelEnabled
- monthlyPrice
- isTrialAllowed
- isManualAssignable

예시 플랜
- FREE_10
- PREMIUM_100
- PREMIUM_300
- PREMIUM_500
- PREMIUM_1000
- CUSTOM
- WHITE_LABEL
- TRIAL

실제 프로젝트 명칭에 맞게 정리하되,
절대 여러 파일에서 따로 관리하지 말 것.

━━━━━━━━━━━━━━━━━━
4. RevenueCat 점검 및 재연결
━━━━━━━━━━━━━━━━━━

RevenueCat은 “붙어있는 척만 하는 상태”인지 실제 동작하는지 반드시 검증한다.

점검 항목:
1) SDK 초기화 코드 존재 여부
2) 플랫폼별 API Key 적용 여부
3) 로그인 사용자 식별(appUserID) 일관성 여부
4) offerings / packages 조회 정상 여부
5) purchase 호출 연결 여부
6) restore purchases 연결 여부
7) customerInfo / entitlement 읽는 코드 존재 여부
8) 결제 성공 후 서버/DB 반영 로직 존재 여부
9) webhook 수신 및 DB 동기화 존재 여부
10) trial / expiration / renewal / cancellation 처리 여부

중요:
RevenueCat만 읽고 화면만 바꾸는 구조면 안 된다.
반드시 서버/DB와 동기화되어야 한다.

필요하면 아래 방식으로 재구축해라.
- 앱: RevenueCat purchase / restore / customerInfo 확인
- 서버: webhook 수신
- 서버 DB: pool의 실제 subscription state 갱신
- 앱 화면: DB 기준값 또는 서버 계산 결과 기준으로 노출

“앱 메모리 상태”를 구독 기준값으로 쓰지 마라.

━━━━━━━━━━━━━━━━━━
5. 슈퍼관리자 수동 플랜 부여 기능 재구축
━━━━━━━━━━━━━━━━━━

슈퍼관리자가 결제 없이 플랜을 직접 부여할 수 있어야 한다.
이 기능은 반드시 살아 있어야 한다.

필수 기능:
- 수영장별 현재 플랜 조회
- 플랜 직접 변경
- trial 시작/종료일 수정
- 회원 한도 직접 변경 가능 여부 선택
- 저장 용량 직접 변경 가능 여부 선택
- 영상 기능 on/off
- 화이트라벨 on/off
- 메모 남기기
- 변경 이력 저장
- 누가 언제 무엇을 바꿨는지 로그 저장

권장 우선순위 규칙:
1) 슈퍼관리자 수동 override가 있으면 그것이 최우선
2) override가 없으면 RevenueCat/정기결제 상태 사용
3) 둘 다 없으면 free 기본값 적용

반드시 override 적용 여부를 명시적으로 저장해라.
예:
- source = manual | revenuecat | free_default
- override_active = true/false

슈퍼관리자가 플랜을 바꾸면
수영장관리자 화면에 즉시 반영되어야 한다.

━━━━━━━━━━━━━━━━━━
6. 플랜 변경 시 자동 반영되어야 하는 항목
━━━━━━━━━━━━━━━━━━

플랜이 바뀔 때 아래 값들이 자동 변경되어야 한다.
수동 수정 없이 한 번에 반영되게 구현해라.

필수 자동 반영 항목:
- 현재 플랜명
- 구독 상태 배지
- 체험중 / 운영중 / 만료 / 무료 상태
- 회원 한도
- 저장 용량
- 영상 업로드 가능 여부
- 화이트라벨 여부
- 결제 플랫폼 표기
- 구독 시작일 / 종료일 / 체험 종료일
- 수영장관리자 설정 화면 내 옵션 제한
- 회원 추가 가능 여부
- 사진/영상 업로드 제한 경고
- 저장 공간 부족 경고 기준

추가로,
회원 한도 초과 시 처리 정책도 점검해라.
예:
- 기존 회원은 유지
- 신규 등록만 차단
- 관리자에게 경고 표시
- 초과 상태 배지 표시

이 정책이 현재 없다면 명확히 구현해라.

━━━━━━━━━━━━━━━━━━
7. 수영장관리자 구독 버튼 이동 오류 수정
━━━━━━━━━━━━━━━━━━

현재 수영장관리자에서 구독 버튼 클릭 시 이상한 페이지로 이동하는 문제를 수정한다.

해야 할 일:
- 현재 버튼 onPress / navigation / href / route 이름 전수 확인
- 잘못 연결된 route 제거
- 중복 페이지/예전 페이지/죽은 화면(dead screen) 있으면 제거
- 실제 구독 관리 화면으로 단일 연결
- 탭 구조/스택 구조 꼬임 있으면 정리
- 권한별 진입 경로 분리
  - 수영장관리자: 자신의 구독 관리 화면만
  - 슈퍼관리자: 전체 수영장 구독 관리 화면

화면 이름이 비슷한 예전 파일이 남아 있으면 삭제 또는 완전 비활성화하라.
half-migrated 상태 절대 금지.

━━━━━━━━━━━━━━━━━━
8. 코드 찌꺼기 제거 기준
━━━━━━━━━━━━━━━━━━

다음 조건에 해당하면 기존 코드는 제거 또는 교체한다.

- 예전 플랜명 하드코딩 문자열
- 더 이상 사용하지 않는 임시 subscription state
- 화면마다 따로 계산하는 memberLimit
- RevenueCat과 무관한 fake purchase mock 코드
- 죽은 route / 죽은 screen / 죽은 util
- 서버에는 안 쓰는데 화면에서만 남은 legacy 필드
- manual override와 충돌하는 오래된 billing 로직
- 실제 저장 안 되고 local state만 바뀌는 코드

단,
삭제 전 어디서 참조되는지 먼저 확인하고
사용 중이면 새 구조로 참조처를 옮긴 뒤 제거해라.

“일단 두고 덧붙이는 방식” 금지.
기존 쓰레기 코드 때문에 안 바뀌는 구조면 과감히 걷어내고 새로 짜라.

━━━━━━━━━━━━━━━━━━
9. 구현 방식 지침
━━━━━━━━━━━━━━━━━━

반드시 서비스 레이어를 분리한다.

권장 분리:
- planDefinitions.ts
- subscriptionResolver.ts
- revenuecatService.ts
- adminSubscriptionOverrideService.ts
- poolCapabilityService.ts
- subscriptionWebhookHandler.ts
- subscriptionAuditLogService.ts

핵심 규칙:
- 화면 컴포넌트 내부에서 플랜 계산 금지
- DB row를 바로 화면에 뿌리지 말고 resolver를 거쳐 정규화된 상태로 사용
- 슈퍼관리자 화면 / 수영장관리자 화면 모두 같은 resolver 결과 사용
- 타입 명확히 정의
- nullable / expired / trial / canceled 상태 예외처리 철저히

예시 반환 구조:
{
  source,
  planCode,
  planName,
  status,
  memberLimit,
  storageLimitMb,
  videoEnabled,
  whiteLabelEnabled,
  startsAt,
  endsAt,
  trialEndsAt,
  paymentPlatform,
  isOverMemberLimit,
  effectiveReason
}

━━━━━━━━━━━━━━━━━━
10. 반드시 수행할 테스트
━━━━━━━━━━━━━━━━━━

수정 후 아래 시나리오를 실제로 검증해라.

[기본 시나리오]
1) free 수영장 진입 → free 정보 표시 확인
2) 수영장관리자 구독 버튼 클릭 → 올바른 구독 화면 이동
3) Premium100 결제 성공 → 즉시 플랜 반영
4) 앱 재실행 후에도 Premium100 유지 확인
5) restore purchases 동작 확인

[슈퍼관리자 수동 부여]
6) 슈퍼관리자가 free → Premium100 수동 변경
7) 회원 한도 / 저장용량 / 상태배지 즉시 반영 확인
8) Premium100 → Premium1000 수동 변경
9) 회원 한도 및 저장용량 확대 반영 확인
10) trial 부여 및 trial 종료일 반영 확인

[충돌 시나리오]
11) RevenueCat 결제 상태가 있는데 슈퍼관리자 override 적용
12) 화면/DB/권한이 override 기준으로 일치하는지 확인
13) override 해제 시 RevenueCat 상태로 정상 복귀 확인

[제한 시나리오]
14) 회원 수가 플랜 한도 초과일 때 신규 회원 등록 차단 여부
15) 저장용량 초과 시 업로드 차단/경고 여부

12. 작업 방식 제한
━━━━━━━━━━━━━━━━━━

- 임시 하드코딩 금지
- UI만 수정하고 끝내기 금지
- console.log로만 확인하고 종료 금지
- 기존 코드에 덧대기만 하는 방식 금지
- 구조가 꼬였으면 새 서비스 레이어로 분리하고 기존 찌꺼기 제거
- 수정 후 반드시 실제 클릭/변경/반영 테스트까지 완료
- 테스트 안 한 기능을 “완료”라고 보고하지 말 것

“구독 관련 모든 상태가 어디서 바뀌고 어디에 반영되는지 한눈에 설명 가능한 구조”로 만드는 것이다.

반드시 코드 기준으로 확인하고,
자동 반영이 실제로 되는 상태까지 만든 뒤,
보고서와 함께 제출하라.
