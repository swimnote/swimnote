[스윔노트 보강 정책 기능 구현 + 관리자/선생님 보강 리스트 연동 최종 지시서]

지금부터 “보강 정책” 기능을 실제 작동 기준으로 구현한다.
이번 작업은 단순 버튼 연결이 아니라
수영장 관리자 설정 → 결석 처리 → 보강 생성 → 보강 완료 → 기간 만료 분기까지
전체 흐름이 자동으로 이어지게 만드는 작업이다.

중요:
- 현재 보강 정책 버튼이 잘못된 화면에 연결되어 있다면 반드시 실제 보강 정책 설정 화면으로 다시 연결할 것
- UI만 만들고 실제 계산/날짜 적용/리스트 반영 안 되는 상태 금지
- 관리자 모드와 선생님 모드에 동일하게 적용되어야 함
- 기능끼리 꼬이지 않게 현재 출결/보강/학생/시간표 구조와 자연스럽게 연결할 것
- 수정 후 반드시 실제로 결석 처리, 날짜 계산, 보강 생성, 만료 이동까지 테스트하고 보고할 것

==================================================
1. 구현 목표
==================================================

보강 정책은 플랫폼 공통 고정값이 아니라
수영장마다 다르게 정할 수 있어야 한다.

핵심 목표:
1. 수영장 관리자가 보강 정책을 직접 설정
2. 결석 처리 시 정책에 따라 자동으로 보강 리스트 생성 여부 판단
3. 보강 유효기간 자동 계산
4. 보강 가능 횟수 초과 시 더 이상 생성 안 함
5. 보강 완료 시 별도 “보강 출석 완료” 상태 반영
6. 기간 만료 시 자동으로 “보강 기간 만료 회원” 리스트로 이동
7. 관리자/선생님 모드에서 동일 정책이 동일하게 적용
8. 날짜 계산과 달력/리스트 반영까지 실제 작동

==================================================
2. 잘못 연결된 버튼 수정
==================================================

현재 “보강 정책” 버튼이 다른 화면에 연결되어 있다면
반드시 실제 보강 정책 설정 페이지로 다시 연결한다.

대상:
- 수영장 관리자 모드 내 보강 정책 버튼
- 관련 하위 진입 버튼/카드/메뉴도 전부 실제 연결 확인

빈 화면, 임시 화면, 다른 메뉴 연결 금지.

==================================================
3. 보강 정책 설정 화면 구현
==================================================

위치:
수영장 관리자 모드 > 수업 설정 또는 보강 정책 메뉴

메뉴명:
보강 정책

화면에서 수영장 관리자가 직접 설정할 수 있게 한다.

--------------------------------------------------
3-1. 보강 가능 기간 설정
--------------------------------------------------

결석한 회원이 언제까지 보강을 받을 수 있는지 설정하는 항목 구현.

설정 방식:
단일 선택 + 직접 입력 지원

기본 옵션:
- 해당 월 말일까지
- 다음 달 말일까지
- 제한 없음
- 결석일 기준 직접 입력 N일 이내

직접 입력 예:
- 20일
- 30일
- 60일
- 90일

저장 방식:
예시 필드
- make_up_expiry_type = end_of_month / end_of_next_month / unlimited / custom_days
- make_up_expiry_days = nullable integer

동작:
- 결석일을 기준으로 실제 만료일 계산 가능해야 함
- 화면에 설명만 쓰지 말고 실제 시스템 계산에 반영할 것

--------------------------------------------------
3-2. 보강 가능 횟수 설정
--------------------------------------------------

회원의 수업 빈도별로 결석 시 몇 번까지 보강 기회를 줄지 직접 설정할 수 있어야 한다.

최소 설정 대상:
- 주 1회 회원
- 주 2회 회원
- 주 3회 회원
- 필요 시 주 4회 이상 확장 가능 구조

각 항목별 입력:
- 주 1회 회원 보강 가능 횟수
- 주 2회 회원 보강 가능 횟수
- 주 3회 회원 보강 가능 횟수

예:
- 주 1회 회원 → 보강 2회 가능
- 주 2회 회원 → 보강 4회 가능
- 주 3회 회원 → 보강 5회 가능

중요:
플랫폼이 고정값 강제 금지.
수영장 관리자가 직접 숫자를 입력하거나 선택할 수 있게 구현.

저장 예시:
- make_up_limit_weekly_1
- make_up_limit_weekly_2
- make_up_limit_weekly_3

==================================================
4. 결석 처리 시 보강 자동 생성 규칙
==================================================

선생님 또는 시스템이 학생을 결석 처리하면
즉시 아래 기준으로 보강 가능 여부를 계산해야 한다.

판단 기준:
1. 해당 학생의 수업 유형(주 1회 / 주 2회 / 주 3회 ...)
2. 수영장이 설정한 보강 가능 횟수
3. 현재 월 또는 정책 기준 기간 내 이미 생성된 보강 기회 수
4. 보강 유효기간 정책

동작 규칙:
- 남은 보강 가능 횟수가 있으면 보강 리스트 생성
- 남은 보강 가능 횟수가 없으면 보강 리스트 생성 안 함
- 무제한 생성 금지

예시 1:
- 주 1회 회원
- 보강 허용 2회
- 해당 기간 결석 3회
→ 보강 리스트 2개만 생성

예시 2:
- 주 2회 회원
- 보강 허용 4회
- 해당 기간 결석 5회
→ 보강 리스트 4개만 생성

중요:
결석할 때마다 무조건 보강으로 넘기지 말고,
정책 기준과 현재 누적 보강 생성 수를 비교해서 결정할 것.

==================================================
5. 보강 만료일 자동 계산
==================================================

결석 발생 시 보강 가능 만료일을 자동 계산해야 한다.

예시:
- 결석일: 2026-03-10
- 정책: 결석일 기준 30일 이내
→ 만료일: 2026-04-09

- 결석일: 2026-03-10
- 정책: 해당 월 말일까지
→ 만료일: 2026-03-31

- 결석일: 2026-03-10
- 정책: 다음 달 말일까지
→ 만료일: 2026-04-30

- 정책: 제한 없음
→ 만료일 없음(nullable)

저장 필드 예시:
- make_up_created_at
- make_up_expire_at
- make_up_status

중요:
날짜를 화면에만 보여주는 게 아니라
실제로 리스트 분기와 상태 이동에 반영되어야 한다.

==================================================
6. 보강 리스트 구조 개편
==================================================

관리자 모드와 선생님 모드 모두
기존 보강 관련 화면에 탭 구조를 만들어 아래 리스트를 분리한다.

필수 탭:
1. 보강 가능 회원
2. 보강 완료 회원
3. 보강 기간 만료 회원

현재 구조에 맞게 최소한 기존 보강 리스트 안에 새 탭으로 추가하라.

--------------------------------------------------
6-1. 보강 가능 회원
--------------------------------------------------
조건:
- 보강 생성됨
- 아직 사용 안 함
- 만료일 안 지남

표시 항목 예시:
- 학생명
- 반명
- 원래 결석일
- 보강 생성일
- 보강 만료일
- 수업 유형(주 1회/주 2회 등)
- 남은 보강 가능 여부

--------------------------------------------------
6-2. 보강 완료 회원
--------------------------------------------------
조건:
- 보강 수업에 실제 출석 완료
- 상태 = completed

표시 항목 예시:
- 학생명
- 보강 수업 날짜
- 처리 선생님
- 보강 출석 완료 표시

--------------------------------------------------
6-3. 보강 기간 만료 회원
--------------------------------------------------
조건:
- 보강 생성됨
- 사용 안 함
- 만료일 지남

표시 항목 예시:
- 학생명
- 결석일
- 만료일
- 만료 사유
- 만료 처리일

중요:
이 탭은 관리자와 선생님 모두에게 보여야 한다.

==================================================
7. 보강 완료 처리
==================================================

보강 수업에 학생이 실제 출석하면
그 출석은 일반 출석이 아니라 “보강 출석 완료”로 저장/표시되어야 한다.

동작 방식:
- 시간표에서 해당 보강 수업 슬롯에는 “보강” 배지 표시
- 선생님은 출석 버튼을 누르더라도
  시스템은 해당 건을 일반 출석이 아닌 보강 출석으로 기록
- 보강 리스트에서는 “보강 완료 회원” 탭으로 이동

저장 예시:
- attendance_type = regular / make_up
- make_up_status = available / completed / expired

중요:
일반 수업 출석과 보강 출석이 섞이지 않게 구분할 것.

==================================================
8. 관리자/선생님 동일 적용
==================================================

보강 정책은 관리자 설정이지만
실제 운영은 선생님이 하기 때문에 두 모드 모두 동일하게 반영되어야 한다.

관리자 모드:
- 보강 정책 설정 가능
- 보강 가능/완료/만료 리스트 조회 가능

선생님 모드:
- 결석 처리 가능
- 보강 자동 생성 결과 확인 가능
- 보강 가능/완료/만료 리스트 조회 가능
- 보강 출석 완료 처리 가능

중요:
정책은 한 곳에서 관리하고,
결과는 관리자/선생님 두 화면에서 동일하게 봐야 한다.

==================================================
9. 데이터 구조 및 상태값
==================================================

현재 구조를 해치지 않는 선에서
보강 관련 데이터를 명확히 관리한다.

필수 상태 예시:
- available
- completed
- expired

권장 필드 예시:
- student_id
- class_id
- pool_id
- absence_date
- make_up_created_at
- make_up_expire_at
- make_up_completed_at
- make_up_status
- created_by
- completed_by
- source_attendance_id
- attendance_type

중요:
결석 원본과 보강 생성 기록이 연결되어 있어야 한다.
그래야 누가 어떤 결석으로 인해 보강이 생성됐는지 추적 가능하다.

==================================================
10. UI/UX 규칙
==================================================

- 보강 정책 화면은 설정값이 한눈에 보이게 정리
- 직접 입력 숫자 필드는 실제 저장/반영되어야 함
- 날짜 계산 결과를 필요 시 안내 문구로 보여줘도 좋음
- 보강 리스트 탭은 깨지지 않게 안정적으로 배치
- 모바일 화면에서 탭/카드/리스트 잘림 금지
- 데이터 없는 경우 빈 상태 안내 문구 표시
  예: “현재 보강 기간 만료 회원이 없습니다”

==================================================
11. 수정 금지 영역
==================================================

다음은 이번 작업에서 절대 깨지면 안 된다.

- 기존 출결 기본 구조
- 기존 반배정 구조
- 기존 학생/회원 연결 구조
- 기존 학부모 화면 연결
- 기존 관리자 권한 경계
- 기존 선생님 수업 처리 흐름

이번 작업은 “보강 정책 확장”이지
전체 출결 시스템을 갈아엎는 작업이 아니다.

==================================================
12. 테스트 및 검증 필수
==================================================

작업 후 반드시 직접 테스트하고 결과를 항목별로 보고하라.
확인하지 않은 기능은 완료라고 하지 마라.

최소 테스트 항목:
1. 보강 정책 버튼이 실제 정책 화면으로 연결되는지
2. 보강 가능 기간 설정 저장이 되는지
3. 결석일 기준 만료일 계산이 정확한지
4. 주 1회/주 2회/주 3회 보강 횟수 제한 저장이 되는지
5. 결석 처리 시 제한 횟수 안에서만 보강 생성되는지
6. 제한 횟수 초과 시 추가 보강이 생성되지 않는지
7. 보강 가능 회원 탭에 정상 반영되는지
8. 보강 완료 처리 시 완료 탭으로 이동하는지
9. 기간 경과 시 만료 탭으로 이동하는지
10. 관리자 모드/선생님 모드 모두 동일하게 보이는지
11. 일반 출석과 보강 출석이 구분되는지
12. 기존 출결/반배정/학생 기능이 깨지지 않았는지

[작업 완료 보고 형식]
1. 구현 완료 기능 목록
2. 수정한 파일 목록
3. 실제 테스트 완료 항목
4. 아직 추후 보완 필요한 부분
5. 기존 기능 영향 여부
6. 남아 있는 제한사항

거짓 완료 보고 금지.
실제 시뮬레이션 안 했으면 안 했다고 적어라.

==================================================
13. 최종 목적
==================================================

이번 작업의 목적은
수영장마다 다른 보강 규칙을 실제 운영에 맞게 설정하고,
그 정책이 결석/보강/완료/만료 흐름 전체에 자동 적용되도록 만드는 것이다.

핵심:
- 보강 정책 실제 설정 가능
- 결석 시 자동 계산
- 날짜/달력 반영
- 횟수 제한 반영
- 관리자/선생님 동일 적용
- 보강 완료/기간 만료 분리
- 기존 시스템과 충돌 없음

이 기준으로 꼼꼼하게 구현하라.
