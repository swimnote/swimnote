[우선 마감 지시서 - growth 잔재 완전 제거 + 선생님 스케줄러 기타수업 저장 오류 긴급 수정]

지금부터 아래 2가지를 최우선으로 동시에 처리한다.

1. 300명 플랜의 growth 잔재를 프로젝트 전체에서 완전히 제거하고
   표시명/로그/리포트/이벤트 문구를 전부 ADVANCE(어드밴스)로 통일한다.

2. 선생님 모드에서 스케줄러 진입 후 “기타수업” 저장 시 발생하는 오류를
   실제 저장/반영/재조회까지 정상 작동하도록 긴급 수정한다.

중요:
- 이번 작업은 단순 텍스트 수정이 아니다
- growth 잔재는 화면/로그/API 응답/리포트/이벤트 기록까지 전수 제거
- 기타수업 저장 오류는 UI만 닫히는 척하는 수정 금지
- 실제 DB 저장, 리스트 반영, 재조회 성공까지 확인해야 완료
- iPhone Expo Go 기준으로도 반드시 확인
- 거짓 완료 보고 절대 금지

==================================================
1. growth 잔재 완전 제거
==================================================

최종 기준:
- 300명 플랜의 사용자 표시명은 “어드밴스”
- 영문 표시가 필요하면 “ADVANCE”
- growth / GROWTH / 성장형 등 과거 표현은 전부 제거

--------------------------------------------------
1-1. 전수 검색 대상
--------------------------------------------------

프로젝트 전체 검색:
- growth
- GROWTH
- Growth
- 성장형
- tier=growth
- subscription_tier=growth
- event_logs 내 growth 문구
- revenue_logs 표시명
- billing 관련 plan_name 매핑
- super 정산/상품설정/분석 카드
- 수영장 관리자 구독관리
- 결제 성공/변경/환불 메시지
- 시뮬레이션 출력 문자열
- seed/mock 데이터
- constants / enum / label map / tier map

--------------------------------------------------
1-2. 구분 원칙
--------------------------------------------------

내부 ID:
- plan_id = swimnote_300 유지 가능

표시명:
- 반드시 “어드밴스” 또는 “ADVANCE”

금지:
- 사용자/운영자 화면에 growth 노출
- 로그/리포트 문구에 growth 노출
- 테스트 출력 문구에 growth 노출

--------------------------------------------------
1-3. 수정 후 확인
--------------------------------------------------

반드시 확인:
- 슈퍼관리자 구독 상품 설정
- 슈퍼관리자 매출/정산 관리
- 수영장 관리자 구독관리
- 결제/업그레이드/환불 로그
- 시뮬레이션 보고서 출력 문자열

중요:
검색 결과에서 growth 문자열 0개가 될 때까지 반복 확인
(내부 식별자 plan_id=swimnote_300 제외)

==================================================
2. 선생님 모드 스케줄러 > 기타수업 저장 오류 긴급 수정
==================================================

현재 문제:
- 선생님 모드에서 스케줄러로 들어감
- 기타수업 저장 시 오류 발생
- 저장 실패 / 앱 오류 / 상태 꼬임 가능성 있음

이번 목표:
- 기타수업 저장이 정상 작동
- 저장 후 스케줄러에 반영
- 재조회 시 유지
- Expo Go에서도 멈춤 없이 동작

--------------------------------------------------
2-1. 반드시 먼저 확인할 것
--------------------------------------------------

아래를 우선 점검:
- 기타수업 저장 API 엔드포인트
- 요청 payload 필드 누락 여부
- pool_id / teacher_id / class_group_id / date / time 필수값 누락 여부
- “기타수업”용 분기 처리 로직 존재 여부
- 일반 수업 저장 로직과 기타수업 저장 로직이 섞여 있는지
- null/undefined 상태에서 저장 버튼 눌러지는지
- 저장 후 라우트/바텀시트/모달 충돌 있는지
- Expo Go에서만 터지는지, API 자체가 실패하는지 분리해서 확인

--------------------------------------------------
2-2. 진단 로그 강제
--------------------------------------------------

반드시 로그 찍어서 확인:
- 저장 버튼 클릭 직전 payload
- 요청 URL / method
- 응답 status / body
- 저장 성공 후 state update
- 저장 성공 후 refetch 실행 여부
- 저장 실패 시 정확한 에러 메시지

중요:
어디서 실패하는지 정확히 잡아라
(프론트 유효성 / API 4xx / API 5xx / 저장 후 렌더링 오류 / 네비게이션 오류)

--------------------------------------------------
2-3. 수정 원칙
--------------------------------------------------

- 기타수업 저장에 필요한 필드가 일반 수업과 다르면 명확히 분기
- 저장 성공 시 즉시 스케줄러 반영
- 저장 후 팝업/바텀시트 정리
- 저장 실패 시 사용자에게 의미 있는 오류 문구 표시
- 저장 중 중복 클릭 방지
- 저장 후 홈으로 튀지 않기
- Expo Go 기준으로 멈춤/터치불가 없게 할 것

--------------------------------------------------
2-4. 저장 성공 후 기대 동작
--------------------------------------------------

기타수업 저장 성공 시:
- DB 저장 완료
- 해당 날짜/시간 또는 리스트에 즉시 반영
- 다시 들어와도 남아 있음
- 선생님 스케줄러 월간/주간/일간 어디서든 필요 시 확인 가능
- 뒤에 바텀시트 잔상 없음

==================================================
3. 함께 점검할 것
==================================================

기타수업 저장 오류 수정하면서 아래도 같이 확인:
- 일반 수업 저장은 안 깨졌는지
- 반배정 변경 진입 시 팝업 잔상 없는지
- 날짜 선택 → 시간 선택 → 저장 흐름이 끊기지 않는지
- 뒤로가기 규칙 안 깨졌는지
- 월간/주간/일간 전환 시 저장된 기타수업이 정상 보이는지

==================================================
4. Expo Go 우선 확인
==================================================

웹 기준 완료 금지.
반드시 iPhone Expo Go에서 확인:

- 선생님 로그인
- 스케줄러 진입
- 기타수업 입력
- 저장
- 저장 후 반영
- 뒤로가기
- 재진입 후 유지 확인

조금이라도 멈춤/프리징/버튼 무반응 있으면 바로 수정

==================================================

중요:
