지금부터 새 기능 추가를 전면 중단하고, “DB 이원화 전 마지막 안정 버전”을 기준으로 현재 버전의 전체 기능 복구 감사(Audit) 작업을 수행해라.

핵심 목표는 DB 이원화 자체를 더 진행하는 것이 아니라,
데이터베이스 분리 이후 깨진 저장/조회/권한/모드 전환/페이지 연결을
이원화 전 마지막 안정 버전의 동작 기준에 맞춰 다시 복구하는 것이다.

중요:
- 부분 땜질 금지
- 화면만 열리게 하고 저장 안 되는 상태 금지
- mock 데이터로 정상처럼 보이게 만들기 금지
- “진입 성공”만 체크하지 말고 반드시 저장/수정/삭제/목록 반영까지 확인
- 확인 못한 것을 완료로 쓰지 마라

━━━━━━━━━━━━━━━━━━
1. 기준 원칙
━━━━━━━━━━━━━━━━━━

기준 버전:
- 데이터베이스 이원화 적용 직전, 리플릿에서 마지막으로 안정적으로 동작하던 버전

현재 작업 목표:
- 그 안정 버전의 화면 구조 / 모드 구조 / 아이콘별 기능 연결 / 페이지 진입 흐름 / 저장 동작을
  현재 DB 이원화 버전 위에 다시 맞춰 복구한다

즉,
“DB 이원화는 유지하되, 기능 동작은 안정 버전 수준으로 되돌린다”

━━━━━━━━━━━━━━━━━━
2. 현재 확인된 주요 장애
━━━━━━━━━━━━━━━━━━

반드시 우선 분석/복구할 항목:

1. 일지/피드백에서 글 수정 후 저장이 안 됨
2. 관리자 모드 알림설정 진입 시 설정 화면이 아니라 “알림이 없습니다”가 나옴
3. 주간 스케줄러가 사라졌거나 진입/표시가 깨짐
4. 관리자가 선생님 모드로 가서 반 개설하려고 하면 “소속된 수영장이 없습니다”가 나옴
5. DB 이원화 후 전체적으로 저장/조회/권한/소속 연결이 꼬임
6. 각 모드별 아이콘/페이지/기능 연결이 안정 버전과 달라졌을 가능성 높음

━━━━━━━━━━━━━━━━━━
3. 이번 작업 방식
━━━━━━━━━━━━━━━━━━

이번 작업은 버그 하나씩이 아니라 아래 4단계로 진행해라.

1단계. 안정 버전과 현재 버전 비교
- DB 이원화 전 마지막 안정 버전의 화면 구조, 라우트, API 연결, 저장 흐름을 정리
- 현재 버전과 비교해서 달라진 파일과 로직을 찾기
- 특히 auth, role, pool_id, teacher, notification, scheduler, diary 저장 흐름을 비교

2단계. 전체 기능 감사표 작성
- 모드별
  - 슈퍼관리자
  - 관리자
  - 선생님
  - 학부모
- 각 모드의 아이콘/메뉴/페이지별로
  - 진입 가능 여부
  - 목록 조회 가능 여부
  - 저장 가능 여부
  - 수정 가능 여부
  - 삭제 가능 여부
  - 관련 API 정상 여부
  - DB 저장 반영 여부
  를 표로 정리

3단계. 핵심 기능 우선 복구
우선순위:
A. 저장이 안 되는 기능
B. 모드 전환/권한/소속 문제
C. 주간 스케줄러/알림설정/일지피드백
D. 나머지 메뉴

4단계. 항목별 재검증
- 진입만 확인하지 말고
- 실제 입력 → 저장 → 재조회 → 수정 → 반영 여부까지 확인

━━━━━━━━━━━━━━━━━━
4. 반드시 점검할 공통 원인
━━━━━━━━━━━━━━━━━━

모든 깨진 기능에서 아래 항목을 공통 점검해라.

- 현재 화면이 super DB를 봐야 하는지 pool DB를 봐야 하는지
- 테이블명/컬럼명이 변경되었는데 프론트가 옛 이름을 쓰는지
- save/update API가 pool DB로 저장한 뒤 응답 구조가 바뀌었는지
- 프론트가 여전히 예전 응답 shape를 기대하는지
- currentPool / pool_id / user_pool / teacher profile / role mapping이 누락됐는지
- 권한 가드가 너무 일찍 return 하는지
- 진입은 되는데 저장 API body 필드가 틀린지
- 저장 성공 후 목록 재조회 또는 state 갱신이 안 되는지
- 빈 상태 UI가 설정 화면을 덮어쓰는지
- 모달/페이지 라우팅이 잘못 연결됐는지

━━━━━━━━━━━━━━━━━━
5. 우선 복구 대상 상세
━━━━━━━━━━━━━━━━━━

A. 일지 / 피드백 저장
반드시 확인:
- 일지 작성 API
- 피드백 기본설정 저장 API
- 수정 버튼 클릭 이벤트
- 저장 request body
- API 응답
- DB 반영
- 저장 후 목록 재조회

목표:
- 글 작성 후 저장
- 수정 후 저장
- 다시 들어가도 내용 유지

B. 알림설정
현재 “알림이 없습니다”가 뜨는 것은
설정 화면과 알림 목록 화면 연결이 꼬였을 가능성이 높다.
반드시 확인:
- 관리자모드 알림설정 아이콘이 어느 route로 가는지
- 그 route가 설정 페이지인지, 알림 목록 페이지인지
- notification_settings / push_settings / pool_push_settings 중 무엇을 읽는지
- 빈 상태 메시지가 설정 화면을 잘못 대체하는지

목표:
- 알림설정 아이콘 → 설정 화면으로 정확히 이동
- 설정값 조회/저장 가능

C. 주간 스케줄러
반드시 확인:
- 월/주/일 탭 구조
- 주간 스케줄러 route 존재 여부
- class_groups / classes / teacher schedule 데이터 조회
- 주간 모드 렌더링 조건
- 스케줄 데이터 없을 때도 주간 UI는 보여야 함

목표:
- 주간 스케줄러 다시 표시
- 빈 상태여도 UI 유지
- 반 개설/보강 버튼 정상 동작

D. 관리자 → 선생님 모드 / 반 개설
현재 “소속된 수영장이 없습니다”는
관리자 계정의 teacher 연결 또는 pool_id 연결 누락 가능성이 높다.
반드시 확인:
- 관리자 가입 시 user_pools 생성 여부
- teacher/staff row 생성 여부
- switch-role API 응답
- teacher me/profile API
- teacher route guard
- 반 개설 시 어느 테이블의 pool_id를 참조하는지

목표:
- 관리자 계정이 선생님 모드 진입 가능
- 반 개설 시 자기 수영장 소속으로 정상 동작

━━━━━━━━━━━━━━━━━━
