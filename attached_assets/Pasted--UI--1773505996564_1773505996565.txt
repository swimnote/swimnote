지금부터는 화면만 수정하지 말고 실제 데이터 구조와 삭제 로직을 포함해서 수정해줘.
임시 UI 수정 말고, 관리자/선생님/학부모 모든 권한 화면이 같은 원본 데이터를 바라보는 구조로 고쳐줘.

[핵심 원칙]
1. 반 삭제와 회원 삭제 버튼은 실제로 작동해야 한다.
2. 삭제는 hard delete가 아니라 서비스 정책에 맞는 soft delete / archive 로직으로 처리한다.
3. 다른 로그인 권한 화면까지 전부 동일하게 반영되어야 한다.
4. 각 화면이 제각각 mock 데이터나 개별 useState를 쓰지 말고 공통 저장소(DB/store)를 기준으로 렌더링해야 한다.
5. 수정 후 내가 바로 테스트할 수 있게 테스트 데이터와 체크리스트도 같이 제공해라.

1. 삭제 정책 정의
==================================================

[A. 반 삭제 정책]
- 반 삭제 시 회원은 삭제하지 않는다.
- 해당 반에 속한 회원들은 모두 “미배정” 상태로 변경한다.
- 즉:
  - students.classId = null
  - students.className = null
  - students.assignmentStatus = 'unassigned'
  - 필요한 경우 scheduleSlots / assignedDays / assignedTimes 도 반 기준 연결값 제거
- 반 자체는 삭제 처리하되, 참조 무결성 깨지지 않게 한다.
- 가능하면 classes.isDeleted = true 또는 deletedAt 저장 방식으로 soft delete 처리
- 삭제된 반은 기본 목록에서 보이지 않게 하고, 관리자 통계에서도 제외
- 선생님 로그인 화면에서도 해당 반은 완전히 사라져야 한다.

[B. 회원 삭제 정책]
- 회원 삭제 시 완전 삭제하지 말고 “삭제회원” 보관 영역으로 이동시켜라.
- 즉, activeStudents 목록에서는 제거하되 기록은 archiveStudents 또는 students.status='deleted' 로 보존한다.
- 삭제 후에도 학부모 계정은 탈퇴시키지 않는다.
- 학부모는 추후 자녀의 수업정보/기록을 확인할 수 있어야 한다.
- 따라서 회원 삭제는 “운영목록에서 제외 + 기록보존” 개념이다.
- 회원 삭제 시:
  - students.status = 'deleted'
  - students.deletedAt 저장
  - students.active = false
  - 현재 반배정 해제
  - 출결 입력 대상에서는 제외
  - 선생님 학생목록에서는 제외
  - 관리자 기본 회원목록에서는 제외
  - 별도 “삭제회원” 탭/필터에서만 보이게 처리
- 학부모 계정과 연결은 유지하되,
  - parentChildLinks는 유지
  - 학부모 화면에서는 “현재 수강중 아님” 또는 “삭제회원/보관회원” 상태 표시
  - 과거 수업정보, 공지, 기록 조회는 가능하게 유지

3. 반 삭제 기능 구현
==================================================

반관리 화면의 삭제 버튼이 실제로 동작하게 수정해라.

반드시 구현할 것:
- 삭제 버튼 onPress / onClick 실제 연결
- classId 정확히 전달
- 삭제 전 confirm 모달 표시
- confirm 문구:
  “이 반을 삭제하면 소속 회원은 미배정 상태로 변경되고, 선생님/출결/스케줄 화면에서도 반영됩니다. 삭제하시겠습니까?”

[반 삭제 트랜잭션 순서]
1. 삭제 대상 class 조회
2. 해당 classId를 가진 active 학생 조회
3. 학생 전원을 미배정 상태로 업데이트
   - classId null
   - className null
   - assignmentStatus 'unassigned'
   - 필요 연결 schedule/classSlot 제거
4. teacherClassMappings 또는 관련 캐시 정리
5. class는 hard delete 하지 말고 isDeleted=true, deletedAt 저장
6. 모든 화면에서 재조회 또는 글로벌 store 업데이트
7. 성공 토스트 출력
8. 실패 시 롤백 또는 에러 표시

[반 삭제 후 즉시 반영되어야 하는 화면]
- 관리자 > 반관리 목록
- 관리자 > 회원관리 목록의 반배정 상태
- 관리자 > 대시보드 반 통계
- 선생님 > 내 반 목록
- 선생님 > 오늘 스케줄
- 선생님 > 출결 대상
- 학부모 > 자녀 반 정보

4. 회원 삭제 기능 구현
==================================================

회원관리 화면의 삭제 버튼이 실제로 동작하게 수정해라.

[회원 삭제 전 confirm 문구]
“이 회원은 운영 목록에서 제거되고 삭제회원으로 보관됩니다. 학부모 계정은 유지되며 기존 수업정보는 보존됩니다. 진행하시겠습니까?”

[회원 삭제 처리 순서]
1. 대상 student 조회
2. student를 hard delete 하지 말고 상태 변경
   - status='deleted'
   - active=false
   - deletedAt 저장
   - archivedReason='member_deleted'
3. 현재 반배정 해제
   - classId null
   - className null
   - assignmentStatus='unassigned' 또는 'deleted'
4. 출결 대상에서 제외
5. 선생님 반 학생목록에서 제외
6. 관리자 기본 회원목록에서 제외
7. 삭제회원 보관목록에서 조회 가능하게 처리
8. 학부모 연결은 유지
9. 학부모 화면에서는 과거 정보 조회 가능하게 처리

[회원 삭제 후 즉시 반영되어야 하는 화면]
- 관리자 > 회원 목록
- 관리자 > 삭제회원 목록 또는 필터
- 관리자 > 반관리 인원수 재계산
- 선생님 > 학생목록
- 선생님 > 출결대상
- 학부모 > 자녀 목록 / 자녀 상세
  (현재 수강중 아님 또는 보관회원 상태 표시)

5. 공지사항 사진 전송 문제 수정
==================================================

현재 공지사항 메시지 발송 시 사진이 전송되지 않는 문제를 수정해라.

반드시 구현:
- 공지 작성 시 text만 저장하지 말고 imageUrls 배열도 함께 저장
- 이미지 업로드 성공 후 공지 저장
- 저장 실패 시 공지 발송 실패 처리
- 관리자 공지 목록/상세에서 이미지 표시
- 학부모 공지 목록/상세에서도 이미지 표시
- 여러 장이면 전부 렌더링
- 카드 목록에는 대표 썸네일 또는 이미지 개수 표시
- 앱 재실행/새로고침 후에도 이미지가 유지되어야 함

점검할 것:
- 이미지 업로드 스토리지 경로
- 업로드 후 URL 저장 누락 여부
- 공지 데이터 저장 시 imageUrls 미포함 문제
- 학부모 상세화면에서 렌더링 누락 문제

==================================================
6. 가장 중요한 구조 수정
==================================================

현재 앱이 작동 안 하는 이유가 아래 중 하나일 가능성이 높다. 전부 점검하고 수정해라.

- 버튼 onPress / onClick 미연결
- 삭제 함수는 실행되지만 id 전달 오류
- 화면에서만 제거하고 DB에는 반영 안 함
- 관리자/선생님/학부모가 서로 다른 mock 데이터를 사용 중
- soft delete 했는데 조회 필터가 잘못돼서 계속 보임
- 반 인원수 계산이 캐시값이라 갱신 안 됨
- 공지사항 이미지 URL을 저장하지 않음
- 공지 상세 렌더러가 text만 보여주고 images는 무시함

반드시 구조를 이렇게 맞춰라:
- 단일 데이터 소스 사용
- repository/service 계층으로 CRUD 분리
- 화면 컴포넌트에서 직접 임시배열 수정 금지
- 삭제/수정 후 refetch 또는 글로벌 상태 동기화
- 모든 권한 화면이 같은 source를 바라보게 수정

권장 구조 예시:
- /services/classService.ts
- /services/studentService.ts
- /services/announcementService.ts
- /store/useAppStore.ts 또는 context/provider
- /screens/admin/*
- /screens/teacher/*
- /screens/parent/*

그리고 수정할 때 절대 하면 안 되는 것:
- 화면에서만 사라지게 하는 fake delete
- 학생을 완전 삭제해서 기록까지 날리는 것
- 학부모 계정까지 같이 삭제하는 것
- 반 삭제 시 회원까지 삭제하는 것
- 관리자/선생님/학부모 각각 다른 임시배열을 쓰는 것

이 작업은 UI 수정이 아니라 운영 가능한 MVP 구조 수정이다.
코드는 실제 동작 기준으로 작성하고, mock 처리한 부분이 있으면 어디인지 명확히 표시해라.
