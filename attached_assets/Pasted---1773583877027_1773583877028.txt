회원관리 연결구조를 전체적으로 다시 설계하고 구현해줘.
이번 수정은 수영장 관리자 / 선생님 / 학부모가 같은 학생 데이터를 서로 다른 권한으로 연결해서 사용하는 구조로 재정비하는 작업이다.
기존 화면 일부만 덧붙이는 방식이 아니라 회원관리 흐름 전체를 운영형 구조로 리팩토링해줘.

핵심 목표:
1. 학생 회원을 중심으로 관리자-선생님-학부모가 연결되는 구조로 재설계
2. 관리자 등록 경로와 학부모 선가입 경로를 모두 지원
3. 선생님은 자기 반 학생만 명확하게 조회하고 출결/일지/사진/오늘스케줄에 자연스럽게 연결
4. 학부모는 자기 자녀만 조회 가능
5. 관리자만 회원 등록/반 배정/수업횟수 설정 가능
6. 회원 목록에서 주1/주2/주3, 수업요일/시간, 학부모 연결상태, 반배정 상태를 한눈에 볼 수 있게 구현
7. 데이터 저장 구조는 실제 DB 기준으로 안정적으로 연결
8. 기존 다른 기능과 충돌하지 않게 공통 상태 구조로 정리

전체 운영 구조:
학생 회원이 중심이다.
- 관리자: 학생 생성, 학부모 연결 승인, 주 몇 회 설정, 반 배정, 수정
- 선생님: 본인 반에 배정된 학생만 조회, 출결/일지/사진/오늘스케줄에 활용
- 학부모: 승인된 자기 자녀만 조회

회원 등록 경로는 2개:
A. 관리자 직접 등록
B. 학부모가 먼저 앱 가입 후 자녀 정보 입력 → 승인 요청

최종적으로는 두 경로 모두 같은 StudentMember 구조로 합쳐져야 한다.

반드시 구현할 흐름 1: 관리자 직접 등록
1. 수영장 관리자가 학생을 먼저 등록할 수 있어야 함
2. 승인 절차 없이 학생 회원은 바로 생성
3. 입력값:
   - 학생 이름
   - 출생년도
   - 보호자 전화번호
   - 보호자 이름(선택)
4. 저장 즉시 StudentMember 생성
5. 상태는 pending_parent_link 또는 active 로 관리
6. 등록 완료 후 문자 초대문구를 자동 생성
7. 초대문구에는
   - 수영장명
   - 학생 이름
   - 앱 다운로드 안내
   - 초대코드
   - 접속 링크
   가 포함되게 해줘
8. MVP에서는 실제 문자 API 대신
   - 초대문구 보기
   - 복사하기
   - 공유하기
   버튼으로 먼저 구현
9. inviteCode, inviteMessage는 상태/DB에 저장 가능하도록 구조를 열어둬

반드시 구현할 흐름 2: 학부모 선가입 요청
1. 학부모가 앱에서 먼저 가입할 수 있어야 함
2. 등록된 수영장을 검색하거나 선택
3. 아이 이름 / 출생년도 / 전화번호 입력 후 승인 요청
4. 이 요청은 ParentJoinRequest 로 저장
5. 관리자는 회원관리 > 학부모 요청 승인에서 확인
6. 관리자는 요청을 보고
   - 기존 학생과 연결
   - 새 학생 생성 후 연결
   중 하나를 선택 가능하게 해줘
7. 승인 완료 후 parentUserId 가 StudentMember 에 연결되게 해줘
8. 학부모는 자기에게 승인된 학생만 조회 가능하게 API 레벨에서 제한

권한 규칙:
1. 관리자만 학생 생성 가능
2. 관리자만 주1/주2/주3 설정 가능
3. 관리자만 반 배정 가능
4. 선생님은 반 배정 불가
5. 선생님은 자기 담당 반 학생만 조회 가능
6. 학부모는 자기 자녀만 조회 가능
7. 선생님 개인메모와 별도로 학생 공용 데이터만 학부모와 연결

중복 체크 규칙:
1. 같은 수영장 안에서 학생 이름 + 출생년도 + 보호자 전화번호가 같으면 중복 가능성 경고
2. 완전 동일하면 바로 저장하지 말고 모달 표시
3. 모달 버튼:
   - 기존 회원 연결
   - 새 회원으로 계속 등록
   - 취소
4. 중복 검사는 관리자 직접 등록 / 학부모 요청 승인 둘 다 공통 적용

회원관리 화면 구조:
회원관리 화면은 현재 카드만 있는 형태를 실제 운영용으로 바꿔줘.
상단 구조는 기존 공통 고정 레이아웃 시스템에 맞춰줘.

상단:
- 제목: 회원 관리
- 검색창
- 상단 액션 버튼 2개
  1. 어린이 직접 등록
  2. 학부모 요청 승인

상태 필터:
- 전체
- 미배정
- 연결대기
- 배정불일치
- 주1
- 주2
- 주3
- 학부모 연결완료

목록 카드에 반드시 표시:
- 학생 이름
- 출생년도
- 주1/주2/주3 배지
- 수업 라벨 (예: 월4 · 목7)
- 학부모 연결 상태
- 반 배정 상태
- 담당 선생님 이름 또는 수
- 등록 경로(admin_created / parent_requested)
- 배정불일치 경고가 있으면 빨간 배지

배지 색 규칙:
- 주1 = 파랑
- 주2 = 초록
- 주3 = 보라
- 연결대기 = 회색 또는 주황
- 미배정 = 주황 또는 빨강
- 배정불일치 = 빨강
- 학부모 연결완료 = 초록 작은 배지

회원 상세/배정 화면:
회원 목록에서 이름을 누르면 상세 화면으로 이동

상세 화면에 표시:
- 학생 이름
- 출생년도
- 보호자 이름
- 보호자 전화번호
- 학부모 연결 상태
- 등록 경로
- 주 몇 회
- 현재 반 배정 목록
- 수업 라벨
- 담당 선생님
- 초대문구 보기/복사
- 반배정 수정 버튼

반배정 규칙:
1. 관리자만 가능
2. 주1/주2/주3 먼저 선택
3. 선택 횟수만큼 반 선택 가능
4. weeklyCount 와 선택 반 수가 다르면 저장 불가
5. 같은 시간대에 같은 학생 중복 배정 불가
6. 저장 시 students / classAssignments / teacherClasses / teacherSchedule / attendance 대상에 즉시 반영
7. 페이지 새로고침 없이 즉시 상태 갱신

수업 라벨 표기 규칙:
- 월요일 16:00 → 월4
- 목요일 19:00 → 목7
- 토요일 09:00 → 토9
이 형식으로 scheduleLabels 저장

선생님 연동 구조:
1. 선생님은 자기 teacher_user_id 로 연결된 반만 조회
2. 선생님 내반 탭에는 자기 반 학생만 보여야 함
3. 선생님 출결 탭은 자기 반 학생만
4. 선생님 수영일지 탭도 자기 반 기준
5. 선생님 사진/영상도 자기 반 기준
6. 오늘 스케쥴 탭도 자기 반 기준
7. 회원이 반 배정되거나 이동되면 선생님 화면에 즉시 반영

관리자 연동 구조:
1. 관리자는 전체 학생 조회 가능
2. 관리자 화면에서 회원 수정/반배정/학부모 승인 요청 처리 가능
3. 관리자 화면과 선생님 화면이 같은 학생 데이터를 공유하되 권한만 다르게 적용
4. 반 이동 시 기존 assignedClassIds 정리까지 정확히 처리

학부모 연동 구조:
1. 학부모는 자기 parentUserId 에 연결된 학생만 조회
2. 여러 자녀가 있을 경우 다중 조회 가능
3. 학부모는
   - 출결
   - 수영일지
   - 사진/영상
   - 공지
   만 조회 가능
4. 다른 학생 데이터는 절대 조회 불가
5. 서버 API 레벨에서 차단

배정불일치 규칙:
아래 상황은 배정불일치로 분류
- 주2인데 반이 1개만 배정
- 주3인데 반이 2개 이하만 배정
- weeklyCount 와 assignedClassIds 수가 다름
- assignedClassIds 는 있는데 scheduleLabels 가 비어 있음
- 반은 배정됐지만 teacherId 연결이 빠진 경우
목록과 필터에서 모두 보이게 해줘

연결대기 규칙:
- pending_parent_link 상태이거나
- parentUserId 가 없으면 연결대기
- 필터로 바로 볼 수 있게 해줘

미배정 규칙:
- assignedClassIds 가 비어 있거나
- weeklyCount 는 있는데 실제 classAssignments 가 부족하면 미배정으로 분류

정렬/검색 규칙:
- 기본 정렬은 최근 등록순
- 검색은 학생 이름 / 보호자 이름 / 전화번호 지원
- 필터 변경 시 상단 영역 크기나 카드 크기가 변하지 않게
- 상단 고정 / 하단 목록 스크롤 구조 유지

안정성 규칙:
1. 필수값 없으면 저장 불가
2. 출생년도 형식 검증
3. 전화번호 형식 검증
4. weeklyCount 보다 많이 선택 불가
5. weeklyCount 보다 적게 선택 후 저장 시 경고
6. 이미 같은 시간대 반에 같은 학생 중복 배정 불가
7. 반 이동 시 이전 반 흔적 남지 않게 classAssignments와 assignedClassIds 모두 정리

API/DB 구현 요구:
1. 현재 실제 운영 중인 PostgreSQL 구조와 충돌 없이 구현
2. students, class_groups, attendance, parent_students, users 구조와 자연스럽게 연결
3. 새 테이블이 필요하면 최소화해서 추가
4. 가능하면 기존 students 중심으로 확장
5. 실제 저장/조회가 되는 구조로 구현
6. 앱 재실행 후에도 데이터 유지되게 DB 저장 기준으로 구현

구현 방식:
- Expo React Native + TypeScript
- Express + PostgreSQL + Drizzle ORM 구조 유지
- 현재 mock 카드형 회원관리 화면을 실제 운영형 구조로 교체
- 공통 컴포넌트 분리
  - MemberListCard
  - MemberFilterBar
  - MemberRegisterModal
  - ParentRequestPanel
  - MemberDetailSheet or Screen
  - ClassAssignmentPicker
- 상단 고정 / 목록 독립 스크롤
- 관리자 / 선생님 / 학부모 권한별 조회 API 분리 또는 공통 서비스 레이어로 정리

반드시 마지막에 점검해줘:
1. 관리자 직접 등록 → 회원 목록 반영 되는지
2. 학부모 요청 승인 → 학생 연결 되는지
3. 주1/주2/주3 저장 후 목록 배지 반영 되는지
4. 반 배정 후 선생님 내반/출결/오늘스케쥴에 반영 되는지
5. 학부모는 자기 자녀만 조회 가능한지
6. 반 이동 시 이전 배정이 남지 않는지
7. 앱 재실행 후 데이터 유지되는지

최종 결과:
- 관리자-선생님-학부모가 같은 학생 데이터를 역할별로 정확히 연결해서 사용
- 회원 등록과 반 배정 흐름이 실제 수영장 운영 방식에 맞게 동작
- 선생님과 학부모가 헷갈리지 않도록 표시와 권한이 명확
- 한 번 수정으로 운영 가능한 수준의 회원관리 구조로 정리
