현재 프로젝트를 관리자 모드 중심으로 대대적으로 업데이트해줘.

중요:
이 작업은 단순 UI 수정이 아니라 관리자 시스템 구조 재정비다.
기존 프로젝트에서 쓸 수 있는 레이아웃, 스타일, 공통 컴포넌트는 최대한 재사용하되,
정보 구조, 라우팅, 데이터 연결, 권한 처리, 보강 시스템, 선생님/학부모 연결 흐름은 아래 기준으로 전면 재설계해줘.
기존에 만든 화면은 유지 가능한 것은 살리고, 구조적으로 맞지 않는 것은 과감히 교체해도 된다.

======================================
[0. 시스템의 핵심 목적]
======================================

이 앱은 학원 전체 전산 프로그램이 아니다.
이 앱의 본질은 수영장 선생님이 학생을 관리하고 학부모와 소통하는 운영 앱이다.
학부모는 가장 큰 혜택을 받는 사용자이며,
관리자는 전체 흐름을 감독하고 문제 발생 시 수정/삭제/복구/통제가 가능한 권한을 가진다.

즉 구조는 아래 원칙을 따른다.

1. 선생님은 수업 관리와 학부모 전달의 자유도가 높아야 한다.
2. 학부모는 선생님이 올린 정보를 많이 받고 보기 쉽게 받아야 한다.
3. 관리자는 전체를 직접 지시하는 구조가 아니라 감독/통제/복구 권한을 가진다.
4. 관리자 화면은 전체현황 + 개별 선생님 현황 + 회원별 상세 흐름을 볼 수 있어야 한다.
5. 선생님이 관리하는 항목은 앞으로 계속 추가될 수 있으므로 구조는 확장 가능해야 한다.
6. 사진/영상은 중요한 전달 데이터이며 잘 분류되어야 한다.
7. 관리자 개입 이력은 학부모에게 노출하지 않는다.

- 대시보드
- 사람
- 수업
- 커뮤니케이션
- 더보기

기존 하단 탭이 다르면 위 구조로 정리해줘.
빠른메뉴와 하단탭 역할이 겹치지 않게 정리해줘.

======================================
[3. 관리자 대시보드 전면 개편]
======================================

대시보드는 총합과 경고만 보여주는 운영 관제 화면으로 개편해줘.

상단:
- 수영장 이름/브랜드
- Powered by 스윔노트 표기 유지
- 알림 버튼
- 로그아웃 버튼
- 통합 검색창

대시보드 주요 KPI:
- 전체 회원 수
- 전체 학부모 연결 수
- 전체 선생님 수
- 전체 반 수
- 오늘 출석 수
- 오늘 결석 수
- 미작성 수업일지 수
- 레벨 테스트 입력 대기 수
- 보강 대기 수
- 대리 보강 수
- 학부모 요청 미처리 수

대시보드 섹션:
- 오늘 해야 할 일
- 최근 등록 회원
- 최근 수정/삭제 기록
- 선생님별 요약 현황

선생님별 요약 현황 카드 또는 리스트에는 아래 정보가 보이게 해줘.
- 선생님 이름
- 담당 반 수
- 담당 회원 수
- 오늘 출결 처리 수
- 오늘 일지 작성 수
- 보강 대기 수
- 학부모 전달 수

선생님 이름 클릭 시 선생님 운영 현황 허브로 이동

[4. 사람 메뉴 구조]

사람 메뉴는 4개 서브탭으로 구성

- 학부모
- 선생님
- 승인

--------------------------------------
[4-1. 회원]
--------------------------------------

회원 목록은 카드형 또는 모바일 최적화 리스트형으로 구현

각 회원 항목 표시:
- 회원 이름
- 주 수업 횟수
- 현재 반
- 담당 선생님
- 보호자 이름
- 연락처
- 상태 배지

회원 상태 예시:
- 정상
- 미배정
- 불일치
- 휴회

각 회원 행/카드 액션:
- 회원 상세
- 초대 문자
- 삭제

이름 클릭 또는 상세 클릭 시 회원 통합 상세 허브로 이동

--------------------------------------
[4-2. 학부모]
--------------------------------------

학부모 목록 표시:
- 학부모 이름
- 연결된 회원
- 연락처
- 연결 상태

연결 상태:
- 연결완료
- 연결대기
- 미연결

학부모 상세 또는 연결 정보 화면에서 아래 확인 가능:
- 자녀 연결 정보
- 최근 요청 내역
- 최근 수신 공지
- 최근 수신 일지
- 최근 수신 사진/영상

--------------------------------------
[4-3. 선생님]
--------------------------------------

선생님 목록 표시:
- 선생님 이름
- 담당 반 수
- 담당 회원 수
- 오늘 출결 처리 수
- 오늘 일지 작성 수
- 보강 관련 수
- 학부모 전달 수

선생님 이름 클릭 시 선생님 운영 현황 허브로 이동

--------------------------------------
[4-4. 승인]
--------------------------------------

승인 관리 서브탭:
- 학부모 승인
- 선생님 승인

상태:
- 대기
- 승인
- 거절

승인 항목에는 상세보기 가능하게 구현해줘.
승인 전에 연결 대상 정보 확인 가능해야 함.

[5. 회원 통합 상세 허브 신규 설계]

회원 상세 페이지는 이 앱의 핵심 허브다.
route 예시:

상단 요약 카드:
- 회원 이름

- 주 수업 횟수
- 담당 선생님
- 보호자 이름
- 최근 출석
- 결제 상태

탭 구조:
- 기본정보
- 수업정보
- 출결
- 수업일지
- 레벨
- 보강
- 학부모 전달
- 학부모 요청
- 미디어
- 활동로그

--------------------------------------
[5-1. 기본정보]
--------------------------------------
- 회원 이름
- 출생년도
- 연락처
- 보호자 정보
- 특이사항

--------------------------------------
[5-2. 수업정보]
--------------------------------------
- 현재 반
- 주 수업 횟수

- 반 이동 기록

--------------------------------------
[5-3. 출결]
--------------------------------------

- 결석 기록

중요:
결석 처리 시 자동으로 보강 대기 항목이 생성되어야 한다.

--------------------------------------
[5-4. 수업일지]
--------------------------------------
- 선생님 작성 일지
- 개별 피드백
- 수정 여부
- 삭제 여부

--------------------------------------
[5-5. 레벨]
--------------------------------------
- 현재 레벨
- 레벨 변경 기록
- 레벨 테스트 기록

--------------------------------------
[5-6. 보강]
--------------------------------------
이 회원의 보강 관련 모든 흐름을 보여줘.
- 결석일
- 보강 대기 상태
- 보강 예약 상태
- 보강 완료 상태
- 대리보강 기록

예시 기록:

--------------------------------------
[5-7. 학부모 전달]
--------------------------------------
선생님이 학부모에게 전달한 내용 표시
- 수업일지
- 개별 피드백
- 보강 안내
- 사진
- 영상
- 열람 여부

--------------------------------------
[5-8. 학부모 요청]
--------------------------------------
학부모가 보낸 요청 표시

- 상담 요청

[5-9. 미디어]
--------------------------------------

- 영상
- 업로드 선생님
- 업로드 시간
- 공개 범위
- 열람 여부

--------------------------------------
[5-10. 활동로그]
--------------------------------------
관리자 전용
학부모에게 절대 노출하지 않음

표시 항목:
- 수정 기록
- 삭제 기록
- 복구 기록
- 담당 변경 기록
- 보강 이동 기록
- 대리보강 기록

[6. 선생님 운영 현황 허브 신규 설계]
======================================

route 예시:
- /teachers/:teacherId

이 페이지는 관리자가 개별 선생님 운영 상태를 한눈에 보는 허브다.

상단 요약:
- 선생님 이름
- 담당 반 수
- 담당 회원 수
- 오늘 출결 처리 수
- 작성 수업일지 수
- 보강 대기 수
- 대리보강 수
- 학부모 전달 수

탭 구조:
- 담당 회원
- 출결
- 수업일지
- 학부모 전달
- 학부모 요청 처리
- 보강
- 레벨
- 활동 로그

각 탭에서 관리자 가능 액션:
- 열람
- 수정
- 삭제
- 복구

즉 선생님의 운영 결과물을 관리자가 추적/감독/개입할 수 있게 해줘.

======================================
[7. 수업 메뉴 구조]
======================================

수업 메뉴 서브탭:
- 반 관리
- 출결
- 수업일지
- 레벨 테스트
- 보강
- 반배정

--------------------------------------
[7-1. 반 관리]
--------------------------------------
각 반 카드/리스트 표시:
- 반 이름
- 요일
- 시간
- 담당 선생님
- 현재 인원

액션:
- 회원 관리
- 삭제

반 상세 화면에는 다음이 필요:
- 반 정보
- 담당 선생님
- 소속 회원 목록
- 최근 출결
- 최근 수업일지
- 최근 전달 내역

--------------------------------------
[7-2. 반 등록/수정]
--------------------------------------
반 등록 단계:
1. 수업 요일 선택
2. 수업 시간
3. 담당 선생님
4. 정원

--------------------------------------
[7-3. 출결]
--------------------------------------
출석 / 결석 처리

중요:
결석 처리 시 자동으로 보강 대기 생성

--------------------------------------
[7-4. 수업일지]
--------------------------------------
- 반 일지
- 개별 피드백
- 수정/삭제 이력 반영

--------------------------------------
[7-5. 레벨 테스트]
--------------------------------------
- 현재 레벨
- 테스트 기록
- 변경 기록

--------------------------------------
[7-6. 반배정]
--------------------------------------
회원의 정규 반 배정 관리
주의:
보강은 반배정 변경이 아니라 별도 1회성 처리이므로 여기와 섞지 말 것

======================================
[8. 보강 시스템 신규 구현]
======================================

중요:
보강은 정규 반배정 변경이 아니다.
보강은 결석으로 인해 생성되는 1회성 출석 처리 구조다.

보강 메뉴 서브탭:
- 결석자 리스트
- 담당선생님 보강
- 다른선생님 보강
- 완료 기록

--------------------------------------
[8-1. 결석자 리스트]
--------------------------------------
결석한 회원이 자동으로 모이는 리스트

표시:
- 회원 이름
- 원래 반
- 원래 담당 선생님
- 결석 날짜
- 상태

회원 이름 클릭 시 보강 처리 시작

--------------------------------------
[8-2. 담당선생님 보강]
--------------------------------------
회원을 선택하면 해당 담당 선생님의 스케줄표 표시

규칙:
보강 출석 가능한 반만 활성화
활성 조건은 아래 두 개만 본다.
- 시간 가능
- 정원 가능

레벨 판단은 시스템이 강제하지 않는다.
판단은 선생님 몫이다.

활성화된 반을 클릭하면
그 반의 다음 수업 시간에 1회성 보강출석 가능 상태로 등록

중요:
정규 회원 편성으로 넣지 말고 1회성 보강 슬롯으로 넣어라.

--------------------------------------
[8-3. 다른선생님 보강]
--------------------------------------
담당선생님 스케줄이 안 맞는 경우
해당 회원을 다른선생님 보강 리스트로 이동 가능하게 구현

이동 시 기록:
- 원담당 선생님
- 대리 선생님
- 이동 시간
- 처리자

다른선생님은 자기 보강 리스트에서 이 회원을 확인하고
본인 스케줄표에서 가능한 반을 선택해 1회성 보강출석 상태로 등록

--------------------------------------
[8-4. 완료 기록]
--------------------------------------
보강 수업 후 출석 처리 시 자동 완료

자동 처리:
- 보강 종료
- 다음 주 정규 스케줄에 영향 없음
- 슬롯 자동 종료
- 원담당 선생님 기록에 대리보강 이력 남김

예시:
3/18 박선생 15시반 대리보강 완료

--------------------------------------
[8-5. 보강 푸시 알림 관련 준비]
--------------------------------------
나중에 구현할 예정이지만 데이터 구조는 미리 대비해줘.
보강 수업도 일반 수업처럼 학부모에게 푸시 전달 가능해야 한다.
추후 말머리 [보강수업] 표기 지원 가능하도록 메시지 타입 필드 준비

주의:
지금은 푸시 기능 자체를 완성하지 않아도 되지만, 추후 연결 가능한 구조로 설계해줘.

======================================
[9. 커뮤니케이션 메뉴 구조]
======================================

서브탭:
- 공지

[9-4. 앨범/미디어]
--------------------------------------
사진/영상은 중요도가 높다.
회원관리와 전달흐름 양쪽에서 추적 가능해야 한다.

사진/영상 기본 정책:
- 기본 공개 상태 업로드
- 대상은 해당 선생님 + 해당 학부모
- 맨투맨 구조로 업로드되므로 기본값 공개

미디어별 저장 정보:
- 사진 or 영상
- 업로드 선생님
- 대상 회원 또는 반
- 업로드 시간
- 공개 범위
- 열람 여부
- 삭제 여부

======================================
[14. UI/UX 방향]
======================================

- 기존 파스텔 톤과 관리자 앱 느낌은 유지
- 카드 남발하지 말고 리스트형 정보 구조 강화
- 정보 우선순위가 잘 보이게 배치
- 이름 클릭 시 상세 진입이 명확히 보여야 함
- 하단탭 5개 유지
- 대시보드는 합계/경고 중심
- 상세 페이지는 탭 구조로 분리
- 모바일에서 한 손 조작 가능하도록 여백/고정버튼/필터 최적화
- 빠른 액션보다 정보 구조가 먼저 보이게

======================================
[15. 구현 방식]
======================================

중요:
UI 목업 수준으로 끝내지 말고 실제 작동하게 연결해줘.

반드시 연결할 것:
- 버튼 클릭
- 라우팅 이동
- 필터 동작
- 결석 -> 보강 대기 생성
- 보강 리스트 이동
- 담당선생님 보강 등록
- 다른선생님 보강 이동
- 완료 처리 후 기록 남기기
- 회원 상세/선생님 허브 데이터 연결
- 활동 로그 기록
- soft delete/복구

기존 mock 데이터만 쓰지 말고
현재 프로젝트의 저장 구조와 실제 연결 가능한 형태로 정리해줘.
현재 저장 구조가 불안정하면 정리해서 마이그레이션해도 됨.

======================================
[18. 최종 중요 지시]
======================================

지금 목적은 관리자 모드 구조를 싹 정리하는 것이다.
추가 질문으로 흐름을 흐리지 말고 위 기준으로 최대한 한 번에 정리해서 구현해줘.
불명확한 작은 부분은 현재 프로젝트 문맥에 맞게 합리적으로 판단해서 진행해도 된다.
가장 중요한 기준은 아래 4개다.

- 선생님 운영 자유도 유지
- 학부모가 많은 정보를 받는 구조
- 관리자가 전체 흐름을 파악하고 통제할 수 있는 구조
- 보강과 전달 흐름이 꼬이지 않는 구조
