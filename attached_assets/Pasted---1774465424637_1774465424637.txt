[스윜노트 — 테스트 데이터 초기화 + 슈퍼관리자 계정 고정 + 실가입 준비 지시문]

이번 작업은 테스트 흔적을 전부 비우고, 실제 가입 흐름을 검증할 수 있는 깨끗한 상태로 앱을 정리하는 작업이다.  
기능 추가보다 데이터 초기화와 계정/인증 흐름 정리가 우선이다.

━━━━━━━━━━━━━━━━━━
1. 작업 목표
━━━━━━━━━━━━━━━━━━

- 앱 내부 테스트 계정 전부 제거
- 테스트용 학생/학부모/선생님/관리자/수영장 데이터 전부 정리
- 슈퍼관리자 계정은 아래 계정으로 고정
- 앱은 이 상태에서 대기
- 이후 실제 가입으로 흐름 검증 가능하게 준비
- SMS 인증은 아직 미연결 상태이므로 현재 상태를 숨기지 말고 정확히 정리
- 단, 나중에 연결할 SMS 인증 구조는 깨지지 않게 준비

━━━━━━━━━━━━━━━━━━
2. 데이터 초기화 범위
━━━━━━━━━━━━━━━━━━

다음 테스트용 데이터 전부 제거 또는 초기화:

- 테스트 수영장
- 테스트 관리자 계정
- 테스트 선생님 계정
- 테스트 학부모 계정
- 테스트 학생
- 테스트 출결
- 테스트 일지
- 테스트 보강
- 테스트 공지
- 테스트 쪽지/메신저
- 테스트 승인 요청
- 테스트용 연결 데이터
- 테스트용 프로필/설정/푸시토큰
- 테스트용 사진/영상 메타
- 테스트용 스케줄/반배정/반구성
- 테스트용 가입 신청/등록 요청

중요:
- 슈퍼관리자 계정만 남기고 나머지 테스트 데이터는 전부 비운다
- 백업/복구/감사 로그는 운영 확인용이면 보존 가능하되, 테스트 로그인에 쓰이는 계정/운영 데이터는 깨끗하게 정리
- 삭제가 아니라 상태 초기화가 더 안전한 테이블은 상태 초기화로 처리
- 외래키/참조 무결성 깨지지 않게 순서 맞춰 정리

━━━━━━━━━━━━━━━━━━
3. 슈퍼관리자 계정 고정
━━━━━━━━━━━━━━━━━━

아래 계정으로 슈퍼관리자 로그인 가능 상태를 만들 것:

- 이메일: swimnote.admin@gmail.com
- 비밀번호: tmdnlashxm23!

조건:
- 이 계정은 실제 슈퍼관리자 계정으로 고정
- 앱 실행 시 이 계정으로 슈퍼관리자 로그인 정상 동작해야 함
- 역할은 super_admin 계열로 확실히 고정
- 잘못된 activeRole, last_used_role, 이전 세션 찌꺼기 영향 절대 받지 않게 할 것
- 로그인 후 반드시 슈퍼관리자 대시보드로 들어가야 함
- admin/teacher 쪽으로 잘못 연결되면 안 됨

중요:
- 앞으로 이 계정은 유지
- 비밀번호 변경은 이후 관리자모드에서 처리 예정
- 지금 단계에서는 위 계정으로 정상 로그인만 보장

━━━━━━━━━━━━━━━━━━
4. 관리자 계정 없는계정 오류 수정
━━━━━━━━━━━━━━━━━━

현재 문제:
- 관리자 계정이 없는 계정이라고 나옴

수정:
- 위 슈퍼관리자 계정이 실제 로그인 가능하도록 인증/DB/세션/역할 데이터를 전부 맞춘다
- 이메일은 존재하지만 앱 내부 user/profile/role 연결이 깨진 경우까지 전부 점검
- "없는 계정"으로 나오지 않게 수정

점검 대상:
- auth 사용자 존재 여부
- 앱 내부 users/profile 레코드 존재 여부
- super_admin 역할 존재 여부
- 로그인 후 세션 생성 여부
- role 라우팅 정상 여부

━━━━━━━━━━━━━━━━━━
5. 테스트 계정 비운 뒤 앱 상태
━━━━━━━━━━━━━━━━━━

정리 완료 후 앱 상태는 아래처럼 유지:

- 슈퍼관리자 계정만 정상 로그인 가능
- 테스트용 관리자/선생님/학부모/학생 없음
- 실제 사용자가 새로 가입해서 흐름을 처음부터 검증할 수 있음
- 앱은 깨끗한 상태에서 대기

즉,
👉 테스트 계정 잔존 없이 “실가입 시작 전 초기 상태”를 만드는 것이 목표

━━━━━━━━━━━━━━━━━━
6. SMS 인증 관련 처리 원칙
━━━━━━━━━━━━━━━━━━

현재 상태:
- 휴대폰 인증 / SMS 인증번호 발송은 아직 실제 연결 안 됨

이번 작업에서 해야 할 것:

1. 현재 미연결 상태를 숨기지 말고 정확히 확인
2. 실제 가입 흐름에서 SMS가 어디서 막히는지 확인
3. UI/로직이 SMS 연결 전제 때문에 깨지는 부분 있으면 정리
4. 나중에 SMS 서비스 붙였을 때 바로 연결 가능한 구조로 유지
5. 가짜 성공 처리, mock 성공 처리, 인증된 척 넘기기 금지

중요:
- SMS가 아직 안 붙었으면 “실제 인증 완료”처럼 보이게 만들지 말 것
- 대신 연결 지점, 필요한 환경변수, 필요한 provider, 현재 막히는 단계 정확히 보고
- 기능 완성 전까지 fake pass 금지

━━━━━━━━━━━━━━━━━━
7. 실제 가입 검증 준비
━━━━━━━━━━━━━━━━━━

데이터 초기화 후 아래 흐름을 실제로 검증 가능한 상태로 만들 것:

- 수영장 관리자 가입
- 선생님 가입 신청
- 학부모 가입
- 승인/연결 흐름
- 로그인 및 라우팅

단,
SMS 미연결이면 그 지점까지 실제 진행 후 정확히 막히는 단계 보고

예:
- 휴대폰번호 입력 가능
- 인증번호 발송 호출 실패
- provider 미설정
- 서버 응답 없음
- 환경변수 누락

이렇게 실제 원인 기준으로 보고

━━━━━━━━━━━━━━━━━━
8. 금지 사항
━━━━━━━━━━━━━━━━━━

- 테스트 계정을 숨기고 남겨두기
- mock 인증으로 통과시켜놓고 된 것처럼 보고하기
- 슈퍼관리자 계정을 임시 계정으로 바꾸기
- 이전 세션/스토리지 찌꺼기 때문에 다른 역할로 잘못 들어가게 두기
- “정상 작동합니다” 식 보고만 하기
- 실제 DB 확인 없이 초기화했다고 말하기

━━━━━━━━━━━━━━━━━━
9. 실제 검증 필수
━━━━━━━━━━━━━━━━━━

반드시 직접 확인:

1. 테스트 계정/테스트 수영장/테스트 학생 데이터가 실제로 비워졌는지
2. 슈퍼관리자 계정 로그인 성공하는지
3. 로그인 후 슈퍼 대시보드로 정확히 들어가는지
4. 없는 계정 오류 사라졌는지
5. 이전 세션 영향 없이 role 라우팅 정상인지
6. 가입 흐름에서 SMS가 정확히 어디서 막히는지
7. SMS 미연결 상태를 fake로 통과시키지 않았는지

━━━━━━━━━━━━━━━━━━
10. 보고 형식
━━━━━━━━━━━━━━━━━━

반드시 아래 형식으로 제출:

- 수정 파일 경로
- 삭제/초기화한 데이터 범위
- 남겨둔 계정 목록
- 슈퍼관리자 로그인 테스트 결과
- 슈퍼 대시보드 진입 결과
- “없는 계정” 오류 수정 결과
- SMS 인증 현재 막히는 정확한 단계
- 필요한 외부 연결 항목 목록
- 실제 DB 상태 확인 결과

“초기화 완료”, “로그인 정상” 같은 말만 쓰지 말고 실제 결과를 같이 제출할 것
