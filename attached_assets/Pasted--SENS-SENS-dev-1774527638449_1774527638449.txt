[스윔노트 — SENS 연결 구조 검증 요청]

SENS를 리플릿에 연결했다고 해서 완료로 처리하지 않는다.
지금 필요한 것은 “연결됨”이 아니라
“구조가 올바르고, dev/prod 분리가 유지되며, 실제 운영에서 안전하게 동작하는지” 검증하는 것이다.

설명 금지
추측 금지
반드시 코드 / 환경변수 / 실제 테스트 결과로 제출할 것

━━━━━━━━━━━━━━━━━━

1. 현재 SMS provider 구조 제출
   ━━━━━━━━━━━━━━━━━━

아래를 코드 기준으로 제출

* SMS provider 관련 파일 전체 목록
* 파일 경로
* 실제 provider 선택 로직 코드
* dev provider / SENS provider 분기 코드
* production에서 dev provider 차단 코드

반드시 포함:

* 어떤 조건에서 dev를 쓰는지
* 어떤 조건에서 SENS를 쓰는지
* 현재 배포 환경에서 실제 어떤 provider가 선택되는지

━━━━━━━━━━━━━━━━━━
2. 환경변수 구조 제출
━━━━━━━━━━━━━━━━━━

SMS 관련 환경변수 전체 제출

예:

* SENS_ACCESS_KEY
* SENS_SECRET_KEY
* SENS_SERVICE_ID
* SENS_SENDER_PHONE
* NODE_ENV
* SMS_PROVIDER 또는 동등 변수

주의:
민감한 실제 값은 마스킹해도 됨
하지만 변수명, 존재 여부, 로딩 여부는 반드시 보여줄 것

반드시 포함:

* 현재 환경에서 어떤 값이 세팅되어 있는지
* 누락된 변수 없는지
* provider 전환이 환경변수만으로 가능한 구조인지

━━━━━━━━━━━━━━━━━━
3. 인증 요청 흐름 코드 제출
━━━━━━━━━━━━━━━━━━

아래 흐름의 실제 코드 제출

* 인증번호 요청 API
* 코드 저장 로직
* phone_verifications 저장 구조
* hash 저장 여부
* expires_at 저장 여부
* 재요청 제한 여부
* production에서 dev_code 응답 차단 여부

반드시 포함:

* request 처리 코드
* response 처리 코드
* DB 저장 코드

━━━━━━━━━━━━━━━━━━
4. 실제 발송 테스트 결과 제출
━━━━━━━━━━━━━━━━━━

아래 테스트를 실제 수행하고 결과 제출

[정상 발송]

* 실제 번호로 인증 요청
* SMS 수신 여부
* response status
* DB 저장 결과

[잘못된 번호]

* 잘못된 번호 요청
* 에러 응답 내용
* 가짜 성공 여부

[재요청]

* 동일 번호 연속 요청
* 제한 여부
* 응답 결과

[검증]

* 받은 인증번호로 검증 성공 여부
* 틀린 인증번호 입력 시 실패 여부

━━━━━━━━━━━━━━━━━━
5. dev/prod 분리 검증
━━━━━━━━━━━━━━━━━━

반드시 아래를 확인

* production에서 dev_code가 응답되지 않는지
* production에서 실제 SMS만 발송되는지
* dev provider fallback이 숨어 있지 않은지
* 환경변수 누락 시 가짜 성공하지 않는지

실패 조건:

* production인데 dev_code 노출
* SENS 실패 시 조용히 성공 처리
* dev provider가 자동 fallback

━━━━━━━━━━━━━━━━━━
6. 비용/보안 보호 구조 확인
━━━━━━━━━━━━━━━━━━

아래 항목 확인 후 제출

* 동일 번호 요청 제한
* IP 또는 사용자 기준 요청 제한
* 인증번호 만료시간
* 시도 횟수 제한
* brute force 방지
* 로그 기록 여부

없으면 “없음”이라고 명확히 적을 것
있는 척 금지

━━━━━━━━━━━━━━━━━━
7. 최종 구조 판정
━━━━━━━━━━━━━━━━━━

아래 질문에 예/아니오로 답변

* 현재 배포 환경에서 실제 SENS provider가 선택되는가
* production에서 dev provider가 차단되는가
* 인증번호 요청 시 DB 저장이 정상인가
* 인증번호 검증이 정상인가
* 잘못된 번호 처리 시 가짜 성공이 없는가
* 재요청 제한이 있는가
* 운영 중 비용 폭탄 위험이 낮은가

━━━━━━━━━━━━━━━━━━
8. 제출 형식
━━━━━━━━━━━━━━━━━━

반드시 아래 포함

* 관련 파일 목록
* provider 분기 코드
* 환경변수 목록
* 실제 테스트 결과
* DB 저장 결과
* 실패 테스트 결과
* 구조상 위험 요소
* 지금 상태가 운영 가능인지 여부

━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━

* “정상 작동”
* “문제 없음”

이렇게만 쓰는 것 금지

코드 + 테스트 결과 + DB 결과
셋 다 제출할 것
