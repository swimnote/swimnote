[스윔노트 슈퍼관리자 콘솔 구조 보완 및 매출·공지·보안 통합 구현 최종 지시서]

지금부터 슈퍼관리자 콘솔을 아래 기준대로 수정/추가 구현한다.
이번 작업은 UI만 만드는 작업이 아니다.
모든 항목은 실제 데이터 연결, 상태 저장, 필터 반영, 로그 기록, 상세 진입, 검증까지 작동해야 한다.
버튼만 보이고 기능이 없거나, 더미 화면만 있는 상태로 완료 처리하는 것을 금지한다.
작업 완료 전 반드시 직접 실행해서 기능별 점검 결과를 항목별로 보고하라.
확인하지 않은 기능은 완료라고 말하지 마라.

[핵심 개발 원칙]
- Replit 환경에서 실제 작동 기준으로 구현
- Supabase 데이터와 반드시 연결
- 카드/탭/필터/버튼/팝업/상세페이지 전부 실제 동작
- 이벤트 로그 필수 기록
- 상태값은 enum 또는 상수 체계로 관리
- Invalid Date, NaN, undefined 저장 금지
- 데이터 흐름이 꼬이지 않게 단방향 상태 흐름 유지
- 민감 작업은 재인증 구조 필수
- 기능 축소, 임의 생략, 임의 문구 변경 금지
- 물리적으로 말이 되게 설계: 결제일 기준 매출, 월별 지출, 로그 시간 순서, 공지 노출 규칙이 서로 충돌하면 안 된다

2. 매출·정산 관리 페이지 구현
==================================================

이 페이지는 슈퍼관리자 플랫폼 운영 매출을 한눈에 보는 화면이다.
수영장별 상세 결제 관리 화면이 아니라 플랫폼 전체 매출과 지출을 보는 화면으로 구현한다.

[상단 필터 탭]
반드시 구현:
- 주간
- 월간
- 연간

[매출 계산 기준: 매우 중요]
매출은 “결제일 기준”으로 집계한다.
예를 들어 3월 매출이면 3월 1일 00:01부터 3월 말일까지 실제 결제가 발생한 금액만 넣는다.
기준은 활성 상태나 회원 수가 아니라 “실제 결제 시점”이다.

포함:
- 신규 결제
- 구독 갱신 결제
- 업그레이드 결제
- 해당 기간에 실제 결제 완료된 모든 금액

제외:
- 미결제
- 실패 결제
- 단순 가입
- 추정 매출
- 활성 회원 수 기반 계산

즉, 해당 달에 실제 결제된 금액은 전부 매출 계산에 들어가야 한다.
중간 해지가 있더라도 이미 결제된 금액은 해당 결제월 매출에 포함한다.

[매출 영역 표시 항목]
- 선택 기간 총 매출
- 결제 성공 건수
- 결제 실패 건수
- 신규 결제 수
- 구독 갱신 수
- 업그레이드 결제 수
- 해지 수
- 환불 수
- 환불 금액
- 이전 기간 대비 증감률
- 예상 추이(가능하면)

[상품별 매출 분석]
플랜별로 아래 항목 표시:
- 상품명/플랜명
- 해당 기간 결제 건수
- 해당 기간 매출 금액
- 가입자 수(참고값)
- 해지 수(참고값)

예:
- 100명 플랜
- 300명 플랜
- 500명 플랜
- 기타 구독 상품

[지출 영역]
아래 항목을 지출로 반영한다.
정확한 실과금 연동이 가능하면 실제 값 사용, 불가하면 현재 사용량 기반 근사치 계산 구조로 구현하라.

반드시 포함:
- PG 수수료
- 앱스토어/플랫폼 수수료 예상치
- Supabase 데이터베이스 비용
- Cloudflare R2 또는 스토리지 비용
- 트래픽 비용
- 백업 데이터베이스/스토리지 유지 비용
- 기타 운영비(추가 입력 가능 구조)

중요:
내가 플랫폼 운영을 위해 사용하는 데이터베이스 비용도 매월 지출 항목에 반드시 넣어라.
즉, 수영장 데이터만이 아니라 슈퍼관리자 운영 자체에서 발생하는 DB/백업/스토리지 비용도 포함해야 한다.

[순이익 영역]
- 총 매출
- 총 지출
- 순이익 = 매출 - 지출

[지출 설명]
정확한 비용 API가 없으면
- 사용량
- 예상 단가
- 대략 월 비용
이 보이게 구현하고, 추정치라는 표시를 함께 넣어라.

3. 공지·팝업 관리 기능 추가
==================================================

현재 누락된 공지사항 작성 기능을 반드시 추가한다.

위치:
- 운영 콘솔 상단 “운영 콘솔” 제목 오른쪽에 아이콘으로 추가하거나
- 운영 관리 내부에 “공지·팝업 관리” 메뉴를 추가하되
가장 빨리 접근할 수 있는 위치에 배치

메뉴명:
공지·팝업 관리

[공지 작성 기능]
필드:
- 제목
- 내용
- 대상 선택
  - 전체
  - 운영자
  - 선생님
  - 학부모
- 공지 유형
  - 업데이트 예정
  - 일반 안내
  - 서버/장애 안내
  - 특별 공지
- 노출 시작일시
- 강제 확인 여부

[공지 노출 규칙: 매우 중요]
- 최근 공지 1개만 노출
- 지나간 공지는 사용자(운영자/선생님/학부모)에게 저장하지 않음
- 공지 히스토리는 슈퍼관리자만 관리용으로 저장
- 사용자는 최신 공지 1개만 본다
- 이전 공지는 찾아볼 수 없게 한다

[사용자 쪽 팝업 동작]
앱 실행 후 로그인 흐름에서 최신 공지 1개를 팝업으로 띄운다.

버튼:
- 확인
- 다시 보지 않기

동작 규칙:
- 공지가 떠 있으면 다음 화면으로 진행 불가
- 확인 또는 다시 보지 않기 처리 후에만 진행 가능
- 닫기만 하고 넘어가는 구조 금지

노출 규칙:
- 내가 올린 최신 공지 1개만 계속 보여준다
- 사용자가 “다시 보지 않기”를 눌러도, 그 최신 공지가 유지되는 동안은 그 최신 공지를 기준으로 처리
- 지나간 공지는 아예 사용자에게 제공하지 않는다
- 새 공지를 등록하면 무조건 그 새 공지가 최신 공지가 된다
- 사용자는 새 공지만 다시 본다
- 이전 공지들은 슈퍼관리자 보관용으로만 저장한다

중요:
사용자에게 공지 보관함, 지난 공지 목록, 과거 공지 다시보기 기능 만들지 마라.

9. 테스트 및 검증 의무
==================================================

작업 후 반드시 직접 실행하고 아래 항목을 기능별로 검증하라.
검증 없이 완료 보고 금지.

반드시 점검:
- 신규 6번째 아이콘 “매출·정산” 노출 및 진입 여부
- 주간/월간/연간 필터 정상 작동 여부
- 결제일 기준 매출 계산 정상 여부
- 해당 월 실제 결제 금액 전부 반영 여부
- 지출 항목에 DB/백업/스토리지 비용 반영 여부
- 순이익 계산 여부
- 상품별 매출 집계 여부
- 공지·팝업 관리 메뉴 진입 여부
- 공지 작성/수정/저장 여부
- 최신 공지 1개만 노출되는지 여부
- 사용자에게 이전 공지 목록이 안 보이는지 여부
- 강제 확인 동작 여부
- 슈 아이콘 보안 페이지 진입 여부
- 아이디/비밀번호/2차 인증/세션 기능 작동 여부
- 고위험 작업 시 재인증 걸리는지 여부
- 문자 발송 기록 로그 유지 여부
- 복구 실패 문의 자동 연결 여부
- 이벤트 로그 필수 필드 누락 없는지 여부
- Invalid Date / NaN / undefined 저장 여부 점검
- 모바일 UI 깨짐 여부

==================================================
10. 최종 목적
==================================================

이번 작업 목적은 예쁜 관리자 화면이 아니라
“슈퍼관리자가 실제로 플랫폼을 운영할 수 있는 콘솔”을 만드는 것이다.

핵심 목표:
- 결제일 기준 정확한 매출 집계
- 실제 운영 DB/백업 비용까지 포함한 지출 계산
- 최신 공지 1개만 강제 노출하는 공지 체계
- 슈퍼관리자 전용 최고 보안 체계
- 복구 실패 시 즉시 긴급 문의 연결
- 감사/리스크/운영/보안/매출이 한 흐름으로 이어지는 구조

이 기준에서 벗어나지 말고 꼼꼼하게 구현하라.
