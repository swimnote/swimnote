[스윔노트 — 복구 경고 시스템 최종 정리 지시문]

지금 구현된 복구 안정성 보강은 유지하고, 경고 시스템을 운영용 기준으로 정리한다.
구조 변경 금지, 정합성 보강만 수행한다.

━━━━━━━━━━━━━━━━━━
1. pool_id 키 이름 통일 점검
━━━━━━━━━━━━━━━━━━

[목표]
프로젝트 전반에서 수영장 구분 키 이름이 혼용되지 않도록 통일한다.

[작업]
- 다음 이름 사용 현황 전체 점검:
  - pool_id
  - swimming_pool_id
- 테이블별 실제 컬럼명 목록 출력
- 복구 로직, 조회 로직, 무결성 검사 로직에서 어떤 이름을 쓰는지 전수 확인
- 프로젝트 표준 키 이름을 하나로 고정한다.

[원칙]
- 문서 기준 핵심 키는 pool_id
- 실제 DB가 swimming_pool_id를 쓰는 테이블이 있다면:
  - 즉시 전면 변경하지 말고
  - 매핑 규칙을 명시적으로 둔다.
- 복구/조회/권한 필터는 혼용 금지

[보고]
- 테이블별 실제 컬럼명
- 표준 키 이름
- 예외 테이블 목록
- 매핑 처리 방식

━━━━━━━━━━━━━━━━━━
2. 누락 테이블 경고 필터링
━━━━━━━━━━━━━━━━━━

[목표]
복구 제외가 맞는 전역 테이블은 경고에서 빼고,
실제로 복구 대상이어야 하는 누락만 경고로 남긴다.

[작업]
누락 감지 시 아래 3분류로 나눈다.

A. 복구 대상 누락 (경고 유지)
- parent_accounts
- parent_students
- 수영장 운영 테이블
- 수영장 귀속 메시지/일지/보강/학생 관련 테이블

B. 전역 테이블 제외 (경고 제거 또는 ignore 처리)
- users
- subscriptions
- payment_logs
- revenue_logs
- monthly_settlements
- backup_logs
- restore_logs
- 기타 플랫폼 전역 테이블

C. 검토 필요 (경고 유지 + review 태그)
- parent_pool_requests
- student_registration_requests
- swim_diary
- manual_handover_makeups
- 실제 운영 포함 여부 미확정 테이블

[구현]
warning_details JSON 구조를 아래처럼 확장:
{
  "missing_restore_tables": [],
  "ignored_global_tables": [],
  "review_required_tables": [],
  "broken_relations": []
}

━━━━━━━━━━━━━━━━━━
3. parent_accounts / parent_students 처리 확정 반영
━━━━━━━━━━━━━━━━━━

[확정 정책]
- 학부모 계정은 수영장별 소속 계정
- 같은 휴대폰번호라도 pool_id 다르면 별도 계정
- 수영장별 복구 대상 포함

[작업]
- parent_accounts 복구 대상 포함 여부 최종 반영
- parent_students 복구 대상 포함 여부 최종 반영
- 복구 후 해당 pool 데이터만 되살아나는지 테스트

━━━━━━━━━━━━━━━━━━
4. 무결성 검사 설명과 실제 검사 일치화
━━━━━━━━━━━━━━━━━━

[목표]
문서 설명과 실제 검사 SQL 목적을 맞춘다.

[현재 문제]
- class_diary_student_notes 검사가 “학생 연결”로 설명되지만
  실제 체크가 diary_id → class_diaries 이면 설명 불일치

[수정]
다음 검사를 분리한다.

1. class_diary_student_notes.diary_id → class_diaries.id
2. class_diary_student_notes.student_id → students.id
   (student_id 컬럼이 실제 존재할 경우)

[보고]
- 각 검사 항목의 실제 SQL 또는 실제 연결 컬럼
- 검사 목적 설명
- 통과/실패 결과

━━━━━━━━━━━━━━━━━━
5. review_required 테이블 판정 작업
━━━━━━━━━━━━━━━━━━

아래 테이블은 실제 스키마 기준으로 복구 포함/제외를 확정한다.

- manual_handover_makeups
- parent_pool_requests
- student_registration_requests
- swim_diary

각 테이블마다 아래 형식으로 판정:
- 테이블명
- 수영장 귀속 여부
- pool_id 또는 swimming_pool_id 존재 여부
- 복구 포함 / 제외
- 이유

━━━━━━━━━━━━━━━━━━
6. 최종 검증
━━━━━━━━━━━━━━━━━━

[테스트]
1. 수영장 A, B 데이터 생성
2. parent_accounts / parent_students 포함 여부 반영
3. A 일부 삭제
4. A만 복구
5. warning_details 구조 확인
6. ignored_global_tables / review_required_tables / missing_restore_tables 분리 기록 확인
7. broken_relations 0건 확인

[성공 기준]
- 전역 테이블이 경고 노이즈를 만들지 않음
- 실제 복구 대상 누락만 경고로 남음
- 학부모 계정 복구 정책 반영됨
- 무결성 검사 설명과 실제 검사 내용이 일치함

━━━━━━━━━━━━━━━━━━
7. 보고 형식
━━━━━━━━━━━━━━━━━━

반드시 포함:
- 수정 파일 경로
- 키 이름 통일 점검 결과
- 누락 경고 재분류 결과
- review_required 판정표
- 복구 테스트 결과
- restore_logs warning_details 실제 저장값

“정상 작동” 같은 말만 쓰지 말고 실제 DB/로그 결과를 같이 제출할 것
