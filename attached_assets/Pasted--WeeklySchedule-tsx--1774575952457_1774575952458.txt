[스윔노트 — 선생님 모드 주간스케줄러 실제 렌더 경로 추적 및 근본 수정 지시]

현재 문제:
WeeklySchedule.tsx 스타일 변경 보고는 제출되었지만, 실제 앱 화면에는 변경이 반영되지 않았다.

따라서 이번 작업은 “스타일 추가 수정”이 아니라,
선생님 모드 주간스케줄러가 실제로 어떤 파일 경로를 통해 렌더되는지
코드 기준으로 추적해서 정확한 렌더 진입점을 찾는 작업부터 다시 수행한다.

이번 작업의 핵심:
스타일 수정보다 먼저 “실제 렌더 파일”을 찾는다.
틀린 파일을 계속 수정하는 작업 금지.

━━━━━━━━━━━━━━━━━━
1. 1차 목표
━━━━━━━━━━━━━━━━━━

반드시 아래를 먼저 찾는다.

1. 선생님 모드에서 주간스케줄러 화면으로 진입하는 라우트 파일
2. 해당 라우트가 import하는 상위 화면 컴포넌트
3. 그 상위 화면이 실제로 렌더하는 주간스케줄러 컴포넌트
4. 최종적으로 화면에 보이는 보드/리스트/헤더를 그리는 실제 파일

즉,
“어떤 파일을 수정해야 실제 화면이 바뀌는지”
정확한 경로를 추적해서 보고한다.

━━━━━━━━━━━━━━━━━━
2. 절대 금지
━━━━━━━━━━━━━━━━━━

- 추정으로 파일 수정 금지
- WeeklySchedule.tsx만 다시 수정하는 식의 반복 금지
- 비슷한 이름 파일 아무거나 수정 금지
- 보고서만 쓰고 실제 렌더 경로 확인 안 하는 작업 금지

━━━━━━━━━━━━━━━━━━
3. 실제 렌더 경로 추적 방법
━━━━━━━━━━━━━━━━━━

아래 순서대로 수행한다.

1. 선생님 모드 주간스케줄러로 이동하는 route 파일 찾기
   예:
   - app/
   - screens/
   - teacher/
   - (teacher)/
   - timetable/
   - scheduler/
   관련 폴더 전부 검색

2. “WeeklySchedule”, “weekly”, “schedule”, “scheduler”, “timetable” 키워드로 전체 검색

3. 찾은 파일마다 아래를 확인
   - export default 파일인지
   - 실제 화면 컴포넌트인지
   - import되어만 있는지
   - 조건부 렌더링으로 대체되는지
   - 관리자용/선생님용 분리되어 있는지

4. 최종적으로 앱 실행 중인 화면에
   임시로 아주 명확한 식별 텍스트를 삽입해서 실제 렌더 파일을 확인한다

예시:
- 화면 상단에 "REAL_TEACHER_WEEKLY_SCHEDULE"
- 또는 헤더 옆에 "__WS_DEBUG__"

중요:
이 디버그 문구는 “후보 파일마다 하나씩” 넣어서
어느 파일이 실제 화면을 바꾸는지 눈으로 검증한다.

━━━━━━━━━━━━━━━━━━
4. 디버그 확인 방법
━━━━━━━━━━━━━━━━━━

후보 파일에 각각 서로 다른 임시 텍스트를 넣고 실행한다.

예시:
- components/teacher/WeeklySchedule.tsx → "__A__"
- screens/teacher/WeeklyScheduleScreen.tsx → "__B__"
- app/(teacher)/schedule.tsx → "__C__"

실제 앱 화면에 어떤 문구가 보이는지 확인해서
진짜 렌더 파일을 확정한다.

이 작업 없이 스타일 수정 다시 시작하지 않는다.

━━━━━━━━━━━━━━━━━━
5. 실제 수정 원칙
━━━━━━━━━━━━━━━━━━

실제 렌더 파일이 확정되면
그 다음에만 디자인 수정에 들어간다.

수정 기준:
- 기능 / 로직 / 이벤트 / 데이터 fetch 수정 금지
- 스타일 / 레이아웃만 수정
- 단, 스타일이 여러 파일에 분산되어 있으면
  실제 렌더 파일 + 연결된 하위 UI 컴포넌트까지 함께 정리

즉,
경로를 먼저 확정하고
그 다음 실제 화면을 바꾸는 파일만 수정한다.

━━━━━━━━━━━━━━━━━━
6. 같이 확인할 구조
━━━━━━━━━━━━━━━━━━

경로 추적 시 반드시 아래도 함께 확인한다.

- teacher 전용 파일인지
- admin과 공용 컴포넌트인지
- DayBar / SlotList / CardItem 같은 하위 컴포넌트로 분리되어 있는지
- StyleSheet가 현재 파일에 있는지
- theme 파일에서 색이 덮이고 있는지
- inline style이 최종 우선순위를 가져가고 있는지

━━━━━━━━━━━━━━━━━━
7. 근본 수정 방식
━━━━━━━━━━━━━━━━━━

실제 렌더 파일이 맞다면 아래 방식으로 바꾼다.

1. 기존 스타일 소스 위치 파악
   - StyleSheet
   - inline style
   - theme override
   - className / utility 스타일
   중 무엇이 최종 적용되는지 확인

2. 스타일 우선순위 정리
   - inline style이 있으면 우선 제거 검토
   - 중복 스타일 키 통합
   - theme 값이 덮는 경우 해당 theme source 같이 추적

3. 최종 화면 기준으로 수정
   - 상단 헤더
   - 요일 바
   - 시간축
   - 보드 선
   - 수업 카드
   이 5개를 실제 렌더 파일 기준으로 수정

━━━━━━━━━━━━━━━━━━
8. 필수 제출 보고서
━━━━━━━━━━━━━━━━━━

반드시 아래 형식으로 제출한다.

1. 실제 진입 route 파일 경로
2. 실제 렌더 상위 파일 경로
3. 실제 렌더 하위 컴포넌트 경로
4. 디버그 텍스트 삽입 파일 목록
5. 실제 화면에 보인 디버그 문구
6. 최종적으로 “진짜 수정해야 하는 파일” 1개 또는 여러 개
7. 왜 이전 수정이 반영되지 않았는지 원인
8. 그 후 수정한 스타일 파일 경로
9. 변경 후 실제 화면 스크린샷

━━━━━━━━━━━━━━━━━━
9. 금지 표현
━━━━━━━━━━━━━━━━━━

- 찾았습니다
- 수정했습니다
- 완료했습니다
- 정상 작동합니다

위 표현만 쓰지 말고,
반드시 “어떤 파일에 어떤 디버그 문구를 넣었고, 화면에 무엇이 보였는지” 증거로 설명한다.
