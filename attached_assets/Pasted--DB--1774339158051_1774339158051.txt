지금부터 아래 작업을 즉시 진행해라.
질문으로 시간을 끌지 말고, 먼저 코드/환경변수/DB 연결 상태를 점검하고 수정한 뒤 결과를 보고해라.
확인한 것과 추정한 것을 절대 섞지 말고, 직접 확인 못한 것은 “확인 필요”라고 분리해서 적어라.

[절대 전제]
- 이 프로젝트는 어린이 수영장용 “스윔노트”다
- 성인 기능 섞지 말 것
- growth 표기 부활 금지
- 미배정과 학부모 미연결 합치기 금지
- 결제 실패 사유를 선생님/학부모에게 직접 노출 금지
- 사진/영상 파일을 슈퍼관리자 DB에 저장 금지
- 반 제외 시 과거 기록 수정 금지
- 기타수업 다시 넣지 말 것
- mock 데이터로 작동하는 척 금지

[현재 상황]
- 사용자가 Replit Secrets에 POOL DB 정보를 제공했다
- 따라서 이제 “실제 물리 분리”를 기준으로 점검해야 한다
- 기존 fallback은 이미 제거 방향으로 작업되었고, 이제 실제 연결/마이그레이션/검증이 핵심이다

[지금부터 해야 할 최우선 작업]
1. 현재 서버가 실제로 super DB / pool DB를 각각 어떤 URL로 연결하는지 확인
2. POOL_DATABASE_URL이 실제 적용되었는지 확인
3. 서버를 재시작해서 physical DB separation이 실제로 동작하는지 확인
4. pool DB에 필요한 운영 테이블이 실제 생성되었는지 확인
5. super DB에는 super 전용 테이블만, pool DB에는 운영 테이블이 들어가도록 분리 상태 점검
6. pool-event-logger가
   - superAdminDb의 pool_event_logs
   - poolDb의 pool_change_logs
   두 군데에 실제로 각각 기록하는지 검증
7. dead_letter_queue / resendDeadLetter 흐름이 실제 분리 상태에서도 작동하는지 검증
8. /super/db-status/diagnostic
   /super/db-status/verify
   /super/db-status/dead-letters
   /super/db-status/dead-letters/:id/resend
   이 엔드포인트들이 실제 접근 가능한지 확인
9. db-status.tsx에서
   - 슈퍼관리자용 연결상태
   - 유저서비스 연결상태
   2계층이 실제 ping 기반으로 동작하는지 확인
10. 그 다음 반 제외 / 변경 메모 / 주간 스케줄러 작업으로 이어가라

[반드시 확인할 항목]
A. 환경변수
- DATABASE_URL
- POOL_DATABASE_URL
- POOL_DB_PASSWORD
실제로 어떤 값이 어떤 DB에 연결되는지 코드 기준으로 확인해라

B. DB 연결 코드
- superAdminDb 생성 위치
- poolDb 생성 위치
- fallback 제거 여부
- POOL_DATABASE_URL 없을 때 서버가 실제 종료되는지

C. pool DB 마이그레이션
아래 운영 테이블이 실제 pool DB에 있어야 한다

- attendance_logs
- class_journals
- makeups
- level_settings
- notices
- notification_settings
- albums
- photo_assets_meta
- schedule_change_logs
- pool_change_logs
- dead_letter_queue

D. super DB 테이블
아래는 super DB에 있어야 한다
- operators

- pool_event_logs

[중요 구현 원칙]
- 수영장 운영 데이터는 pool DB에 먼저 저장
- 슈퍼관리자는 운영 로그만 복제 저장
- 사진/영상 실제 파일은 DB 저장 금지
- 사진/영상 바이너리는 super DB에 절대 저장 금지
- 결제/구독/정산/OTP/슈퍼보안은 super DB 유지

[실행 순서]
1단계. 코드 확인
- 어떤 파일이 DB 연결을 만드는지 찾고
- 현재 super / pool이 각각 어디를 바라보는지 정리

2단계. 서버 기동 검증
- API server 재시작
- POOL_DATABASE_URL 적용 후 정상 기동 여부 확인
- 미설정 시 실제 크래시하도록 되어 있는지 재확인

3단계. pool DB 마이그레이션 검증
- 누락 테이블 있으면 실제 생성
- super DB에 있어야 할 것과 pool DB에 있어야 할 것이 섞여 있지 않은지 점검

4단계. 이벤트 복제 검증
아래 이벤트를 실제로 발생시키고 어느 DB에 기록되는지 확인
- journal.create
- journal.update
- journal.delete
- subscription.change
- read_only_mode.on
- read_only_mode.off
- schedule.change
- class_assign
필요하면 테스트 데이터 1건씩 넣고 로그 레벨로 확인

5단계. 모니터링 API 검증
- /super/db-status/diagnostic 응답 확인
- /super/db-status/verify 응답 확인
- /super/db-status/dead-letters 응답 확인
- resend API 동작 확인

6단계. UI 검증
- db-status.tsx에서 super / pool 상태가 분리 표기되는지 확인
- “동일 DB 사용” 정적 문구가 남아 있으면 제거
- 실제 분리 ping 결과가 반영되게 수정

7단계. 다음 작업 착수
- 반 제외-미배정 이동 날짜기준 구조
- 변경 적용 주차 dot 표시
- 반 눌렀을 때 변경 메모 표시
- 다음 주 이후는 깔끔한 최종 상태만 표시
- 주간 스케줄러 월~일 / 오전6시~오후9시 / 토요일 잘림 없음 / 일요일 포함 / 마지막 칸 깨짐 없음
- Expo Go 프리징 최우선 점검

[반 제외 로직 절대 기준]
- 과거 기록 수정 금지
- 지금부터 제외하면 지금 이후만 빠짐
- 다음 주부터 제외하면 이번 주는 유지
- 다다음 주부터 제외하면 다음 주까지 유지
- assigned_from / assigned_until / removed_effective_date 구조로 구현
- 변경이 실제 적용되는 주차의 해당 반에만 dot 표시
- 반 눌렀을 때 변경 메모 표시
- 다음 주 이후 스케줄에는 점/메모 없이 최종 상태만 표시

[주간 스케줄러 절대 기준]
- 기타수업 제거 유지
- 월 화 수 목 금 토 일
- 오전 6시 ~ 오후 9시 전체 시간 칸
- 토요일 잘림 없음
- 일요일 추가
- 마지막 칸 깨짐 없음
- 모바일 터치 시 하단 정보 표시
- Expo Go 프리징 의심 코드까지 같이 점검
  - ConfirmModal 외부 이동
  - selectedDateRef 사용
  - useMemo로 dateClassMap O(1) 조회
