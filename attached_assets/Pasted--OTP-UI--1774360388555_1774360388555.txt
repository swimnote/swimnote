지금부터 슈퍼관리자 로그인 후 OTP 입력이 먹히지 않는 버그를 “겉수정”이 아니라 인증 플로우 전체를 끝까지 추적해서 근본 수정해라.

중요:
- 입력창 UI만 만지는 식의 가짜 수정 금지
- “수정 완료”라고 쓰기 전에 반드시 실제 재현 → 수정 → 재검증까지 해라
- 확인 못한 것은 완료로 쓰지 마라
- mock 응답 금지
- 콘솔 로그 한두 줄 보고 끝내지 마라
- 프론트 / API / DB / 세션 / 라우팅을 전부 추적해라

━━━━━━━━━━━━━━━━━━
1. 문제 정의
━━━━━━━━━━━━━━━━━━

현재 문제:
- 슈퍼관리자 계정/비밀번호 입력 후 OTP 입력 단계로 진입
- OTP를 입력해도 인증이 완료되지 않음
- 두 차례 수정 시도 후에도 해결되지 않음

목표:
- 슈퍼관리자 OTP 입력 후 실제 로그인 성공
- 잘못된 OTP일 때만 정확한 실패 메시지
- 올바른 OTP일 때는 세션 생성 후 슈퍼관리자 홈으로 정상 이동

━━━━━━━━━━━━━━━━━━
2. 반드시 점검할 전체 흐름
━━━━━━━━━━━━━━━━━━

아래 경로를 처음부터 끝까지 전수 점검해라.

A. 프론트
- 로그인 화면에서 super admin 계정 입력
- 로그인 요청 payload
- OTP 화면 진입 시 전달되는 값
- OTP 입력값 state
- 입력값 trim 여부
- 숫자만 허용인지
- 길이 제한(예: 6자리) 정상인지
- submit 버튼 disabled 조건
- onPress/onSubmitEditing 실제 호출 여부
- API 요청 body 필드명 확인
- verify 성공 후 navigation 처리 확인

B. API
- 로그인 1차 요청 응답 구조
- OTP verify 엔드포인트 실제 호출 여부
- request body 파싱 결과
- super admin 전용 분기 여부
- challenge_id / otp_session_id / user_id 전달 여부
- secret 조회 로직
- otp code 비교 로직
- expires_at 비교 로직
- 사용완료 처리 로직
- 실패횟수 증가 로직
- 성공 시 세션 발급 로직

C. DB
- super admin OTP 관련 테이블/컬럼 확인
- secret/hashed code 저장 위치 확인
- expires_at 값 확인
- used_at / consumed_at 값 확인
- latest challenge가 맞는지 확인
- 로그인 시 challenge row가 생성되는지 확인
- verify 시 해당 row를 정확히 찾는지 확인

D. 시간/보안
- 서버 시간이 올바른지
- UTC/KST 비교 오류 없는지
- 만료 계산이 초/밀리초 혼동 없는지
- code normalize 여부(공백/하이픈 제거)
- 문자열 비교/숫자 비교 실수 없는지
- secret mismatch 여부
- 환경변수 누락 여부

E. 성공 후 이동
- verify 성공했는데 화면 이동이 안 되는 경우도 포함해 추적
- auth token 저장
- secure storage 저장
- super role 확인
- super home route push/replace 확인
- middleware/guard에서 다시 튕기는지 확인

━━━━━━━━━━━━━━━━━━
3. 반드시 넣어야 할 디버깅 로그
━━━━━━━━━━━━━━━━━━

민감정보 원문은 노출하지 말고 마스킹해서 아래 로그를 추가해라.

프론트 로그:
- login submit 시작
- login 응답 수신
- otp screen 진입 시 전달값
- otp input 현재 길이
- otp verify 요청 직전 payload shape
- otp verify 응답 상태
- verify 성공 후 navigation 호출 여부

서버 로그:
- login request 진입
- super admin 계정 식별 여부
- otp challenge 생성 여부
- verify request 진입
- body 필드 존재 여부
- challenge row 조회 성공 여부
- expires_at 비교 결과
- code comparison 결과(true/false)
- success 시 session 발급 여부
- failure 이유 분기값

중요:
- OTP 원문 출력 금지
- secret 원문 출력 금지
- 마스킹 또는 length만 표시

━━━━━━━━━━━━━━━━━━
4. 우선 의심해야 할 고장 지점
━━━━━━━━━━━━━━━━━━

아래를 우선순위 높게 확인해라.

1) 프론트가 otp 값을 빈값 또는 잘못된 필드명으로 보내는지
2) verify API가 super admin용 challenge를 못 찾는지
3) expires_at 계산이 이미 만료로 처리되는지
4) 사용완료 플래그가 먼저 찍혀 재시도가 막히는지
5) verify 성공 후 세션 저장 또는 화면 이동이 막히는지
6) super admin만 다른 auth guard를 타서 다시 로그인 화면으로 튕기는지
7) 입력값에 공백/자동완성/숫자포맷 문제가 있는지

━━━━━━━━━━━━━━━━━━
5. 수정 방식
━━━━━━━━━━━━━━━━━━

아래 기준으로 수정해라.

프론트:
- OTP 입력값은 trim 처리
- 숫자 외 문자 제거
- 길이 정확히 검증
- 버튼 disabled 조건을 값 길이 기준으로 명확히
- API body 필드명 서버와 완전 일치
- verify 성공 시 token 저장 후 super 홈으로 replace
- 실패 메시지는 “OTP가 올바르지 않거나 만료되었습니다”로 통일 가능

서버:
- verify 로직을 공통 함수로 분리
- super admin OTP 검증이 일반 로그인과 다르면 분기 명확화
- challenge 조회는 최신/미사용/미만료 조건으로 정확히 조회
- 성공 후 used_at 업데이트는 검증 성공 뒤에만 수행
- 만료 비교는 서버 기준 현재시각으로 정확히 수행
- 실패 사유를 code mismatch / expired / challenge not found / already used 로 내부 로그에 명확히 구분

DB:
- 로그인 1차 요청 시 challenge row 생성 확인
- verify 시 해당 row가 사용되도록 연결
- used_at, expires_at, attempts 컬럼 동작 확인

6. 재현 테스트를 반드시 해라
━━━━━━━━━━━━━━━━━━

수정 후 아래를 실제로 전부 검증해라.

TC1. 올바른 super admin 계정 + 올바른 OTP
→ 로그인 성공
→ super home 진입

TC2. 올바른 계정 + 틀린 OTP
→ 실패 메시지 표시
→ 로그인 성공하면 안 됨

TC3. 올바른 계정 + 만료된 OTP
→ 만료 메시지 또는 실패 메시지
→ 새 OTP 재발급 흐름 확인

TC4. OTP 입력창에 공백/문자 포함
→ 자동 정리되거나 유효성 실패

TC5. 성공 후 앱 재시작
→ super 세션 유지 여부 확인

TC6. 성공 직후 route guard
→ 다시 로그인으로 튕기지 않는지 확인

━━━━━━━━━━━━━━━━━━

- super admin login screen
- otp verify screen / modal
- auth api client
- super auth route
- otp generation service
- otp verify route
- auth middleware / guard
- secure storage token save

실제 파일 경로를 답변에 적어라.
