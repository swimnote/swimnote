지금부터 스윔노트 앱 전체를 “최종 논리 설계도” 기준으로 전면 구조 감사(Audit)하라.

이번 작업의 목적은 단순 버그 수정이 아니다.
기능이 화면에 보이는지만 확인하는 것도 아니다.

목표는 아래 3개다.

1. 기능이 설계된 “정확한 위치”에 있는지 확인
2. 그 위치의 기능이 저장/조회/수정/삭제까지 정상 동작하는지 확인
3. 저장된 결과가 다른 모드 및 슈퍼관리자까지 설계 논리대로 충돌 없이 연결되는지 확인

즉, 이번 작업은
“앱 구조 논리 + DB 저장 위치 + 모드 간 연동 + 슈퍼 반영”
전체를 한 번에 검증하는 구조 감사 작업이다.

중요:
- mock 데이터 금지
- 화면만 열리면 PASS 금지
- API 200만 뜨면 PASS 금지
- 저장 안 되는데 화면만 바뀌는 상태 금지
- 확인 못한 기능을 정상으로 쓰지 마라
- 설계도에 있는 구조/위치/책임과 다르면 반드시 FAIL로 기록하라

════════════════════════════════════
0. 절대 기준
════════════════════════════════════

아래 기준은 절대 변경 금지다.

- 스윔노트는 어린이 수영장 운영 플랫폼이다
- 성인 구조 섞지 말 것
- 학부모 중심 구조 유지
- 학부모 모드를 회원 모드로 바꾸지 말 것
- 미배정과 학부모 미연결은 절대 합치지 말 것
- pending은 미배정과 다른 상태다
- 기타수업 기능은 폐기 상태 유지
- 반 제외 시 과거 기록 수정 금지
- 반 삭제 시 과거 기록 삭제 금지
- 사진/영상 파일을 슈퍼관리자 DB에 저장 금지
- 운영 원본은 pool DB 먼저 저장
- 슈퍼관리자는 운영 로그 복제본 저장
- growth 표기 금지
- 300명 플랜은 무조건 어드밴스
- 결제 실패 원인을 선생님/학부모에게 직접 결제 문제로 노출 금지

════════════════════════════════════
1. 이번 감사의 판정 기준
════════════════════════════════════

각 기능은 아래 8단계를 모두 통과해야 PASS다.

1. 설계도상 맞는 메뉴/아이콘/위치에 존재하는가
2. 진입이 되는가
3. 입력/조작이 되는가
4. 저장/수정/삭제 API가 정상 동작하는가
5. 올바른 DB(super 또는 pool)에 저장되는가
6. 같은 모드에서 재조회 시 반영되는가
7. 관련 다른 모드에서도 반영되는가
8. 슈퍼관리자 집계/로그/현황에도 반영되는가

즉,
위치 확인 → 진입 → 입력 → 저장 → DB 확인 → 재조회 → 타모드 확인 → 슈퍼 확인
순서로 검증해라.

화면 열림만 확인하면 FAIL 처리하라.
저장 후 DB 미반영이면 FAIL 처리하라.
DB에 저장됐는데 재조회 안 되면 FAIL 처리하라.
타 모드에 안 보이면 FAIL 처리하라.
슈퍼 로그/집계 누락이면 FAIL 처리하라.
