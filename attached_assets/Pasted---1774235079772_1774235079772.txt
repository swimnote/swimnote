[스윔노트 전체 플랫폼 자동 점검 + 오류 수정 + 구조 정렬 + 기능 보호 통합 지시서]

지금부터 이 프로젝트 전체를 “기능 보호를 최우선으로 한 플랫폼 안정화 작업”으로 진행한다.
이번 작업은 일부 화면 수정이 아니라, 스윔노트 전체 플랫폼을 운영 가능한 수준으로 끝까지 점검하고 오류를 수정하는 작업이다.
대상 범위는 아래 4개 모드 전체다.

- 슈퍼관리자
- 수영장 관리자
- 선생님
- 학부모

중요:
이번 작업의 목적은 “예쁘게 보이게 하는 것”이 아니라
“실제 운영에서 한 번도 안 꼬이고, 모드 간 연결이 안 깨지고, 새 기능을 넣어도 구조 충돌이 없는 상태”를 만드는 것이다.

절대 원칙:
- UI만 흉내 내고 완료 처리 금지
- 버튼만 보이고 작동 안 하는 상태 금지
- 연결된 기능을 건드려서 다른 기능을 깨뜨리는 수정 금지
- 확인하지 않은 기능을 완료라고 보고하는 것 금지
- 한 번 수정하고 끝내지 말고 여러 차례 시뮬레이션과 재점검을 반복할 것
- 단 하나의 오류도 남기지 않는다는 기준으로 진행할 것

==================================================
1. 작업 목표
==================================================

이 프로젝트 전체를 아래 기준으로 완성 수준까지 안정화한다.

핵심 목표:
1. 전체 모드의 메뉴/화면/권한/데이터 연결을 전부 점검
2. 기능 간 충돌, 상태값 불일치, 화면 침범, 잘못된 흐름을 전부 수정
3. 아직 미완성인 기능이 있으면 현재 구조 안에서 자연스럽게 들어갈 수 있는지 호환성까지 점검
4. 새 기능을 추가해도 기존 기능이 깨지지 않는 구조인지 검증
5. 모든 모드에서 실제 운영 시나리오를 반복 시뮬레이션하여 오류가 없어질 때까지 재수정
6. 기능 보호를 최우선으로 하면서 UI/구조/데이터 흐름을 정렬
7. 최종적으로 “실제 유료 운영 가능한 MVP” 수준으로 맞출 것

==================================================
2. 절대 수정 금지 영역
==================================================

다음은 함부로 깨뜨리거나 임의로 갈아엎지 마라.

- 기존 로그인 구조
- 권한 체계의 기본 철학
- 슈퍼관리자 개입 최소화 구조
- 관리자는 읽기 중심, 수업 실무는 선생님 모드 처리 원칙
- 결제일 기준 매출 집계 원칙
- 연기회원 과금 포함 원칙
- 복구는 전체 시점 복구만 허용 원칙
- 학부모 다자녀 구조
- 학생-학부모 연결 구조
- 미배정 → 반배정 흐름
- 공지의 역할 구분
  - 슈퍼관리자 공지: 최신 1개 전체 알림
  - 수영장 관리자 공지: 학부모 쪽지함/공지 누적 구조
- 삭제/탈퇴/퇴원 철학
- 문자 판매는 하지 않지만 문자 발송 로그는 유지하는 원칙

즉,
기존 철학은 건드리지 말고
그 철학이 실제 코드와 UI에서 정확히 작동하게 맞추는 것이 이번 작업의 목적이다.

==================================================
3. 전체 운영 기준(반드시 이 기준으로 점검/수정)
==================================================

[회원 상태 기준]
- 정상회원: 과금 O / 수업 O / 현재 회원수 집계 O
- 연기회원: 과금 O / 수업 X / 현재 회원수 집계 O
- 퇴원회원: 과금 X / 현재 회원수 집계 X / 기록은 남아 있음 / 학부모가 아직 볼 수 있음
- 탈퇴회원: 완전 삭제와 동일 / 복구 불가 / 학부모 조회 불가 / 현재 회원수 집계 X
- 미승인: 접근 불가
- 선생님 승인: 관리자 승인 필요
- 학부모 승인: 학생 정보가 입력되어 있으면 자동 승인 가능
- 학부모 자녀 추가 시: 자녀 이름 + 생년월일 입력 후 자동 연결

[매출 기준]
- 매출은 결제일 기준으로 집계
- 해당 기간 실제 결제 완료된 모든 금액은 매출에 포함
- 예: 3월이면 3월 1일 00:01부터 3월 말일까지 실제 결제된 금액 전부 포함
- 신규 결제 / 구독 갱신 / 업그레이드 결제 포함
- 실패 결제 제외
- 환불은 환불 처리된 달에 마이너스로 차감
- 전달 결제건을 이번 달에 환불하면 이번 달 매출에 마이너스 환불로 표시
- 부분 환불은 일할 계산
- 중도 해지 수수료는 10%
- 연기회원 과금 포함

[지출 기준]
- 매출과 같은 기간 기준으로 계산
- PG 수수료
- 앱스토어/플랫폼 수수료 예상치
- Supabase DB 비용
- Cloudflare R2 또는 저장소 비용
- 백업 DB/스토리지 유지 비용
- 트래픽 비용
- 기타 운영비
- 슈퍼관리자 자체 데이터베이스/백업 운용 비용도 포함
- 정확한 과금 연동이 안 되면 사용량 기반 추정치라도 표시

[삭제/보존 기준]
- 퇴원회원은 기록이 남아 있으나 현재 회원은 아님
- 탈퇴는 완전 삭제와 동일
- 삭제 후 학부모는 예전 기록 조회 불가
- 일지/출결/사진/영상 등 삭제 시 함께 제거
- 수영장 관리자에게는 삭제된 학생 기록을 남기지 않음
- 슈퍼관리자에게는 삭제 로그 파일만 남김
- 삭제 로그는 최소한 누가/언제/무엇을 삭제했는지 추적 가능해야 함

[보강 기준]
- 결석 시 자동 보강 리스트 이동 유지
- 보강 유효기간은 수영장 운영자가 설정
  - 이달 말일
  - 다음 달 말일
  - 제한 없음
  - 직접 일수 입력
- 보강 횟수 제한은 수영장 운영자가 설정
- 보강 완료 시 “보강 출석 완료”가 명확히 표시되어야 함
- 일반 출석과 보강 출석이 구분되어야 함
- 보강 정책이 아직 일부 미완성이라면 현재 구조에 무리 없이 확장 가능한 상태인지 반드시 점검

[권한 기준]
- 관리자는 출결 수정 불가
- 관리자는 수업 조작 불가
- 관리자 화면의 수업/출결 관련 내용은 읽기 전용
- 수업 실무 수정은 반드시 선생님 모드에서 처리
- 이 권한 경계는 절대 무너지면 안 됨

[공지 기준]
- 슈퍼관리자 공지: 최신 1개만 전체 노출 / 과거 조회 불필요 / 강제 확인
- 수영장 관리자 공지: 학부모 쪽지함/공지함에 누적 가능
- 두 공지는 역할이 다르므로 섞이지 않게 분리
- 슈퍼 공지는 새 소식 1개만 노출
- 사용자에게 과거 슈퍼 공지 조회 기능 불필요

[복구 기준]
- 부분 복구 금지
- 전체 시점 복구만 허용
- 복구 전 자동 백업 필수
- 복구 실패 시 실패 사유 표시 + 문의 연결 필수

[우선 수정 순서]
1. 데이터 오류
2. 기능 오류
3. 인터페이스 오류

5. 점검 방식: 반드시 여러 번 반복
==================================================

한 번만 훑지 말고, 아래 순서로 반복한다.

1차 점검:
- 메뉴/탭/버튼/페이지 진입 여부 확인
- 빈 화면, 깨진 링크, 이동 불가 여부 확인

2차 점검:
- CRUD 흐름 확인
- 생성/수정/삭제/조회/상태 변경 확인

3차 점검:
- 모드 간 연결 확인
- 한 모드에서 만든 데이터가 다른 모드에 정확히 반영되는지 확인

4차 점검:
- 권한 경계 확인
- 하면 안 되는 작업이 우회로 가능하지 않은지 확인

5차 점검:
- 운영 시나리오 시뮬레이션
- 실제 사용 흐름 끝까지 따라가며 오류 확인

6차 점검:
- UI 안정화 점검
- 카드 침범, safe area, 스크롤, 탭 겹침, 카드 잘림, 버튼 클릭 불가 여부 확인

7차 점검:
- 새 기능 추가 호환성 점검
- 아직 미완성인 기능이 구조상 어디에 들어가도 기존 기능을 깨뜨리지 않는지 확인

한 번 수정 후 반드시 다시 처음부터 재검증하라.
오류가 사라질 때까지 반복하라.

==================================================
6. 핵심 시나리오 시뮬레이션(반드시 끝까지 돌릴 것)
==================================================

아래 시나리오는 실제로 끝까지 돌려보고 오류가 없는지 확인한다.

[회원/승인/배정]
- 신규 선생님 가입 → 관리자 승인 → 선생님 로그인 → 수업 접근 가능 여부
- 신규 학부모 가입 → 학생 정보 기반 자동 승인 여부
- 학부모 다자녀 추가 → 즉시 연결 → 탭 반영 → 미배정 상태 표시
- 학부모 추가 자녀가 관리자/선생님 화면 미배정 목록에도 보이는지
- 미배정 학생을 선생님이 반에 배정 가능한지

[수업/출결/보강]
- 학생 반배정
- 출결 체크
- 결석 처리
- 자동 보강 리스트 이동
- 보강 배정
- 보강 출석 완료 표시
- 일반 출석과 보강 출석 구분
- 학부모에게 반영

[일지/피드백]
- 선생님이 반 공통 일지 작성
- 개별 학생 추가 코멘트 작성
- 학부모는 자기 아이 기준으로 자연스럽게 읽히는지
- 형제자매 간 내용 섞임 없는지

[매출/결제/지출]
- 실제 결제 발생
- 결제일 기준 매출 반영
- 환불 시 해당 월 마이너스 반영
- 연기회원 과금 포함
- 지출과 매출 기간 기준 일치 여부
- 순이익 계산 가능 여부

[공지]
- 슈퍼관리자 공지 등록
- 최신 1개만 강제 노출
- 확인 또는 다시 보지 않기 후 진행 가능
- 과거 슈퍼 공지 조회 안 되는지
- 수영장 관리자 공지는 학부모 쪽지함/공지함에 누적되는지

[복구]
- 백업 목록
- 복구 미리보기
- 전체 시점 복구만 가능한지
- 복구 실패 시 실패 사유 + 관리자 문의 + 긴급 문의 등록 여부

[삭제/보존]
- 퇴원 처리
- 탈퇴 처리
- 탈퇴 후 조회 불가 여부
- 슈퍼관리자 삭제 로그 남는지
- 수영장 쪽에서는 삭제된 학생 기록이 남지 않는지

[보안]
- 슈퍼관리자 보안 페이지
- OTP/재인증
- 민감 작업 재인증
- 세션/IP/기기 로그
- 고위험 작업 시 추가 인증 여부

==================================================
7. UI 안정화 기준
==================================================

기능을 절대 깨지 않게 유지하면서 UI를 안정화한다.

오류로 간주할 것:
- 카드끼리 겹침
- 카드가 화면 밖으로 잘림
- 탭이 다른 섹션에 가려짐
- safe area 침범
- header/tab에 가려짐
- 텍스트 잘림/줄깨짐
- 버튼이 눌리지 않음
- 이중 스크롤
- 가로 스크롤이 의도 없이 생김
- 모달이 화면 밖으로 나감
- 첫 화면 카드 수정 시 아래 카드들이 위로 밀려 올라가는 현상

허용되는 수정:
- padding
- margin
- gap
- flex/grid 정렬
- min-height
- overflow 구조 정리
- safe area 대응
- 반응형 width/height 조정

금지:
- 기능 로직 수정으로 UI를 억지 해결
- absolute 남용
- z-index 덮어쓰기 남용
- 임의 네비게이션 변경
- API/상태/이벤트 핸들러 변경으로 UI 맞추기

원칙:
UI 수정은 반드시 기능 보호 상태에서만 진행하라.

==================================================
8. 구조 정렬 기준
==================================================

현재 기능 중 아직 덜 만들어진 것이 있더라도
구조 안에서 자연스럽게 추가될 수 있어야 한다.
이걸 반드시 점검하라.

점검할 것:
- 새 기능이 들어갈 위치가 논리적으로 맞는지
- 메뉴 명칭/분류가 운영 흐름에 맞는지
- 같은 개념의 데이터가 여기저기 중복 저장되지 않는지
- 하나의 기능을 추가할 때 다른 모드와의 연결 포인트가 이미 확보되어 있는지
- 상태값 체계가 확장 가능하게 되어 있는지
- 공통 컴포넌트 수정이 다른 화면을 깨뜨리지 않는 구조인지

즉,
“지금 당장 있는 기능”만 보지 말고
“나중에 기능을 넣었을 때도 안 깨지는 구조”인지까지 점검하라.

==================================================
9. 상태값/데이터 정합성 점검
==================================================

반드시 전수 점검:
- 회원 상태가 모드마다 다르게 보이지 않는지
- 학생 상태(미배정/배정/종료 등) 일관성
- 승인 상태 일관성
- 결제 상태와 매출 집계 일치 여부
- 퇴원/탈퇴/삭제 처리 후 남는 데이터 일관성
- 학부모 다자녀 연결 일관성
- 관리자/선생님/학부모 각각 보는 데이터가 같은 원본을 기준으로 하는지

절대 허용하지 말 것:
- 같은 학생이 화면마다 다른 상태로 보임
- 결제 화면과 매출 화면 숫자가 다름
- 삭제됐는데 다른 화면에 남아 있음
- 미배정인데 학부모에 안 보임
- 자녀 탭 전환 시 다른 아이 데이터 섞임

==================================================
10. 권한 경계 점검
==================================================

권한 우회 가능성까지 검사하라.

반드시 확인:
- 관리자는 수업 조작 불가
- 관리자는 출결 수정 불가
- 관리자 화면은 읽기 전용이 맞는지
- 선생님만 수업 실무 수정 가능
- 학부모는 자기 자녀 정보만 접근 가능
- 슈퍼관리자는 실무 대신 로그/복구/결제/보안 중심인지
- 권한 없는 사용자가 직접 URL/상태 변경/버튼 우회로 접근 못 하는지

==================================================
11. 로그/감사/추적 점검
==================================================

모든 민감 작업은 로그가 남아야 한다.

반드시 확인:
- 승인/거절/재승인
- 결제 실패 처리
- 공지 등록
- 삭제
- 퇴원/탈퇴
- 복구 시도/실패
- 보안 설정 변경
- 플래그/정책 변경
- 문자 발송 기록
- 학부모 자녀 추가 연결

로그 필수 필드:
- createdAt
- actor
- target
- summary

추가 확인:
- Invalid Date / NaN / undefined 저장 금지
- 로그 누락 여부
- 삭제 로그가 슈퍼관리자에게 전달되는지

==================================================
12. 구현/수정 원칙
==================================================

- 한 번에 큰 범위를 갈아엎지 말고 원인 단위로 수정
- 공통 컴포넌트 수정 시 모든 화면 영향 확인
- 데이터 오류를 먼저 잡고, 그다음 기능 오류, 그다음 인터페이스 오류를 수정
- 추정으로 수정하지 말고 실제 시뮬레이션 후 수정
- 수정 후 반드시 회귀 테스트
- 한 화면 고친 뒤 다른 화면 깨졌는지 반드시 재확인
- “아마 될 것 같다” 금지
- 실제 실행/클릭/입력/저장/전환/조회까지 확인

==================================================
13. 테스트 완료 기준
==================================================

아래 기준을 충족하기 전까지 완료 처리 금지.

완료 기준:
- 4개 모드 모든 핵심 메뉴 진입 가능
- 모든 핵심 버튼/탭/카드 실제 작동
- CRUD 흐름 정상
- 권한 우회 없음
- 결제/매출/지출 계산 일관성 확보
- 공지 동작 정상
- 보강 흐름 정상
- 복구 흐름 정상
- 학부모 다자녀 전환 정상
- 삭제/퇴원/탈퇴 처리 일관성 확보
- UI 겹침/잘림/가림/버튼 미작동 없음
- 새 기능이 들어와도 구조 충돌 없도록 정렬 완료

==================================================
14. 작업 완료 보고 형식
==================================================

작업이 끝나면 반드시 아래 형식으로 상세 보고하라.

1. 전체 점검한 모드와 화면 목록
2. 발견한 데이터 오류 목록
3. 발견한 기능 오류 목록
4. 발견한 UI/인터페이스 오류 목록
5. 수정한 항목 상세
6. 수정한 파일 목록
7. 실제 시뮬레이션한 시나리오 목록
8. 여전히 남아 있는 미완성 기능 목록
9. 현재 구조상 추가 구현 가능한 기능과 충돌 없는지 여부
10. 아직 추후 결정이 필요한 운영 정책이 있는지 여부
11. 회귀 테스트 결과
12. 최종적으로 남은 오류가 있는지 여부

중요:
- “완료했습니다” 한 줄 보고 금지
- 확인 안 한 것은 “미확인”으로 적어라
- 남은 오류가 있으면 숨기지 말고 적어라
- 실제 실행해보지 않은 것은 테스트 완료라고 쓰지 마라

==================================================
15. 최종 목적
==================================================

이번 작업의 최종 목적은
스윔노트를 단순 프로토타입이 아니라
“실제 유료 운영 가능한 안정적인 어린이 수영 운영 플랫폼 MVP”로 마감하는 것이다.

핵심:
- 전체 모드 자동 점검
- 여러 차례 시뮬레이션
- 데이터 오류 우선 수정
- 기능 오류 수정
- UI 안정화
- 구조 정렬
- 기능 보호
- 추후 기능 추가 호환성 확보

시간이 오래 걸려도 괜찮다.
대충 끝내지 말고, 여러 번 반복해서 단 하나의 오류도 남기지 않는 기준으로 끝까지 점검하고 수정하라.
