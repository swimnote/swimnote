[스윔노트 — 복구 기능 운영 마감 처리 + 다음 단계 준비 지시문]

현재 복구 기능은 운영 가능한 수준까지 검증 완료되었다.
이번 작업은 구조 변경이 아니라 “운영 마감 정리 + 다음 단계 진입 준비” 작업이다.

━━━━━━━━━━━━━━━━━━
1. 복구 기능 최종 마감 처리
━━━━━━━━━━━━━━━━━━

[목표]
복구 기능을 운영 기준 문서/코드/로그 해석까지 마감 가능한 상태로 정리한다.

[작업]

1. restore 응답값 보강
- rows_restored가 0일 경우 reason_message 포함
- 예:
  - "해당 백업 시점에 복구 대상 데이터가 없습니다."
  - "변경/삭제된 대상이 없어 복원 행 수가 0입니다."

2. restore_logs 기록 보강
- rows_restored = 0 인 경우에도
  - status = success 유지
  - reason_message 또는 memo 컬럼/필드에 사유 기록

3. 슈퍼관리자 백업/복구 화면 문구 보강
- 복구 성공
- 경고 없음
- 복원된 행 수
- 0건인 경우 안내 문구 표시

━━━━━━━━━━━━━━━━━━
2. 복구 기능 운영 기준 고정
━━━━━━━━━━━━━━━━━━

[기준 문서화]
다음 내용을 코드 주석 또는 운영 문서에 고정:

- 수영장 운영 데이터 구분 표준 키는 swimming_pool_id
- pool_id는 메타/설정/로그 용도로만 쓰는 예외 테이블 존재
- 수영장별 복구는 swimming_pool_id 기준 운영 테이블만 대상
- users, subscriptions 등 전역 테이블은 ignored_global_tables로 관리
- 감사/이력 로그는 기본 복구 제외
- 복구 성공과 경고는 별개로 본다

━━━━━━━━━━━━━━━━━━
3. 복구 기능 회귀 테스트 세트 저장
━━━━━━━━━━━━━━━━━━

[목표]
다음 수정 때 복구 기능이 다시 깨지지 않도록 재사용 가능한 테스트 절차를 고정한다.

[작업]
다음 시나리오를 테스트 문서 또는 스크립트로 저장:

1. 수영장 A/B 생성
2. 학생/출결/공지 생성
3. 전체 백업
4. pool 백업
5. A 일부 삭제
6. A만 복구
7. B 영향 없음 확인
8. 전체 복구
9. restore_logs 확인
10. warning_details 확인

━━━━━━━━━━━━━━━━━━
4. 다음 단계 진입 준비 — 출결/보강 안정화 시작
━━━━━━━━━━━━━━━━━━

이제 다음 작업 우선순위는 출결/보강 안정화다.
아래 규칙을 기준으로 현재 코드와 UI를 점검한다.

[출결]
- 아무것도 안 누르면 출석
- 결석 클릭 시 즉시 absent 저장
- 결석 즉시 makeup_sessions 생성
- 출석 복구 시 makeup_sessions.status = cancelled
- 취소된 보강은 모든 대기 리스트에서 숨김

[보강]
- waiting / completed / cancelled / expired / transferred 상태 확인
- 기타보강:
  - 다른 선생님에게 보내기 → handover_to_teacher, 선택 선생님 기타 +1
  - 보강 소멸 → self_extinguish, 현재 선생님 기타 +1
- 같은 makeup_session_id 중복 처리 금지
- 정산은 반드시 reason 로그 먼저 기록 후 반영

[UI]
- 선생님:
  - 출석/결석 = 즉시 처리
  - 반이동 = 시트
  - 상세 = 페이지 이동
- 관리자:
  - 수정/삭제 = 팝업/시트
  - 조회 = 페이지

━━━━━━━━━━━━━━━━━━
5. 다음 작업 보고 형식
━━━━━━━━━━━━━━━━━━

- 정산 로그 생성 여부
