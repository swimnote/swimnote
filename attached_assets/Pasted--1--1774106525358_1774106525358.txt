지금부터 슈퍼관리자 모드를 실제 플랫폼 운영 콘솔 기준으로 전면 구현해줘.
이번 작업은 메뉴 화면을 예쁘게 만드는 게 아니라,
플랫폼 운영자가 대규모 운영자(수영장/1인 코치/대관팀/프랜차이즈)를
한눈에 보고, 위험을 감지하고, 바로 조치할 수 있는 운영 콘솔을 만드는 작업이다.

반드시 아래 기준으로 구현해줘.

==================================================
[0. 최상위 원칙]
==================================================

1. 슈퍼관리자는 플랫폼 운영자다.
- 수영장 내부 수업/일지/출결을 직접 운영하는 역할이 아니다.
- 플랫폼 전체 승인 / 정책 / 구독 / 저장공간 / 리스크 / 로그 / 고객센터 / 긴급통제를 담당한다.

2. 슈퍼관리자는 "보안 이슈"와 "기능 이슈"만 보는 것이 아니라
실제 운영 중 터질 수 있는 모든 리스크를 다뤄야 한다.

반드시 다뤄야 하는 리스크 축:
- 보안 리스크
- 기능/품질 리스크
- 개인정보/정책 리스크
- 결제/환불/분쟁 리스크
- 저장공간/업로드 리스크
- 데이터 삭제/복구 리스크
- 운영 실수 리스크
- 고객센터 대응 리스크
- 외부 서비스 장애 리스크
- 악용/남용 리스크

3. 문제 대응 방식은 롤백만으로 해결하지 않는다.
반드시 아래 4개를 같이 갖춰야 한다.
- 기능 플래그 ON/OFF
- 읽기전용 전환
- 핵심 데이터 백업/복구
- 긴급 공지/긴급 차단

4. 사진/영상 원본 백업은 슈퍼관리자가 하지 않는다.
- 사진/영상 원본 보관 책임은 수영장 운영자에게 있다.
- 슈퍼관리자는 원본 파일이 아니라 핵심 운영 데이터만 백업/복구 대상으로 관리한다.

5. 홈은 메뉴보다 "오늘 처리할 일"이 먼저 보여야 한다.
- 예쁜 아이콘 홈이 아니라 운영 콘솔이어야 한다.

==================================================
[1. 슈퍼관리자 홈 구조]
==================================================

하단탭 없이 아이콘형 + 운영 대시보드형 홈으로 구성

상단 KPI 6칸:
- 전체 운영자 수
- 활성 운영자 수
- 승인 대기 수
- 결제 이슈 수
- 저장공간 95% 초과 수
- 24시간 내 자동삭제 예정 수

KPI 아래에 "오늘 처리할 일" 섹션 추가:
- 승인 대기 운영자
- 결제 실패 운영자
- 저장공간 95% 초과 운영자
- 자동삭제 예정 운영자
- 정책 미확인 운영자
- 고객센터 미처리 문의
- 기능 오류 리포트
- 보안 이벤트 경고
- 외부 서비스 장애 경고

각 항목에서 바로 액션 가능하게 해줘:
- 승인
- 반려
- 상세
- 유예
- 메시지
- 정책 재알림
- 읽기전용 전환
- 기능 OFF
- 공지 발송

그 아래 아이콘 8개:
1. 운영자 관리
2. 구독·결제 관리
3. 저장공간 관리
4. 데이터·킬스위치
5. 정책·컴플라이언스
6. 운영 로그·감사
7. 고객센터
8. 장애·리스크 센터

==================================================
[2. 운영자 관리]
==================================================

운영자 = 수영장 / 1인 코치 / 대관팀 / 프랜차이즈 등
회원을 운영하는 결제 주체

운영자 관리 화면은 카드형이 아니라 리스트형(행형)으로 구성

각 행 표시:
- 상태
- 운영자명
- 운영 유형 (수영장 / 1인 코치 / 대관팀 / 프랜차이즈)
- 활성 회원 수 (정상 + 연기)
- 현재 구독 플랜
- 저장공간 사용률
- 결제 상태
- 마지막 접속일

오른쪽 빠른 액션:
- 상세
- 승인/반려
- 제한
- 구독
- 저장공간
- 더보기

다중 선택 가능
일괄 처리:
- 승인
- 반려
- 제한
- 읽기전용 전환
- 메시지 발송
- 정책 재알림
- 업로드 차단

상단 고정 필터 칩:
- 전체
- 승인 대기
- 결제 실패
- 저장 95% 초과
- 삭제 예정
- 크레딧 보유
- 무료 + 활성 회원 30명 초과
- 이번 주 신규
- 1인 코치
- 프랜차이즈
- 반복 환불
- 업로드 급증
- 정책 미확인

정렬:
- 최신순
- 이름순
- 회원 수
- 저장 사용률
- 결제 위험도
- 최근 활동순

==================================================
[3. 운영자 상세 페이지]
==================================================

운영자 하나를 누르면 한 페이지 안에서 끝내게 해줘.
탭 구성:
- 기본 정보
- 구독/결제
- 저장공간
- 정책/동의
- 로그
- 강제조치

[기본 정보]
- 운영자명
- 운영 유형
- 대표자명
- 연락처
- 가입일
- 마지막 접속일
- 활성 회원 수 (정상 + 연기)
- 퇴원 회원 수
- 현재 상태
- 운영자 코드
- 현재 권한 구조

[구독/결제]
- 현재 플랜
- 추천 플랜
- 다음 결제일
- credit_balance
- 최근 결제 상태
- 결제 실패 횟수
- 분쟁/차지백 여부
- 읽기전용 여부
- 구독 변경 버튼
- 결제 유예 설정 버튼
- 환불 검토 버튼
- read-only 전환 예정 표시

[저장공간]
- 사용량 / 총량
- 사진/영상 메타 비중
- 80% 경고 여부
- 95% 차단 여부
- 최근 7일 업로드 증가량
- 용량 변경
- 파일 삭제 유도
- 킬스위치 바로가기
- 자동삭제 예정 상태

[정책/동의]
- 환불 정책 읽음 여부
- 개인정보 처리방침 읽음 여부
- 이용약관 동의 여부
- 마지막 확인일
- 적용된 정책 버전
- 정책 재알림 발송 버튼

[로그]
- 해당 운영자 관련 이벤트만 필터링
- 제목 / 작업자 / 시간 / 영향도 / 상세
- undefined / Invalid Date 절대 금지

[강제조치]
- 승인
- 반려
- 읽기전용 전환

- 정책 재알림
- 계정 제한
- 고객센터 티켓 생성
- 기능 플래그 예외 적용

==================================================
[4. 구독·결제 관리]
==================================================

구독은 자동 변경하지 않는다.
운영자가 직접 선택하고, 변경은 즉시 적용한다.

상태 칩:
- 전체
- 체험 중
- 구독 중
- 만료됨
- 정지됨
- 해지됨
- 결제 실패
- 크레딧 보유
- 읽기전용
- 일할계산 발생
- 환불 대기
- 분쟁 대응 필요

각 행 표시:
- 운영자명
- 현재 플랜
- 활성 회원 수
- 다음 결제일
- 최근 결제 상태
- credit_balance
- read-only 여부
- 분쟁 여부

빠른 액션:
- 구독 상태 변경
- 결제 확인
- 결제 유예
- 상세 보기
- 환불 검토
- 재청구 링크 발송

정책:
- 변경 즉시 적용
- 업그레이드: 차액 즉시 결제
- 다운그레이드: 차액은 credit_balance 적립
- 결제일 변경하지 않음
- 다음 결제에서 credit 자동 차감
- 다음 결제 전에 해지하면 남은 크레딧 환불 가능

결제 실패 리스크 관리:
- 실패 결제 큐
- 재시도 예정일
- 마지막 실패 사유
- 실패 횟수
- 수동 유예
- read-only 전환 예정일

분쟁/환불 리스크 관리:
- 환불 요청 상태
- 차지백/분쟁 상태
- 증빙 필요 여부
- 반복 환불 운영자 플래그
- 반복 해지/재가입 운영자 플래그

==================================================
[5. 저장공간 관리]
==================================================

용량은 자동 증설하지 않는다.
운영자가 직접 선택한다.

정책:
- 총 용량 = 기본 용량 + 추가 용량
- 80% 이상 경고
- 95% 이상 신규 업로드 차단
- 조회/삭제는 가능
- 다운로드는 구독 상태 정책에 따름

상단 상태 칩:
- 전체
- 95% 초과
- 80% 초과
- 삭제 예정
- 업로드 급증

정책 설정 화면:
- 플랜별 기본 용량
- 추가 용량 단가
- 수정 버튼

운영자 리스트 표시:
- 운영자명
- 사용량 / 총량
- 사용률
- 업로드 차단 여부
- 최근 7일 증가량

빠른 액션:
- 용량 변경
- 파일 삭제
- 상세 보기
- 삭제 유예

리스크 기능:
- 업로드 급증 탐지
- 과도 업로드 운영자 플래그
- 24시간 삭제 예정 경고
- 용량 정책 변경 로그

==================================================
[6. 데이터·킬스위치]
==================================================

삭제 방식:
- 전체 삭제
- 기간별 삭제
- 항목별 삭제
- 운영자별 삭제

기간 옵션:
- 3개월
- 6개월
- 1년
- 직접 입력

항목 옵션:
- 사진 메타데이터
- 영상 메타데이터
- 수업일지 텍스트
- 출결
- 보강
- 메모
- 쪽지

주의:
- 사진/영상 원본 백업 복구 기능은 만들지 말 것
- 원본 파일 보관/삭제 책임은 수영장 운영자 측 기준으로 안내만 제공

킬스위치 실행 전 확인 화면:
- 대상 운영자
- 삭제 방식
- 기간
- 항목
- 예상 삭제량
- 복구 불가 안내
- 확인 체크박스
- 실행자 기록

추가 기능:
- 삭제 유예
- 삭제 예약
- 최근 실행 로그 보기

문서 기능:

- 수정

운영자 설정 화면에서는 읽기 전용 표시

정책 미확인 리스크 관리:
- 환불 정책 미확인 운영자 큐

이 메뉴는 보안/기능 외의 실제 운영 리스크를 한곳에서 보게 해줘.

리스크 종류:
- 결제 리스크
- 저장공간 리스크
- 데이터 삭제 리스크
- 정책 미확인 리스크
- 고객센터 미처리 리스크
- 보안 이벤트 리스크
- 기능 오류 리스크
- 외부 서비스 리스크
- 백업/동기화 리스크
- 악용/남용 리스크

예시 항목:
- 실패 결제 운영자
- 저장공간 95% 초과 운영자
- 24시간 내 자동삭제 예정 운영자
- 정책 미확인 운영자
- 반복 환불 요청 운영자
- 업로드 급증 운영자
- 비정상 로그인 시도
- 백업 실패
- 외부 서비스 오류
- 앱 기능 먹통 리포트
- 날짜/로그 품질 오류

각 항목은 바로 상세/조치 가능해야 함

==================================================
[11. 기능 플래그 관리]
==================================================

슈퍼관리자가 기능 플래그를 직접 켜고 끌 수 있는 화면을 추가해줘.

대상 예시:
- 새 스케줄러 사용
- 새 구독 정책 사용
- 자동 삭제 정책 사용
- 고객센터 기능 사용
- 새 업로드 구조 사용
- 운영자별 특정 기능 사용

구조:
- 전체 플랫폼 ON/OFF
- 운영자별 ON/OFF
- 테스트 운영자만 ON/OFF 가능

변경 시:
- 감사 로그 남기기

==================================================
[12. 읽기전용 제어 구조]
==================================================

읽기전용은 3단계로 제어 가능하게 해줘.

1. 전체 플랫폼 읽기전용
2. 운영자별 읽기전용
3. 기능별 읽기전용

예:
- 전체 업로드 차단
- 특정 운영자만 업로드 차단
- 결제 기능만 읽기전용
- 스케줄 저장만 읽기전용

기록:
- 누가
- 언제
- 왜
전환했는지 로그 남기기

==================================================
[13. 백업/복구 정책]
==================================================

슈퍼관리자 백업 대상:
- 운영자 정보
- 회원 핵심 정보
- 구독 정보
- 크레딧
- 정책 문서
- 출결
- 일지 텍스트
- 로그
- 권한 상태
- 기능 플래그 상태

슈퍼관리자 백업 제외:
- 사진 원본
- 영상 원본

복구 기능:
- 시점 기준 복구
- 운영자 단위 복구
- 전체 복구가 아니라 핵심 데이터 복구 중심
- 킬스위치/자동삭제 전후 스냅샷 확인 구조 열어둘 것

==================================================
[14. 실제 운영 리스크 선제 대응 항목]
==================================================

반드시 포함:
- 실패 결제 재시도/유예
- 정책 미확인 운영자 알림
- 반복 환불/반복 해지 운영자 탐지
- 과도 업로드 탐지
- 읽기전용 자동/수동 전환
- 삭제 예약/유예
- 외부 서비스 상태 표시
- 보안 이벤트 감사 로그
- 품질 오류 리포트 큐
- 고객센터 SLA 큐
- 기능 먹통/터치 불가/팝업 오류 리포트 큐
- 로그 데이터 품질 경고

==================================================
[15. UI 원칙]
==================================================

- 슈퍼관리자는 예쁜 메뉴형이 아니라 운영 콘솔이어야 함
- 한눈에 배열이 보여야 함
- 리스트형/행형 중심
- 각 화면에서 바로 조치 가능해야 함
- 운영자 하나를 여러 메뉴에서 반복 조회하지 않게 운영자 상세 통합 구조를 만든다
- 홈은 메뉴보다 처리 큐가 먼저 보이게
- 지금 코딩할 때부터 아래 문제 안 생기게 해줘:
  - undefined 표시
  - Invalid Date 표시
  - 빈 영역 과다
  - 클릭 수 과다
  - 같은 운영자를 여러 메뉴에서 중복 조회해야 하는 구조

==================================================
[16. 현재 단계]
==================================================

지금은 코딩 단계로 진행한다.
다른 모드와 완전 연동이 일부 부족해도 우선 슈퍼관리자 구조/상태/정책/리스크/로그/고객센터/기능플래그/읽기전용/핵심 백업 구조부터 먼저 구현해줘.

특히 고객센터는 보수적으로 기능을 많이 넣어줘.
지금은 뺄 것보다 넣는 쪽으로 만들고, 실제 써보면서 나중에 덜어낼 예정이다.
