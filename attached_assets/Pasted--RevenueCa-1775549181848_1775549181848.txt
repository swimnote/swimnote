지금부터 구독 기능은 부분 패치 중단.
기존 코드를 조금씩 덧붙여 고치지 말고, 구독 모듈만 거의 새로 만드는 수준으로 재구축하라.

목표:
구독/플랜/수동 부여/RevenueCat 결제/실시간 반영을 하나의 일관된 구조로 다시 만든다.

━━━━━━━━━━━━━━━━━━
1. 재구축 범위
━━━━━━━━━━━━━━━━━━

전체 앱을 갈아엎는 것이 아니다.
구독 모듈만 재구축한다.

대상:
- subscription_plans
- swimming_pools 구독 상태 필드
- pool_subscriptions
- subscriptionResolver
- billing/status
- super subscription PATCH
- RevenueCat sync / webhook
- 앱 구독 표시 화면

━━━━━━━━━━━━━━━━━━
2. 핵심 원칙
━━━━━━━━━━━━━━━━━━

1. subscription_plans = 플랜 원천 테이블
2. swimming_pools = 현재 적용 상태만 저장
3. subscriptionResolver = 유일한 계산기
4. 앱 구독 UI = /billing/status 하나만 사용
5. 구독 변경 경로는 최종적으로 applySubscriptionState(...) 하나만 타게 통일
6. tier는 내부 코드, plan_name은 표시명
7. 화면에서 tier 직접 노출 금지
8. fallback / 하드코딩 / 중복 계산 제거

━━━━━━━━━━━━━━━━━━
3. 새 구조로 다시 만들 것
━━━━━━━━━━━━━━━━━━

A. Plan Catalog
플랜 정의는 subscription_plans만 사용
필수 필드:
- tier
- name
- member_limit
- storage_mb
- display_storage
- video_enabled
- white_label_enabled
- rc_package_id
- entitlement
- price_per_month

B. Pool Subscription State
swimming_pools에는 현재 상태만 저장
필수 필드:
- subscription_tier
- subscription_status
- subscription_source
- base_storage_mb 또는 base_storage_gb
- video_storage_limit_mb
- white_label_enabled
- trial_end_at
- subscription_start_at
- subscription_end_at
- updated_at

C. Resolver
모든 API가 같은 resolver를 사용
반환값:
- tier
- plan_name
- status
- source
- member_limit
- storage_mb
- display_storage
- video_storage_limit_mb
- white_label_enabled

D. Unified Apply Function
아래 함수(이름은 자유)를 새로 만든다:
applySubscriptionState(poolId, tier, source, status)

이 함수가 반드시 수행:
- swimming_pools 상태 저장
- 플랜 기준 파생값 반영
- 로그 기록
- resolver 기준값 재검증

━━━━━━━━━━━━━━━━━━
4. API 구조 단순화
━━━━━━━━━━━━━━━━━━

1. 앱은 구독 상태를 오직 /billing/status 하나만 사용
2. /pools/my, /super/operators/:id는 필요 시 같은 resolver 결과를 쓰되,
   앱 구독 화면은 /billing/status만 본다
3. 앱에서 여러 API 값을 섞어 구독 카드 구성 금지

━━━━━━━━━━━━━━━━━━
5. 변경 경로 통일
━━━━━━━━━━━━━━━━━━

아래 모든 경로가 마지막에 applySubscriptionState(...)를 호출하게 하라.
- 슈퍼관리자 수동 플랜 변경
- RevenueCat purchase 성공
- restore purchases
- renewal
- expired
- canceled
- free/trial 전환

━━━━━━━━━━━━━━━━━━
6. 제거할 것
━━━━━━━━━━━━━━━━━━

- 화면별 TIER_LABEL fallback
- tier 직접 표시
- 중복 plan 계산
- /pools/my와 /billing/status 혼합 사용
- 구독 관련 로컬 임시 state
- legacy 9999 처리 분산 로직
- 화면마다 다른 storage/member_limit 계산

━━━━━━━━━━━━━━━━━━
7. 검증 시나리오
━━━━━━━━━━━━━━━━━━

1. free → pro 수동 변경
2. pro → max 수동 변경
3. max → free 수동 변경
4. 수영장관리자 pro 구매
5. restore purchases
6. manual override 적용
7. manual override 해제 후 RC active 복귀
8. expired/canceled 처리

각 시나리오마다 아래 4개를 제출:
- swimming_pools DB 값
- /billing/status 응답
- /super/operators/:id 응답
- 앱 화면 캡처

━━━━━━━━━━━━━━━━━━
8. 작업 방식
━━━━━━━━━━━━━━━━━━

- 기존 코드에 덧붙이지 말고 새 서비스 파일로 작성
- 기존 API는 내부 구현만 새 서비스로 교체
- 단계적으로 교체하고, 교체 완료 후 레거시 제거
- 중간에 "대충 맞는 것 같다" 식 완료 금지
- 최종적으로 앱에서 free/pro/max가 정확히 표시되고 실시간 반영되는지 증명

이번 작업은 유지보수 가능한 구독 모듈 재구축이다.
부분 패치 중단하고 구조를 새로 세워라.
