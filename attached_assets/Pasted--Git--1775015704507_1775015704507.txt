[스윔노트 프로젝트 — 재해복구 가능 상태로 Git 백업/이관 구조 구축 지시문]

지금부터 이 프로젝트를 “리플릿이 사라져도 다른 개발환경에서 복구 가능한 상태”로 만드는 작업을 진행해줘.
목표는 단순 코드 백업이 아니라, 사고 발생 시 다른 빌더(예: GitHub 기반 + Vercel/Railway/Render/Expo EAS 등)에서도 복구 가능한 수준으로 정리하는 것이다.

반드시 아래 원칙을 지켜서 작업해줘.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. 작업 목표
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

이 프로젝트를 아래 조건을 만족하는 상태로 만들어줘.

- 리플릿 외부 Git 저장소에 전체 소스가 안전하게 백업될 것
- 다른 환경에서 프로젝트를 내려받아 설치/실행/빌드 가능한 구조일 것
- 환경변수, DB, 스토리지, 결제, 앱 빌드 관련 정보가 누락 없이 문서화될 것
- 사고 발생 시 제3자가 봐도 복구 가능한 수준의 인수 문서가 생성될 것
- 기존 코드에 덮어쓰기식으로 위험하게 수정하지 말고, 현재 구조를 먼저 분석한 뒤 안전하게 진행할 것

중요:
이 작업은 “Git 연결”만 하는 게 아니다.
“복구 가능한 운영 프로젝트”로 만드는 것이 목적이다.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
2. 절대 원칙
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

- 기존 기능을 임의로 삭제하거나 리팩토링하지 말 것
- 먼저 현재 프로젝트 구조를 전수 분석한 뒤 작업 계획을 세울 것
- 직접 수정 전, 어떤 파일과 설정을 건드릴지 먼저 보고할 것
- 기존 코드에 바로 덮어쓰기하지 말고, 변경 범위를 최소화할 것
- 민감정보(API Key, Secret, 토큰)는 Git에 절대 올리지 말 것
- .env, 서비스 키, 인증서, 비밀값은 외부 공개 저장소에 노출되지 않게 처리할 것
- 작업 완료 후 반드시 점검 보고서를 작성할 것

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
3. 1차 점검 — 현재 구조 전수 파악
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

먼저 아래 항목을 빠짐없이 조사해서 보고해줘.

A. 프로젝트 기본 구조
- 프론트엔드 사용 기술
- 백엔드 사용 기술
- 모바일 앱 구조(Expo / React Native / 기타)
- 웹/앱/서버가 분리되어 있는지 여부
- package.json / app.json / eas.json / tsconfig / babel 설정 존재 여부
- 빌드 및 실행 명령어 정리

B. 외부 의존 서비스
- DB (예: Supabase)
- 인증(Auth)
- 파일 저장소(Storage / R2 등)
- 푸시 알림
- 결제 연동
- 분석 툴/로그 툴
- 외부 API 연동
- 리플릿 전용 기능 사용 여부

C. 배포 및 운영 구조
- 현재 앱이 어디에 어떻게 배포되는지
- 스토어 제출용 빌드가 어떤 방식으로 만들어지는지
- 도메인/서버/API 주소 사용 여부
- 환경변수 파일 위치 및 사용 방식

D. 위험요소 탐지
- Git ignore 누락 가능성
- 하드코딩된 비밀키 존재 여부
- 리플릿 전용 런타임 의존 코드 존재 여부
- 마이그레이션 파일 누락 여부
- 패키지/버전 불일치 가능성
- 다른 환경에서 실행 시 깨질 수 있는 부분

이 1차 점검 결과를 먼저 정리해서 보여줘.
점검 없이 바로 Git 연결부터 하지 말 것.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
4. Git 백업 구조 구축
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1차 점검이 끝나면 Git 백업 구조를 안전하게 구축해줘.

해야 할 일:

- 현재 프로젝트 전체를 Git 기준으로 정리
- 불필요한 캐시/빌드 산출물/node_modules 제외
- .gitignore를 프로젝트에 맞게 정밀하게 정리
- Git에 올리면 안 되는 민감 파일/민감 경로 차단
- README 및 복구 문서 포함
- 커밋 단위가 이해 가능하도록 구조 정리

구체적으로 확인할 것:

- .gitignore에 반드시 제외되어야 할 항목
  - .env
  - .env.*
  - node_modules
  - dist
  - build
  - .expo
  - .next
  - coverage
  - OS 캐시 파일
  - 로그 파일
  - 임시 빌드 파일
  - 인증서 및 키 파일
- Git에 이미 잘못 올라간 민감정보가 있는지 검사
- 민감정보가 코드에 하드코딩되어 있으면 분리 제안

중요:
Git 초기화나 연결만 하지 말고,
“이 저장소를 다른 환경에서 안전하게 clone 가능한 상태”로 만들어줘.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
5. 환경변수 분리 및 문서화
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

환경변수는 실제 복구에서 가장 중요하니 특히 자세히 처리해줘.

해야 할 일:

- 프로젝트에서 사용 중인 모든 환경변수 목록 추출
- 각 변수의 용도 설명
- 어느 서비스와 연결되는 값인지 설명
- 필수값 / 선택값 구분
- 개발용 / 운영용 구분 가능하면 구분
- 예시용 .env.example 파일 생성
- 실제 비밀값은 절대 Git에 넣지 말고 placeholder만 넣을 것

출력 형식 예시:
- 변수명
- 설명
- 사용 위치(파일/기능)
- 필수 여부
- 예시값 형식

예:
SUPABASE_URL
SUPABASE_ANON_KEY
SERVICE_ROLE_KEY
EXPO_PUBLIC_API_URL
PAYMENT_SECRET_KEY
R2_ACCESS_KEY_ID
R2_SECRET_ACCESS_KEY
등 실제 프로젝트 기준으로 전부 조사

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
6. DB 복구 가능 구조 점검
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Git 백업만으로 끝내지 말고 DB 복구 가능성도 같이 확보해줘.

반드시 확인할 것:

- 현재 DB가 어디에 있는지
- 스키마를 재생성할 수 있는 SQL / migration 파일이 있는지
- 테이블 구조 문서화 가능 여부
- seed 데이터 필요 여부
- 운영 데이터 백업 방식
- 정기 export 가능 여부
- 사고 시 새 환경에 DB를 복원하는 절차

필요 시 아래 결과물을 만들어줘:
- DB 구조 요약 문서
- 마이그레이션 파일 목록
- 누락된 마이그레이션 여부 점검
- 백업/복원 절차 문서

중요:
운영 데이터 자체를 Git에 넣으라는 뜻이 아니다.
“새 환경에서 DB 구조를 다시 세울 수 있는 상태”를 만드는 것이 목표다.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
7. 다른 빌더로 옮길 때 필요한 실행 문서 작성
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

다른 개발자가 이 프로젝트를 받아서 복구할 수 있도록 실행 문서를 만들어줘.

문서에 반드시 포함할 것:

1) 프로젝트 클론 후 설치 방법
- 저장소 clone
- 패키지 설치
- 환경변수 설정
- 실행 명령어

2) 개발 환경 실행 방법
- 로컬 실행
- 모바일 앱 실행
- 웹 실행
- 서버 실행

3) 운영 빌드 방법
- Android 빌드
- iOS 빌드
- Expo/EAS 필요 여부
- 릴리즈 빌드 시 필요한 값

4) 배포 시 필요한 설정
- API 주소
- 도메인
- 인증서
- 외부 서비스 연결
- 스토리지 설정
- 결제 키 설정

5) 장애 복구 순서
- 코드 clone
- env 주입
- DB 연결
- 외부 서비스 키 연결
- 실행 확인
- 빌드 확인
- 배포 확인

이 문서는 “초보 개발자도 따라가면 최소한 복구 가능”한 수준으로 작성해줘.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━
8. 리플릿 의존성 제거 점검
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

이 프로젝트가 리플릿에 얼마나 묶여 있는지 조사해줘.

점검 항목:
- Replit 전용 DB 사용 여부
- Replit Secrets에만 의존하는 구조인지
- Replit 배포 URL 하드코딩 여부
- Replit 런타임에 종속된 스크립트 존재 여부
- 다른 환경에서 실행 시 깨지는 설정 존재 여부

출력 형식:
- 리플릿 의존 요소
- 위험도(상/중/하)
- 대체 방법
- 지금 당장 수정이 필요한지 여부

중요:
불필요한 대공사 하지 말고,
“어디가 리스크인지”를 먼저 정확히 잡아줘.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━

11. 작업 방식
━━━━━━━━━━━━━━━━━━━━━━━━━━━━

중요:
바로 수정부터 하지 말고 아래 순서로 진행해줘.

1) 현재 구조 전수 분석
2) 위험요소 보고
3) Git 백업/복구 전략 제안
4) 승인 없이 위험한 삭제/이동/리팩토링 금지
5) 안전 범위 내 문서화 및 백업 구조 정리
6) 최종 점검 보고서 제출

기존 기능이 살아있는 것이 최우선이다.
“예쁘게 정리”보다 “사고 시 복구 가능”이 목적이다.

지금부터 위 기준에 맞춰 프로젝트를 분석하고,
복구 가능한 Git 백업 구조를 구축하는 작업을 시작해줘.
먼저 1차 점검 결과부터 보고해줘.
