[구독 플랜/슈퍼관리자 상품설정/매출정산 완전 통일 및 실제 연동 강제 수정 지시서]

지금부터 수영장 관리자 구독관리 화면에 최종 반영된 플랜 옵션을 기준으로,
슈퍼관리자 구독 상품 설정 화면 / 슈퍼관리자 매출·정산 관리 / 수영장 관리자 결제 연동을 전부 하나의 기준으로 완전히 통일한다.

이번 작업은 단순 텍스트 수정이 아니다.
“플랜 정의 → 수영장 선택 → 결제 기록 → 슈퍼관리자 정산 반영”까지 전부 실제 데이터로 이어져야 한다.

절대 대충 하면 안 된다.
이 작업은 스윔노트 수익 구조의 핵심이다.

작업 중 반드시 지켜야 할 원칙:
- UI만 맞추고 데이터 안 맞는 상태 금지
- 화면마다 다른 플랜 이름/용량/가격 남아 있으면 실패
- 임시 mock 값으로 보이게만 하고 저장 안 되는 상태 금지
- 슈퍼관리자 / 수영장 관리자 / 정산 화면이 같은 plan_id를 공유해야 함
- 실제 클릭/저장/선택/반영 테스트 없이 완료 보고 금지

2. 표시 단위 강제 통일
==================================================

사용자 화면에는 아래처럼만 보이게 한다.

- 100MB
- 600MB
- 1GB
- 5GB
- 20GB
- 40GB
- 100GB
- 250GB
- 400GB

금지:
- 614MB 같은 자동 변환 숫자 노출 금지
- 20.00GB, 40.00GB, 100.00GB 같은 소수점 표시 금지
- 내부 계산용 숫자를 그대로 UI에 노출 금지

표시 규칙:
- 1024MB 이상이면 GB로 예쁘게 고정 표기
- 600MB는 그대로 600MB
- 100MB는 그대로 100MB
- 사용자에게 보여주는 값은 확정값이어야 한다

3. 슈퍼관리자 구독 상품 설정 화면 전면 수정
==================================================

슈퍼관리자 > 구독 상품 설정 화면에 위 플랜들을 정확히 반영한다.

반드시 수정:
- 과거 상품 카드 제거
- 무료 10명/512MB 같은 값 제거
- 유료100 5000원, 유료300 10000원 같은 과거값 제거
- PRO_500 16000원, PRO_1000 25000원 같은 과거값 제거
- 엔터프라이즈 무료/무제한/0MB 제거

슈퍼관리자 구독 상품 설정 카드에 반드시 표시될 항목:
- 한글 플랜명
- plan_id
- 최대 회원 수
- 기본 용량
- 가격
- 상태(active/inactive)
- 수정 버튼
- 비활성화 버튼

무료 플랜도 최종 기준대로:
- 5명 / 100MB / 0원

중요:
- 슈퍼관리자 화면의 플랜 정의와 수영장 관리자 구독관리 화면의 플랜 정의가 100% 같아야 함
- 카드 개수, 이름, 가격, 용량, 회원 수 모두 동일

4. 수영장 관리자 구독관리 화면 기준 정리
==================================================

현재 수영장 관리자 구독관리 화면에 반영된 플랜을 기준으로,
아래를 추가/수정한다.

반드시 확인:
- 무료 플랜이 보이면 5명 / 100MB / 무료로 나오는지
- 스타터 용량이 614MB가 아니라 600MB로 보이는지
- 어드밴스 표기 통일 여부
- 엔터프라이즈 2000 / 3000이 정확히 보이는지
- 결제 내역 섹션이 실제 데이터와 연동되는지

버튼:
- 업그레이드
- 현재 플랜 표시
- 다운그레이드가 필요하면 다음 결제일부터 적용 안내 구조로만 처리

중요:
- 인앱구독 기준이므로 내부에서 일할 계산/직접 환불 계산하지 말 것
- 상태와 기록만 관리

플랜 정보는 프로젝트 곳곳에 하드코딩하지 말고 단일 원본 소스로 관리한다.

예시:

포함 필드:

모든 화면은 이 공통 원본을 참조하게 한다:
- 슈퍼관리자 상품설정
- 수영장 관리자 구독관리
- 결제 업그레이드 팝업
- 매출정산 분석
- 결제 내역
- 플랜 제한 로직

6. 수영장 관리자 결제 연동
==================================================

수영장 관리자가 플랜을 선택하고 결제하면,
아래 데이터가 실제로 저장/갱신되어야 한다.

subscriptions 테이블 또는 동등 구조에 반드시 저장:
- id
- pool_id
- plan_id
- status (active / trial / failed / pending_downgrade / expired / refunded)
- started_at
- current_period_start
- current_period_end
- next_billing_date
- billing_cycle = monthly
- payment_provider = app_store / play_store / mock_test
- last_payment_amount_krw
- last_payment_at
- intro_discount_applied (true/false)
- created_at
- updated_at

수영장에 반영되어야 하는 값:
- current_plan_id
- current_plan_name
- member_limit
- storage_limit_mb
- billing_status

중요:
- 결제 후 실제 구독 상태가 저장되지 않으면 실패
- 화면만 바뀌고 DB 미반영 금지

==================================================
7. 최초 결제 50% 할인 기록 반영
==================================================

정책 유지:
- 최초 결제 1회만 50% 할인

저장 필드 예시:
- has_used_intro_offer
- intro_discount_applied
- intro_discount_amount_krw

표시:
- 첫 결제 가격
- 이후 정상가

중요:
- 이미 할인 사용한 수영장은 다시 적용되지 않아야 함
- 슈퍼관리자 정산에도 할인 적용된 실결제 금액이 그대로 반영되어야 함

==================================================
8. 슈퍼관리자 매출·정산 관리 연동
==================================================

결제 발생 시 revenue_logs / settlement_logs / payment_logs 등 실제 정산용 로그가 반드시 생성되어야 한다.

필수 저장 필드:
- id
- pool_id
- pool_name
- plan_id
- plan_name_ko
- event_type (new_subscription / renewal / upgrade / downgrade_scheduled / refund / payment_failed / restored)
- gross_amount_krw
- intro_discount_amount_krw
- charged_amount_krw
- refunded_amount_krw
- store_fee_krw
- estimated_net_amount_krw
- payment_provider
- provider_transaction_id
- occurred_at
- created_at

계산 기준:
- gross_amount_krw = 정상가 또는 이벤트상 기준가
- charged_amount_krw = 실제 결제 금액
- store_fee_krw = charged_amount_krw * 스토어 수수료율
- estimated_net_amount_krw = charged_amount_krw - store_fee_krw

스토어 수수료율:
- 현재는 추정치 사용 가능하되 명확히 표기
- iOS / Android 수수료 추정 로직을 분리할 수 있게 구조 열어둘 것

중요:
- 플랜명/금액/결제건수가 비거나 0으로 들어가는 버그 금지
- 매출 화면에 보이는 값이 실제 revenue_logs 합계와 맞아야 함

==================================================
9. 환불 처리 구조 반영
==================================================

정산 화면에 환불 항목 반드시 포함한다.

환불은 내부 임의 계산 금지.
스토어 이벤트 기반으로 상태와 금액만 반영한다.

필수 처리:
- refund event 수신 시 refunded_amount_krw 기록
- payment_status 또는 subscription status 갱신
- 매출·정산 화면의 환불 건수/환불 금액 반영

슈퍼관리자 정산 화면 상단 카드:
- 구독 갱신
- 환불 건수
- 환불 금액

실제 데이터와 연동될 것

==================================================
10. 업그레이드/다운그레이드 기록 구조
==================================================

스토어 정책에 맞게 내부 동작은 상태 반영 중심으로 처리한다.

업그레이드:
- 즉시 적용
- event_type = upgrade
- 새 plan_id 반영
- 결제 로그 생성

다운그레이드:
- 다음 결제일부터 적용 예정
- event_type = downgrade_scheduled
- pending_plan_id 저장
- 현재 플랜 유지

중요:
- 내부에서 일할 계산/즉시 환불 계산 금지
- 상태와 로그만 반영

- 유료100
- 유료300
- 엔터프라이즈(무료/무한)
- 기타 과거 플랜

상품별 매출 분석 카드에는 최종 플랜명만 보여야 한다:
- 스타터
- 베이직
- 스탠다드
- 어드밴스
- 프로
- 맥스
- 엔터프라이즈 2000

각 카드/리스트에 표시:
- 플랜명

- 환불 차감 후 표시 가능하면 같이 표시
- 할인 적용 여부가 필요한 경우 상세 로그에서 확인 가능하게

12. 수수료 항목 정리
==================================================

현재 매출정산 화면에 PG 수수료(PortOne)가 남아 있는데,
앱스토어/플레이스토어 결제를 쓸 것이므로 아래처럼 정리한다.

원칙:
- 스토어 인앱결제 구조에서는 기본 수수료 항목은 App Store / Play 수수료 중심
- PG 수수료는 현재 미사용이면 비활성/숨김 또는 “외부결제 미사용” 상태로 표시
- 향후 웹결제/기업결제가 생길 때만 활성화 가능하게 구조 열어둠

즉시 수정:
- PG 수수료가 계속 실지출처럼 합산되어 보이면 안 됨
- 현재 수익 구조와 안 맞는 항목은 분리하거나 안내문 추가

15. 실제 테스트 데이터 시뮬레이션
==================================================

다음 시뮬레이션을 반드시 직접 수행하고 결과를 보고한다.

시나리오 A
- 무료 → 스타터 업그레이드
확인:
- subscription 생성
- plan_id 반영
- 매출로그 생성
- 정산 화면 반영

시나리오 B
- 스탠다드 가입
확인:
- 9900원 반영
- 스탠다드로 표시
- member_limit=100
- storage_limit=5GB

시나리오 C
- 어드밴스 1건 / 프로 1건 / 맥스 1건 결제 시뮬레이션
확인:
- 각 금액이 슈퍼관리자 상품별 매출 분석에 반영
- 과거 유료100/유료300 이름으로 안 뜨는지

시나리오 D
- 엔터프라이즈 2000, 3000 결제 시뮬레이션
확인:
- 정산 화면에 정확히 반영
- 무료/무제한/0MB 표시 없는지

시나리오 E
- 환불 이벤트 1건 시뮬레이션
확인:
- 환불 건수 +1
- 환불 금액 반영
- 관련 로그 생성

시나리오 F
- 업그레이드 이벤트 시뮬레이션
확인:
- event_type 기록
- 플랜 변경 반영
- 정산 누락 없음
