지금부터 앱 전체에 “권한 부여 + 역할 전환” 구조를 적용하라.

이번 작업의 목표는
수영장 조직도를 아래처럼 단순하고 실사용 가능하게 만드는 것이다.

조직 구조:
- 관리자
- 부관리자
- 선생님

중요:
한 사람이 동시에 여러 역할을 가질 수 있다.
예:
- 관리자 + 선생님
- 부관리자 + 선생님
- 선생님 단독

하지만 계정은 여러 개 만들지 않는다.
반드시 “계정 1개 + 복수 권한 부여 가능” 구조로 만든다.

==================================================
[1. 최종 원칙]

1. 로그인 계정은 1개만 사용
2. 같은 계정에 여러 역할 부여 가능
3. 관리자 승인 단계에서 권한 지정 가능
4. 권한 2개 이상 받은 계정만 더보기 프로필에서 역할 전환 버튼 노출
5. 역할 전환은 새 로그인 없이 같은 계정에서 모드만 전환
6. 기능 연결은 반드시 실제 체크하고 테스트할 것

==================================================
[2. 계정 구조]

모든 사람은 공통 로그인 계정을 가진다.

공통 계정 정보:
- 아이디
- 비밀번호
- 이름
- 연락처
- 소속 수영장
- 상태

역할 정보는 별도로 관리한다.
한 계정에 아래 역할을 복수 부여 가능:
- admin
- sub_admin
- teacher

즉 예시:
- ["teacher"]
- ["sub_admin","teacher"]
- ["admin","teacher"]

중요:
역할마다 별도 계정을 만들지 말 것
반드시 하나의 계정에 역할 배열 또는 복수 역할 구조로 저장할 것

==================================================
[3. 관리자 승인 단계에서 권한 지정]

선생님 가입 승인 화면을 수정하라.

현재 구조가 단순 승인/거절만 있다면
승인 시 권한을 지정할 수 있게 바꾼다.

승인 화면에서 선택 가능한 권한:
- 선생님
- 부관리자
- 관리자

운영 원칙:
- 기본값은 “선생님”만 체크
- 필요 시 부관리자 또는 관리자 권한 추가 가능

예:
- 선생님만 승인
- 부관리자 + 선생님 승인
- 관리자 + 선생님 승인

중요:
“관리자” 또는 “부관리자”만 단독으로 줄 수 있을지 여부는 현재 구조에 맞게 정하되,
최소한 관리자/부관리자 + 선생님 조합은 가능하게 할 것

==================================================
[4. 권한 승인 저장]

관리자가 승인 시 선택한 권한은
해당 계정에 실제 저장되어야 한다.

즉:
- 승인만 보이게 하고 실제 저장 안 되는 가짜 UI 금지
- 승인 후 해당 계정 roles 값이 바뀌어야 함
- 다음 로그인/다음 화면 진입에도 유지되어야 함

==================================================
[5. 더보기 프로필 영역 역할 전환 버튼]

각 모드의 더보기 > 프로필 영역에
“역할 전환” 기능을 추가하라.

단, 아무 계정에나 보이면 안 된다.

노출 조건:
- 해당 계정이 가진 역할이 2개 이상일 때만 노출

예:
1. 선생님만 가진 계정
   → 역할 전환 버튼 숨김

2. 관리자 + 선생님 계정
   → 역할 전환 버튼 노출

3. 부관리자 + 선생님 계정
   → 역할 전환 버튼 노출

==================================================
[6. 역할 전환 방식]

더보기 프로필 영역에는
버튼 여러 개를 늘어놓지 말고,
“역할 전환” 버튼 1개로 통일하라.

버튼을 누르면
현재 계정이 실제로 보유한 역할만 선택 가능한 팝업/선택창이 열리게 한다.

예:
현재 계정 roles = ["admin","teacher"]

현재 관리자 모드에서 역할 전환 버튼 클릭 시
선택 항목:
- 관리자
- 선생님

현재 선생님 모드에서 역할 전환 버튼 클릭 시
선택 항목:
- 선생님
- 관리자

즉:
보유 권한만 보여야 하며,
권한 없는 역할은 절대 보이면 안 된다.

==================================================
[7. 전환 동작]

역할 전환 버튼에서 다른 역할을 선택하면
새 로그인 없이 즉시 해당 모드로 전환하라.

예:
- 관리자 모드 → 역할 전환 → 선생님 선택
  → 선생님 모드로 이동

- 선생님 모드 → 역할 전환 → 관리자 선택
  → 관리자 모드로 이동

중요:
1. 계정은 그대로 유지
2. 로그인 다시 요구 금지
3. 해당 모드의 첫 화면으로 진입
4. 권한 없는 모드 전환 금지

==================================================
[8. 접근 제한]

각 역할은 해당 권한이 있을 때만 접근 가능하게 할 것.

예:
- teacher만 있는 계정은 관리자/부관리자 모드 진입 금지
- admin 권한이 없는 계정은 관리자 전환 불가
- sub_admin 권한이 없는 계정은 부관리자 전환 불가

즉:
전환 버튼도 권한 기반,
실제 진입도 권한 기반이어야 한다.

==================================================
[9. 추천 UI 구성]

더보기 프로필 카드 안에 아래처럼 보이게 한다.

예:
- 현재 역할: 선생님
- 역할 전환 버튼

역할이 2개 이상이면 버튼 보임
역할이 1개면 버튼 숨김

버튼 누르면:
- 관리자
- 부관리자
- 선생님

중 실제 보유 권한만 리스트로 표시

==================================================
[10. 기존 로그인 구조 보호]

중요:
- 기존 로그인 자체를 다시 갈아엎지 말 것
- 지금 있는 공통 로그인 흐름 최대한 유지
- 역할 전환은 로그인 이후 기능으로 붙일 것
- 승인/회원가입 로직을 깨지 말 것
- 기존 기능 화면 연결 깨지지 않게 할 것

==================================================
[11. 연결 체크 필수]

이번 작업은 기능 연결 체크가 매우 중요하다.
아래 항목을 실제로 확인하면서 작업하라.

체크 항목:
1. 선생님 가입 신청 → 관리자 승인 화면 진입 여부
2. 승인 화면에서 권한 선택 UI 보이는지
3. 승인 후 roles 실제 저장되는지
4. 저장된 권한대로 로그인 후 진입 가능한지
5. 권한 2개 이상이면 더보기 프로필에 역할 전환 버튼이 보이는지
6. 권한 1개면 역할 전환 버튼이 숨겨지는지
7. 역할 전환 클릭 시 실제 보유 권한만 보이는지
8. 역할 선택 후 해당 모드 첫 화면으로 제대로 전환되는지
9. 전환 후 기존 계정 로그인 상태 유지되는지
10. 권한 없는 모드 진입이 막히는지

==================================================
[12. 테스트 시나리오]

반드시 아래 시나리오를 실제로 테스트하라.

시나리오 A.
teacher 단독 계정 생성
- 로그인 가능
- 더보기에 역할 전환 버튼 없음 확인

시나리오 B.
teacher 가입 승인 시
sub_admin + teacher 권한 부여
- 로그인 후 더보기에 역할 전환 버튼 보이는지 확인
- 선생님 ↔ 부관리자 전환 되는지 확인

시나리오 C.
teacher 가입 승인 시
admin + teacher 권한 부여
- 로그인 후 더보기에 역할 전환 버튼 보이는지 확인
- 선생님 ↔ 관리자 전환 되는지 확인

시나리오 D.
teacher 계정이 관리자 모드 진입하려 할 때
admin 권한 없으면 막히는지 확인

==================================================
[13. 파일 수정 전 확인]

작업 전에 먼저 아래를 확인하라.

1. 가입 승인 처리 파일
2. 사용자 계정/권한 저장 파일
3. 모드 진입 처리 파일
4. 더보기 프로필 카드 컴포넌트 파일
5. 현재 로그인 이후 모드 분기 처리 파일

이걸 먼저 파악한 뒤 수정 시작

==================================================
[14. 작업 후 보고 형식]

작업 후 아래 순서로 보고하라.

1. 수정 파일 목록
2. 계정 roles 저장 방식
3. 승인 화면 권한 지정 방식
4. 더보기 역할 전환 버튼 표시 조건
5. 역할 전환 동작 방식
6. 접근 제한 방식
7. 테스트 시나리오 A~D 결과
8. 아직 남은 후속 작업이 있으면 명시

==================================================
[15. 절대 금지]

- 계정 여러 개 생성 구조로 변경 금지
- 역할마다 별도 로그인 만들기 금지
- 승인 UI만 만들고 실제 권한 저장 안 하는 것 금지
- 전환 버튼만 만들고 실제 모드 전환 안 되는 것 금지
- 권한 없는 모드 진입 허용 금지
- 기능 연결 확인 없이 끝났다고 보고 금지

이번 작업은
“공통 계정 1개 + 복수 권한 부여 + 승인단계 권한 지정 + 더보기 역할 전환”
구조를 실제로 동작하게 만드는 작업이다.
