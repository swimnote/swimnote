지금부터 앱 전체 화면 통일 작업을 시작한다.
이번 작업은 “기능 추가”가 아니라 “공통 화면 제어 / 팝업 시스템 / 네비게이션 통일” 작업이다.

중요:
- 기존 기능 로직, 데이터 구조, 저장 구조, 연결 구조는 최대한 건드리지 말 것
- 내용/기능/문구를 멋대로 바꾸지 말 것
- 화면 껍데기와 공통 제어만 먼저 통일할 것
- 기존 버튼/기능 연결은 반드시 유지하고, 이동 후에도 전부 작동 확인할 것
- 이번 1차 작업은 탭 재배치보다 우선 “화면 제어 통일 + 팝업 통일”만 먼저 한다
- 탭 배열 재배치는 가능하면 최소한만 하고, 충돌 우려가 있으면 2차로 미룰 것
- 수정 후 실제 테스트를 상세히 보고할 것

==================================================
[1. 이번 1차 작업 목표]

앱 전체(슈퍼관리자 / 관리자 / 선생님 / 학부모)에 아래 공통 규칙을 적용한다.

1) 공통 상단 화면 제어 영역 통일
2) 뒤로가기 / 홈 이동 규칙 통일
3) 탭 내부 이동 규칙 통일
4) 팝업 UI 규칙 통일
5) 키보드와 팝업 간섭 제거
6) safe area / 상단 버튼 위치 통일
7) 기존 기능을 깨뜨리지 않고 화면만 통일

이번 단계에서는
- 내용/기능 수정 금지
- 데이터 로직 수정 금지
- 승인/출결/정산/회원관리 로직 건드리지 말 것
- 화면 레이아웃과 공통 UI만 먼저 정리

==================================================
[2. 절대 원칙]

이번 작업의 절대 원칙은 아래와 같다.

1. 기능은 이동만 한다. 로직 수정 금지.
2. 기존 화면 내용을 최대한 그대로 둔다.
3. 새 공통 레이아웃/팝업 컴포넌트를 만들고 기존 화면을 감싸는 방식으로 적용한다.
4. 기존 버튼 연결을 끊지 않는다.
5. 공통 규칙 적용 후 반드시 모든 연결 테스트를 한다.

즉,
재개발이 아니라 UI/네비게이션 리팩토링이다.

==================================================
[3. 공통 화면 구조 통일]

전 모드 모든 일반 화면에 아래 구조를 적용한다.

--------------------------------------
[3-1. 일반 화면 공통 구조]

상단:
- iOS safe area 확보
- 공통 헤더 영역
- 뒤로가기 버튼 (필요한 화면만)
- 홈 버튼
- 화면 제목 또는 현재 위치 정보

중단:
- 기존 콘텐츠 영역
- 기존 기능/리스트/폼/카드 그대로 유지

하단:
- 기존 하단 탭 유지
- 이번 단계에서는 탭 배열을 멋대로 대규모 변경하지 말 것

--------------------------------------
[3-2. 첫 화면 규칙]

각 탭의 첫 화면에는:
- 뒤로가기 버튼 없음
- 홈 버튼도 없어도 됨 또는 비활성
- 해당 탭의 기준 화면 역할

예:
- 학부모 홈 탭 첫 화면
- 선생님 수업 탭 첫 화면
- 관리자 사람 탭 첫 화면
- 슈퍼관리자 홈 탭 첫 화면

--------------------------------------
[3-3. 상세 화면 규칙]

탭 안에서 하위 화면으로 들어간 경우:
- 뒤로가기 버튼 표시
- 홈 버튼 표시
- 뒤로가기 = 바로 전 단계
- 홈 = 현재 탭의 첫 화면

==================================================
[4. 네비게이션 규칙 통일]

전 모드 동일 규칙 적용

--------------------------------------
[4-1. 뒤로가기]

뒤로가기는 현재 탭 내부의 직전 화면으로 이동한다.
다른 탭으로 넘어가면 안 된다.

우선순위:
1. 팝업 열려 있으면 팝업 먼저 닫기
2. 팝업 없으면 현재 탭 내부 직전 화면
3. 현재 탭 첫 화면이면 뒤로가기 버튼 없음

--------------------------------------
[4-2. 홈 버튼]

홈 버튼은 앱 전체 홈이 아니라
“현재 탭의 첫 화면”으로 이동한다.

예:
- 출결 탭 하위 화면에서 홈 누르면 출결 탭 첫 화면
- 더보기 하위 설정 화면에서 홈 누르면 더보기 첫 화면

--------------------------------------
[4-3. 탭 이동]

다른 탭으로 이동하려면 반드시 하단 탭을 직접 눌러야 한다.
뒤로가기로 다른 탭으로 넘어가면 안 된다.

탭 클릭 시:
- 항상 그 탭의 첫 화면으로 이동
- 이전에 보던 하위 상세 화면 기억하지 않음

--------------------------------------
[4-4. 각 모드 첫 현황 화면]

각 모드의 첫 현황 화면/첫 홈 화면은 그 모드의 기준점이다.
해당 탭을 다시 누르거나 홈 버튼을 누르면 그 탭 내부 하위 경로를 전부 정리하고 첫 화면으로 복귀해야 한다.

==================================================
[5. 팝업 UI 최종 확정안 적용]

앱 전체 모든 팝업에 아래 규칙 통일 적용

--------------------------------------
[5-1. 팝업 기본 구조]

- 하단에서 올라오는 Bottom Sheet
- 높이: 화면의 75% 고정
- 모든 팝업 같은 크기 / 같은 위치
- 작은 내용이어도 크기 줄이지 않음
- 큰 내용은 내부 스크롤 처리

--------------------------------------
[5-2. 팝업 상단]

- 제목 가운데 정렬
- 우측 상단 X 버튼
- 뒤로가기 없음

--------------------------------------
[5-3. 팝업 닫기 방식]

모든 팝업 동일:
- X 버튼
- 바깥 빈 화면 터치
- 아래로 스와이프

--------------------------------------
[5-4. 팝업 내부 내용]

- 내용 많으면 팝업 내부 스크롤
- 드래그로 위아래 이동 가능
- 절대 내용 잘리지 않게
- 오른쪽에 스크롤바 표시

--------------------------------------
[5-5. 팝업 종류 2개]

A. 선택형 팝업
- 목록 선택
- 버튼 선택
- 확인/취소

B. 입력형 팝업
- 텍스트 입력
- 코드 입력
- 메시지 입력
- 검색 입력

두 종류 모두 같은 위치/같은 크기/같은 스타일 적용

==================================================
[6. 키보드 간섭 방지 규칙]

입력형 팝업과 입력 화면에 아래 규칙 적용

1. 키보드가 입력창을 가리면 안 된다
2. 키보드가 올라와도 팝업 상단 위치는 유지
3. 팝업 본문 영역만 줄어들고 내부 스크롤 가능해야 한다
4. 입력창/버튼은 키보드 위에서 보이게 유지
5. 일반 화면 입력에서도 키보드가 하단 버튼/입력칸을 가리지 않게 처리

즉,
키보드 대응은 화면마다 제각각 하지 말고 공통 처리로 통일한다.

==================================================
[7. safe area / 상단 버튼 위치 규칙]

특히 iPhone 기준으로 상단 시스템 영역과 겹치지 않게 한다.

규칙:
- 모든 화면 상단은 safe area 확보
- 뒤로가기/홈/X 버튼이 시스템 멀티태스킹 UI와 겹치지 않게 여백 적용
- 상단 헤더 높이/패딩 전 화면 통일

==================================================
[8. 구현 방식]

이번 작업은 아래 방식으로 구현할 것

1. 공통 ScreenLayout 컴포넌트 생성
   - safe area
   - 상단 헤더
   - 뒤로가기
   - 홈
   - 본문
   - 하단 탭 감싸기

2. 공통 ModalSheet 컴포넌트 생성
   - 75% 높이
   - 제목 가운데 정렬
   - X 버튼
   - 바깥 터치 닫기
   - 아래로 스와이프 닫기
   - 내부 스크롤
   - 스크롤바 표시

3. 기존 화면들은 내용을 바꾸지 말고
   새 공통 레이아웃/모달로 감싸는 방식으로 교체

4. 기존 팝업은 가능하면 공통 ModalSheet로 전환
   단, 기능 깨질 위험이 있으면 단계적으로 적용

==================================================
[9. 이번 단계에서 탭 재배치 원칙]

탭 재배치는 충돌 위험이 있으므로 이번 1차에서는 최소화한다.

원칙:
- 기존 탭 기능은 유지
- 탭 이름/순서 대규모 변경은 가급적 보류
- 단, 화면 제어 통일을 위해 꼭 필요한 최소 수준만 수정 가능
- 구조 재배치 대공사는 2차로 미룰 수 있음

즉,
이번 단계는 “탭 배열 개편”보다
“탭 안 움직임/화면 통일”이 우선이다.

==================================================
[10. 절대 금지]

- 기능 삭제 금지
- 기능 로직 수정 금지
- 기존 저장/승인/출결/정산/메신저/회원관리 로직 건드리지 말 것
- 탭 배열을 멋대로 전부 갈아엎지 말 것
- 화면 내용/문구를 임의 변경하지 말 것
- 기존 버튼 연결 끊기 금지
- 테스트 로그인 삭제 금지
- 기존 모드 진입 삭제 금지

==================================================
[11. 연결 안정성 체크 - 매 화면마다]

반드시 아래를 체크하면서 작업할 것

[공통 체크]
1. 화면 이동 후 기존 버튼이 그대로 작동하는지
2. 뒤로가기/홈 버튼이 현재 탭 안에서만 움직이는지
3. 팝업 열고 닫기 후 상태가 깨지지 않는지
4. 키보드가 입력창/버튼을 가리지 않는지
5. 탭 전환 시 첫 화면으로 이동하는지
6. 기존 테스트 로그인 및 각 모드 진입이 유지되는지
7. 다른 기능과 연결이 끊기지 않았는지

핵심:
화면 통일 작업이지만 모든 연결은 반드시 확인하면서 진행할 것

==================================================
[12. 테스트 지시 - 반드시 실제 수행]

수정 후 아래 테스트를 실제 수행하고 결과를 보고하라.

테스트 1.
학부모 / 선생님 / 관리자 / 슈퍼관리자 각각 진입 가능한지 확인

테스트 2.
각 모드 각 탭 첫 화면에서 뒤로가기 버튼이 없거나 비활성인 상태인지 확인

테스트 3.
각 탭 하위 상세 화면에서 뒤로가기 버튼 누르면 직전 화면으로만 이동하는지 확인

테스트 4.
각 탭 하위 상세 화면에서 홈 버튼 누르면 현재 탭 첫 화면으로 복귀하는지 확인

테스트 5.
다른 탭으로 이동할 때 반드시 탭 첫 화면으로 가는지 확인

테스트 6.
팝업이 열렸을 때
- X로 닫기
- 바깥 터치로 닫기
- 아래로 스와이프로 닫기
전부 되는지 확인

테스트 7.
내용 많은 팝업에서 내부 스크롤/드래그/스크롤바가 정상인지 확인

테스트 8.
입력형 팝업에서 키보드가 입력창과 버튼을 가리지 않는지 확인

테스트 9.
safe area 때문에 상단 버튼이 시스템 UI와 겹치지 않는지 확인

테스트 10.
기존 주요 기능(로그인, 회원가입, 승인, 출결, 일지, 메신저 등)이 이번 작업으로 깨지지 않았는지 확인

==================================================
[13. 보고 형식]

작업 후 아래 형식으로 보고하라.

1. 어떤 공통 레이아웃 컴포넌트를 만들었는지
2. 어떤 공통 팝업 컴포넌트를 만들었는지
3. 뒤로가기 / 홈 / 탭 이동 규칙을 어떻게 적용했는지
4. 키보드 간섭 방지를 어떻게 처리했는지
5. 이번 단계에서 건드리지 않은 기능 영역
6. 테스트 1~10 결과
7. 아직 남은 충돌 위험 또는 2차 작업 필요 항목
8. 다른 기능/화면에 영향 준 부분이 있으면 숨기지 말고 적기

==================================================
[14. 작업 범위 한정]

이번 작업은
“전 화면 공통 제어 UI + 공통 팝업 시스템 + 네비게이션 통일”
까지만 수행한다.

탭 대규모 재배치, 기능 추가, 로직 수정은 하지 말 것.
다른 기능은 그대로 두고 화면만 통일할 것.
