[스윔노트 — 지원센터 공지사항 관리 + 공지 등록 시 학부모 푸시 자동 발송 지시문]

현재 지원센터는 SMS 관련 항목을 제거하고 처리 중심 구조로 정리된 상태다.
이번 작업은 지원센터 안에서 공지사항을 직접 관리할 수 있게 만들고,
공지 등록 시 해당 수영장 학부모에게 자동으로 푸시가 발송되도록 연결하는 작업이다.

━━━━━━━━━━━━━━━━━━
1. 작업 목표
━━━━━━━━━━━━━━━━━━

지원센터에서 아래가 가능해야 한다.

- 공지사항 목록 조회
- 공지사항 등록
- 공지사항 수정
- 공지사항 삭제 또는 숨김 처리
- 공지 등록 시 학부모에게 자동 푸시 발송
- 공지 수정 시 필요하면 푸시 재발송 가능 구조 준비

핵심:
👉 지원센터에서 공지 관리
👉 공지는 저장되는 순간 자동 푸시 발송

━━━━━━━━━━━━━━━━━━
2. 지원센터 내 메뉴 구성
━━━━━━━━━━━━━━━━━━

지원센터에 “공지사항 관리” 메뉴를 추가한다.

위치:
- 지원센터 메인에서 바로 진입 가능
- 고객센터/정책 옆 또는 같은 레벨의 관리 메뉴로 배치
- 너무 깊은 경로 금지

메뉴명:
- 공지사항 관리

설명 문구는 짧게:
- 공지 작성·수정·발송 관리

━━━━━━━━━━━━━━━━━━
3. 공지사항 관리 화면 구조
━━━━━━━━━━━━━━━━━━

화면은 아래 3영역으로 단순하게 구성한다.

1. 상단
- 검색
- 필터
- 등록 버튼

2. 본문 목록
- 공지 제목
- 수영장명
- 작성일
- 상태
- 푸시 발송 여부

3. 액션
- 수정
- 삭제 또는 숨김
- 상세 보기

중요:
- 리스트는 한눈에 읽히게 단순하게
- 긴 설명문 금지
- 카드 또는 목록형 중 더 가벼운 구조 사용

━━━━━━━━━━━━━━━━━━
4. 공지 등록 화면
━━━━━━━━━━━━━━━━━━

등록 항목:

- 대상 수영장 선택
- 제목
- 내용
- 공지 상태
- 푸시 발송 여부 스위치

기본값:
- 푸시 발송 여부 = ON

선택값 예:
- 전체 수영장 공지 또는
- 특정 수영장 공지

현재 단계에서는 우선:
- 특정 수영장 공지 기준으로 구현

중요:
- 등록 버튼 누르면 저장 + 푸시 자동 발송
- 별도 “푸시 보내기” 버튼을 따로 만들지 말 것

━━━━━━━━━━━━━━━━━━
5. notices 데이터 처리
━━━━━━━━━━━━━━━━━━

공지 저장 시 notices 테이블에 저장

필수 필드 예:
- id
- swimming_pool_id
- title
- content
- status
- created_by
- created_at
- updated_at
- push_sent_at
- push_sent_count

status 예:
- published
- hidden
- deleted

원칙:
- hard delete보다 hidden 또는 status 변경 우선
- 운영 이력 추적 가능하게 유지

━━━━━━━━━━━━━━━━━━
6. 푸시 발송 조건
━━━━━━━━━━━━━━━━━━

공지 등록 시 아래 조건으로 자동 푸시 발송:

- 공지 대상 수영장 = swimming_pool_id 기준
- 해당 수영장 소속 학부모만
- 비활성/탈퇴 계정 제외
- 유효한 push token 있는 계정만

즉:
공지 저장
→ 대상 학부모 조회
→ push_tokens 조회
→ 푸시 발송

다른 수영장 학부모에게 절대 발송 금지

━━━━━━━━━━━━━━━━━━
7. 푸시 메시지 형식
━━━━━━━━━━━━━━━━━━

푸시 제목:
- [수영장명] 공지사항

푸시 내용:
- 공지 제목
또는
- 공지 제목 + 짧은 요약

예:
- [토이키즈] 공지사항
- [토이키즈] 4월 휴강 일정 안내

중요:
- 내부 테이블명/영문 상태값 노출 금지
- 학부모가 바로 이해되는 문구 사용

━━━━━━━━━━━━━━━━━━
8. 푸시 발송 서버 처리
━━━━━━━━━━━━━━━━━━

공지 생성 API 내부에서 처리:

1. 공지 저장
2. 대상 학부모 조회
3. push_tokens 조회
4. 푸시 발송 호출
5. 발송 결과 기록

원칙:
- 공지 저장은 우선 성공해야 함
- 일부 푸시 실패가 있어도 공지 저장 자체는 성공 유지
- 실패 토큰은 로그로 남김
- invalid token은 정리 후보로 기록

동기 처리로 전체 저장 지연시키지 말 것
가능하면 비동기 또는 분리 처리

━━━━━━━━━━━━━━━━━━
9. 푸시 클릭 동작
━━━━━━━━━━━━━━━━━━

푸시를 누르면:

- 앱 실행
- 해당 공지 상세 화면으로 이동

필수:
- 공지 id 기반 딥링크 또는 내부 라우팅 연결
- 공지 상세가 없으면 목록 화면이라도 열리게 처리

즉,
푸시가 오고 끝이 아니라
👉 누르면 바로 공지로 들어가야 함

━━━━━━━━━━━━━━━━━━
10. 지원센터 목록 표시 항목
━━━━━━━━━━━━━━━━━━

공지 목록에는 아래만 간결하게 보이게 한다.

- 제목
- 수영장명
- 작성일
- 상태
- 푸시 발송 여부
- 푸시 발송 건수 또는 성공 여부

예:
- 발송완료
- 미발송
- 일부실패

너무 많은 통계 붙이지 말 것

━━━━━━━━━━━━━━━━━━
11. 금지 사항
━━━━━━━━━━━━━━━━━━

- 공지 저장과 푸시 발송을 완전히 분리해서 관리자가 두 번 누르게 만들기
- 모든 수영장 학부모에게 공지 푸시 보내기
- 공지 저장 실패했는데 푸시만 보내기
- push token 없는 계정까지 무조건 발송 시도
- 긴 설명문 버튼/복잡한 단계 추가
- SMS 구조 다시 끼워넣기

━━━━━━━━━━━━━━━━━━
12. 실제 테스트
━━━━━━━━━━━━━━━━━━

반드시 실제로 확인:

1. 지원센터에서 공지사항 관리 진입 가능
2. 특정 수영장 선택 후 공지 등록 가능
3. notices DB 저장 확인
4. 해당 수영장 학부모에게만 푸시 발송 확인
5. 다른 수영장 학부모에게 안 가는지 확인
6. push_sent_at / push_sent_count 기록 확인
7. 푸시 클릭 시 공지 상세 또는 목록으로 이동 확인
8. 공지 수정/숨김 처리 정상 동작 확인

━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━

반드시 포함:

- 수정 파일 경로
- 추가한 지원센터 공지사항 관리 메뉴 위치
- 공지 등록 화면 구성
- notices 저장 결과
- 푸시 발송 대상 조건
- 실제 푸시 발송 결과
- 다른 수영장 미발송 확인 결과
- 푸시 클릭 후 이동 결과
