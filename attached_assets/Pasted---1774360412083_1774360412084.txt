지금부터 스윔노트 앱의 테스트 데이터를 정리하고, 슈퍼관리자-수영장관리자 연동 상태를 전면 점검한 뒤, 회원가입/승인 정책을 아래 기준으로 수정해라.

중요:
- 부분 수정 금지
- mock 데이터로 되는 척 금지
- 화면만 지우고 DB는 남겨두는 식 금지
- 실제 DB/API/화면 연동을 끝까지 확인해라
- 확인 못한 것은 완료라고 쓰지 마라

━━━━━━━━━━━━━━━━━━
1. 이번 작업 목표
━━━━━━━━━━━━━━━━━━

현재 문제:
- 수영장 관리자 회원가입 시 “승인대기”가 뜸
- 그런데 슈퍼관리자 화면에는 승인대기 목록/현황이 안 뜸
- 슈퍼관리자 현황판 전반이 연동이 끊어진 것처럼 보임
- 테스트용 수영장 데이터가 많이 쌓여 있어 검증이 꼬임

이번 작업 목표:
1) 기존 테스트용 수영장/테스트 데이터 정리
2) 슈퍼관리자 현황/승인/목록/API 전수 점검
3) 수영장 관리자 회원가입 정책을 “승인 없이 즉시 가입”으로 변경
4) 앱을 깨끗한 상태로 만든 뒤 새 수영장 가입 기준으로 처음부터 다시 검증
5) 슈퍼관리자 화면과 실제 데이터가 전부 다시 연동되게 수정

━━━━━━━━━━━━━━━━━━
2. 회원가입 정책 변경 (중요)
━━━━━━━━━━━━━━━━━━

정책을 아래처럼 바꿔라.

[변경 전]
수영장 관리자 가입 → 승인대기 → 슈퍼관리자 승인 후 활성

[변경 후]
수영장 관리자 가입 → 승인 없이 즉시 가입 완료 → 즉시 로그인 가능

즉:
- 관리자모드로 회원가입하는 경우 승인 절차 제거
- 승인대기 상태로 보내지 말 것
- 회원가입 완료 즉시 active 상태
- 슈퍼관리자 승인함에서 기다리지 않게 할 것

중요:
- “관리자 가입 문턱을 낮춘다”가 목표다
- 관리자 회원가입에 pending/approval_required 분기가 남아 있으면 제거
- 다만 선생님 등록요청 구조가 따로 있으면 그건 기존 정책 유지 가능
- 이번 변경 대상은 “수영장 관리자 회원가입”이다

━━━━━━━━━━━━━━━━━━
3. 테스트 데이터 정리
━━━━━━━━━━━━━━━━━━

현재 쌓여 있는 테스트용 수영장/테스트 가입 데이터를 정리해라.

정리 대상:
- 테스트용 수영장 계정
- 테스트 관리자 계정
- 테스트용 swimming_pools
- user_pools 연결
- 승인대기 잔여 데이터
- 테스트용 학생/반/출결/일지/공지/쪽지/사진메타/영상메타
- 슈퍼관리자 현황판에 영향을 주는 집계 잔여값

중요:
- 현재 사용자가 앞으로 직접 새로 가입해서 테스트할 새 수영장만 남길 것이므로
- 기존 테스트용 수영장 다수는 정리해라
- 단, 슈퍼관리자 기본 계정은 유지
- 시스템 동작에 필요한 최소 기본 데이터만 유지
- 삭제 전에 어떤 데이터를 지우는지 목록화해서 보고해라

삭제/정리 기준:
- 데모/테스트/seed 용도로 만든 수영장
- 현재 실제 검증 대상으로 쓰지 않을 계정
- 더미 학생/더미 반/더미 일지
- 중복 수영장/중복 계정

중요:
- 화면에서만 안 보이게 하지 말고 DB 기준으로 정리
- 단, 참조 무결성 깨지지 않게 순서 맞춰 정리
- 필요한 경우 soft delete가 아니라 실제 테스트 정리용 purge를 해라
- purge 전/후 row count를 보고해라

━━━━━━━━━━━━━━━━━━
4. 슈퍼관리자 연동 전수점검
━━━━━━━━━━━━━━━━━━

슈퍼관리자 화면에서 아래 항목이 실제 DB와 연동되는지 전부 점검하고, 끊어진 부분은 고쳐라.

점검 대상:
- 수영장 목록
- 관리자 계정 목록
- 가입 현황
- 승인대기 현황
- 전체 수영장 수
- 전체 회원 수
- 구독/결제 현황
- DB 상태
- pool_event_logs / data_change_logs 반영
- 새 가입 수영장 반영
- 새 관리자 계정 반영

확인 방식:
- 각 화면이 어떤 API를 호출하는지 찾기
- 각 API가 어느 DB를 조회하는지 찾기
- 프론트 표시값과 실제 DB row count가 일치하는지 확인
- 끊긴 쿼리/잘못된 import/superAdminDb/poolDb 혼용 오류가 있는지 수정

특히 확인:
- “승인대기 0건”처럼 보이는데 실제 pending 데이터가 있는지
- 또는 반대로 pending은 없어야 하는데 과거 pending 로직이 UI를 아직 타는지
- 슈퍼관리자 현황판 카드 숫자가 실제 row count와 맞는지
- 새로 가입한 수영장이 슈퍼관리자 리스트에 즉시 반영되는지

━━━━━━━━━━━━━━━━━━
5. 승인 구조 정리
━━━━━━━━━━━━━━━━━━

관리자 회원가입 승인 제거에 맞춰 아래를 정리해라.

1. 관리자 회원가입 API
- pending 생성 금지
- active 즉시 생성
- 승인 요청 생성 금지
- 승인 큐 적재 금지

2. 슈퍼관리자 승인 화면
- 관리자 승인대기 섹션이 있다면 제거 또는 숨김 처리
- 관리자 승인대기 카운트가 더 이상 뜨지 않게 정리
- 단, 선생님 등록요청이나 다른 승인 흐름이 있다면 그 대상만 남겨라

3. 관련 로그
- 관리자 회원가입 성공 시 super DB에 정상 가입 로그 남기기
- pending/approved 이벤트 대신 direct_signup 또는 admin_signup_completed 같은 이벤트로 정리 가능하면 정리
- 과거 approval_required 분기 제거

━━━━━━━━━━━━━━━━━━
6. 빈 상태에서 다시 시작할 수 있게 만들기
━━━━━━━━━━━━━━━━━━

앱을 “깨끗한 테스트 시작 상태”로 만들어라.

유지할 것:
- 슈퍼관리자 기본 계정
- 시스템 필수 정책/설정값
- 필수 마스터 데이터

정리할 것:
- 기존 테스트 수영장들
- 기존 테스트 관리자들
- 기존 더미 학생/반/출결/일지
- 더미 사진/영상 메타
- 승인대기 잔여 데이터
- 테스트용 연결 로그 중 사용자 검증을 방해하는 것

정리 후 상태 목표:
- 슈퍼관리자로 로그인 가능
- 수영장 관리자 신규 가입 가능
- 가입 즉시 active
- 즉시 로그인 가능
- 슈퍼관리자 수영장 목록에 새 수영장 반영
- 현황판 숫자 정상 반영

━━━━━━━━━━━━━━━━━━
7. 반드시 확인할 흐름
━━━━━━━━━━━━━━━━━━

아래 흐름을 실제로 처음부터 끝까지 검증해라.

TC1. 슈퍼관리자 로그인
→ 정상 진입

TC2. 앱 초기 상태 확인
→ 기존 테스트 수영장 대량 데이터가 정리되었는지 확인

TC3. 신규 수영장 관리자 회원가입
→ 승인대기 없이 즉시 가입 완료

TC4. 신규 수영장 관리자 로그인
→ 즉시 로그인 성공

TC5. 슈퍼관리자 화면 진입
→ 새 수영장 목록 반영 확인

TC6. 슈퍼관리자 현황판
→ 수영장 수 / 관리자 수 / 기타 핵심 숫자 반영 확인

TC7. 관리자 가입 후 관련 로그
→ super DB에 가입 이벤트/변경 로그 반영 확인

TC8. 기존 승인대기 UI
→ 관리자 가입 건에 대해 더 이상 뜨지 않는지 확인

━━━━━━━━━━━━━━━━━━

- 삭제 대상 관리자 수
- 삭제 대상 학생 수

10. 구현 기준
━━━━━━━━━━━━━━━━━━

- 관리자 회원가입은 active 즉시 생성
- 관리자 회원가입 후 슈퍼관리자 승인 절차 제거
- 슈퍼관리자 화면은 실제 super DB 기반으로 즉시 반영
- pool DB 운영데이터와 super DB 플랫폼데이터가 화면에서 어긋나지 않게 정리
- 과거 pending 로직이 남아 있으면 제거 또는 관리자 가입에는 타지 않게 분기
- 화면 숫자는 실제 API 응답 기준으로 표시
- 하드코딩된 카운트/샘플 수치 제거

━━━━━━━━━━━━━━━━━━
11. 절대 금지
━━━━━━━━━━━━━━━━━━

- UI에서만 숨기고 DB는 남겨두는 식 금지
- 승인대기 문구만 지우고 실제 pending 생성은 남기는 식 금지
- 테스트 데이터 일부만 지워서 다시 꼬이게 만드는 식 금지
- 새 가입한 수영장이 슈퍼관리자에 즉시 안 보이는데 완료라고 하는 것 금지
- mock 현황판 금지
- 확인 안 한 것을 완료라고 적는 것 금지

━━━━━━━━━━━━━━━━━━
12. 보고 형식
━━━━━━━━━━━━━━━━━━

답변은 반드시 아래 형식으로 작성해라.

1. 실제 원인
- 왜 관리자 가입이 pending으로 갔는지
- 왜 슈퍼관리자에 승인대기가 안 보였는지
- 왜 현황판 연동이 끊겨 보였는지

2. 수정한 파일
- 파일 경로
- 함수/컴포넌트명
- 무엇을 어떻게 바꿨는지

3. 삭제/정리한 데이터
- 어떤 수영장/계정/운영데이터를 정리했는지
- purge 전/후 row count

4. 정책 변경 결과
- 관리자 회원가입이 어떻게 바뀌었는지
- pending 제거가 실제 반영됐는지
- 슈퍼관리자 승인 화면은 어떻게 정리됐는지

5. 실제 검증 결과
- TC1 ~ TC8 각각 PASS/FAIL
- 새 수영장 가입 후 슈퍼관리자 반영 여부
- 현황판 숫자 반영 여부

6. 아직 남은 위험
- 완전히 확신 못하는 부분
- 추가 점검 필요한 부분

7. 내가 직접 확인할 위치
- 앱에서 눌러볼 화면
- Supabase에서 확인할 테이블/프로젝트
- 로그에서 확인할 위치

중요:
- “정리 완료” 한 줄 금지
- 반드시 실제 원인 / 실제 수정 / 실제 검증 / 실제 데이터 정리 결과를 적어라

1) 관리자 회원가입 pending 생성 경로 추적
2) 슈퍼관리자 승인대기/현황 API 연결 추적
3) 테스트 데이터 정리 대상 목록 작성
4) cleanup 실행 계획 작성
이 4개부터 진행해라.
