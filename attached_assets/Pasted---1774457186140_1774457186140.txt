지금부터 백업/복구 기능을 실제로 끝까지 검증해라.
단순히 버튼만 보이는지 확인하는 테스트가 아니라,
실제 데이터 생성 → 백업 → 삭제 → 수영장별 복구 → 전체 복구까지 전부 수행해야 한다.

중요:
- mock 데이터 금지
- 실제 DB 데이터 사용
- 실제 버튼 클릭
- 실제 팝업 동작
- 실제 검색
- 실제 시점 선택
- 실제 restore 실행
- 실제 복구 확인

━━━━━━━━━━━━━━━━━━
1. 테스트용 수영장 생성
━━━━━━━━━━━━━━━━━━

임의 수영장 2개 생성:
- 테스트수영장A
- 테스트수영장B

각 수영장에 아래 데이터 생성:
- 학생 2명 이상
- 출결 2건 이상
- 일지 1건 이상

예:
A수영장:
- 김테스트1
- 김테스트2

B수영장:
- 박테스트1
- 박테스트2

━━━━━━━━━━━━━━━━━━
2. 백업 실행
━━━━━━━━━━━━━━━━━━

반드시 아래 2개 실행:
1. 전체 백업 실행
2. pool만 백업 실행

확인:
- backup_logs 기록 생성
- backup_id 생성
- 마지막 성공 시각 갱신
- 카드 상태 정상 표시

━━━━━━━━━━━━━━━━━━
3. 수영장별 복구 테스트
━━━━━━━━━━━━━━━━━━

A수영장 데이터만 일부 삭제 또는 변경

예:
- 학생 1명 삭제
- 출결 1건 삭제
- 일지 1건 삭제

그 다음 슈퍼관리자 화면에서 아래를 실제로 수행:
1. 수영장별 복구 버튼 클릭
2. 팝업 정상 오픈 확인
3. 수영장 검색창에 "테스트수영장A" 검색
4. 검색 결과에서 A수영장 선택
5. 복구 시점 선택
   - 방금 실행한 백업 시점 선택
6. 최종 확인 입력
7. 복구 실행

확인:
- A수영장 삭제 데이터 복구됨
- B수영장 데이터는 변경 없음
- restore_logs 기록 생성
- 복구 전 자동 선백업 실행 여부 확인

━━━━━━━━━━━━━━━━━━
4. 전체 복구 테스트
━━━━━━━━━━━━━━━━━━

이번에는 A/B 둘 다 데이터 일부 삭제 또는 변경

예:
- A수영장 학생 1명 삭제
- B수영장 출결 1건 삭제
- B수영장 일지 1건 삭제

그 다음 슈퍼관리자 화면에서 아래를 실제로 수행:
1. 전체 복구 버튼 클릭
2. 팝업 정상 오픈 확인
3. 전체 백업 목록에서 복구 시점 선택
4. 최종 확인 문구 입력
5. 복구 실행

확인:
- A/B 수영장 데이터 모두 복구됨
- restore_logs 기록 생성
- 복구 전 자동 선백업 실행 여부 확인

━━━━━━━━━━━━━━━━━━
5. 버튼/팝업/UI 검증
━━━━━━━━━━━━━━━━━━

반드시 실제로 눌러서 확인:
1. 전체 백업 실행 버튼
2. pool만 백업 버튼
3. 전체 복구 버튼
4. 수영장별 복구 버튼

팝업 검증:
- 정상 열림
- 닫힘 동작 정상
- 뒤로가기 정상
- 검색창 입력 가능
- 키보드 간섭 없음
- 복구 시점 선택 가능
- 로딩 표시 정상
- 성공/실패 모달 정상

━━━━━━━━━━━━━━━━━━
6. 실패 테스트
━━━━━━━━━━━━━━━━━━

가능하면 일부러 실패 상황도 1회 만들 것

예:
- 잘못된 backup_id
- 잘못된 restore_point
- 없는 수영장명 검색

확인:
- 실패 시 에러 문구 표시
- restore_logs에 failed 기록
- 카드에 실패 상태 반영 여부

━━━━━━━━━━━━━━━━━━
7. 최종 보고 형식
━━━━━━━━━━━━━━━━━━

반드시 아래 형식으로 제출:

1. 생성한 테스트 수영장명
2. 생성한 테스트 데이터 요약
3. 전체 백업 실행 PASS/FAIL
4. pool만 백업 실행 PASS/FAIL
5. 수영장별 복구 PASS/FAIL
6. 전체 복구 PASS/FAIL
7. 수영장 검색 PASS/FAIL
8. 복구 시점 선택 PASS/FAIL
9. 자동 선백업 PASS/FAIL
10. restore_logs 기록 결과
11. 내가 직접 확인할 화면 위치

중요:
- “확인했습니다” 식 보고 금지
- 실제 복구 전/후 데이터 변화까지 적어라
- 버튼 눌림, 팝업 오픈, 검색, 선택, 복구 결과를 각각 적어라
