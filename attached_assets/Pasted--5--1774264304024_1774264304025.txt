[이번 세션 버그 재검증 + 실패 시 즉시 재수정 + 회귀 테스트 지시서]

지금부터 아래 5개 버그에 대해
“이미 수정했다”는 보고를 그대로 믿지 말고,
반드시 실제 재현 테스트 → 수정 확인 → 실패 시 즉시 재수정 → 회귀 테스트까지 수행하라.

중요:
- 이번 작업은 새 기능 추가가 아니다
- 이번 세션에서 수정했다고 한 버그 5개의 “실제 해결 여부 검증”이 목적이다
- 단순 코드 열람으로 끝내지 말고, 실제 요청/응답/렌더링/로그 기준으로 확인하라
- 수정됐다고 해도 다시 같은 조건으로 테스트해서 재발 없는지 확인하라
- 한 파일만 확인하지 말고, 관련 흐름 전체를 회귀 테스트하라
- 테스트 실패 시 즉시 수정하고, 수정 후 같은 테스트를 다시 반복하라
- “문제 없어 보임” 같은 애매한 보고 금지
- 성공/실패를 각각 명확하게 적어라

==================================================
1. 검증 대상 버그 목록
==================================================

버그 1 — today-schedule.ts
문제:
오늘 스케줄 500 오류

기존 원인:
swimming_pools 테이블에 없는 admin_user_id 컬럼을 조회함

보고된 수정:
JWT 토큰의 poolId 필드를 직접 사용하도록 변경

검증 목표:
- 오늘 스케줄 API가 정상 응답하는지
- pool_id 쿼리 없이도 관리자 JWT의 poolId로 정상 조회되는지
- 더 이상 admin_user_id 컬럼 참조가 없는지
- 다른 today-schedule 관련 흐름이 안 깨졌는지

--------------------------------------------------

버그 2 — push-scheduler.ts
문제:
예약 푸시 스케줄러 런타임 오류

기존 원인:
존재하지 않는 class_schedules 테이블 JOIN 사용

보고된 수정:
JOIN 제거, class_groups의 schedule_days LIKE '%요일%' 와 schedule_time 직접 사용

검증 목표:
- 예약 푸시 스케줄러가 실제로 런타임 에러 없이 동작하는지
- 전날 수업 알림 / 당일 수업 알림 계산이 정상인지
- class_schedules 참조가 완전히 제거됐는지
- schedule_days / schedule_time 기반 계산이 실제 데이터에서 동작하는지
- 푸시 관련 다른 기능이 깨지지 않았는지

--------------------------------------------------

버그 3 — super-sync.ts
문제:
SYNC 통계/테넌트/변경분 500 오류

기존 원인:
db.execute() 반환값을 배열처럼 구조분해해서 TypeError 발생

보고된 수정:
(await db.execute(sql`...`)).rows[0] 패턴으로 교체
(rows as any[]) → (rows.rows as any[])

검증 목표:
- SYNC 통계 API 정상 응답
- 테넌트 목록 정상 응답
- 변경분 API 정상 응답
- 더 이상 db.execute 반환값을 배열처럼 다루는 코드가 남아있지 않은지
- super-sync 관련 다른 조회 흐름이 안 깨졌는지

--------------------------------------------------

버그 4 — super.ts
문제:
플랜 목록 500 오류

기존 원인:
subscription_plans 정렬 컬럼명을 monthly_price로 잘못 사용
실제 컬럼명은 price_per_month

보고된 수정:
ORDER BY price_per_month ASC

검증 목표:
- 플랜 목록 API가 500 없이 정상 응답하는지
- 오름차순 정렬이 실제로 price_per_month 기준으로 되는지
- monthly_price 참조가 남아있지 않은지
- 플랜 관련 다른 화면/응답이 안 깨졌는지

--------------------------------------------------

버그 5 — absences.ts / holidays.ts / extra-classes.ts / pricing.ts
문제:
pool_id 없으면 400 오류

기존 원인:
쿼리 파라미터 pool_id만 강제했고, JWT 안의 poolId fallback을 사용하지 않음

보고된 수정:
pool_id 쿼리 파라미터가 없을 때 JWT tokenPoolId 자동 fallback 사용

검증 목표:
- 위 4개 엔드포인트에서 pool_id 없이 요청해도 관리자 JWT 기준으로 정상 처리되는지
- pool_id를 직접 주는 경우도 정상 처리되는지
- JWT에 poolId가 없으면 적절히 실패하는지
- fallback 추가로 인해 다른 권한/테넌트 분리 문제가 생기지 않았는지

2. 검증 방식
==================================================

각 버그에 대해 아래 순서로 수행하라.

1. 기존 오류 상황 재현 시도
- 수정 전과 같은 조건 또는 최대한 유사한 조건으로 요청/동작 재현

2. 현재 코드 확인
- 보고된 수정 내용이 실제 코드에 반영되었는지 확인

3. 실제 실행 테스트
- API 호출
- 화면 진입
- 스케줄러 실행
- 로그 확인
- 에러 여부 확인

4. 결과 판정
- 해결됨 / 부분 해결 / 미해결

5. 미해결 또는 부분 해결이면 즉시 수정

6. 수정 후 같은 테스트 재실행

7. 관련 회귀 테스트
- 같은 파일/흐름 주변 기능이 깨지지 않았는지 확인

==================================================
3. 반드시 수행할 실제 검증 항목
==================================================

--------------------------------------------------
3-1. today-schedule.ts
--------------------------------------------------
- 관리자 JWT로 today schedule 요청
- pool_id 없이 요청
- 응답 200 여부
- 응답 데이터 구조 정상 여부
- DB 쿼리 로그 또는 코드 검색으로 admin_user_id 참조 제거 확인
- 관련 화면 진입 가능 여부 확인

--------------------------------------------------
3-2. push-scheduler.ts
--------------------------------------------------
- 예약 푸시 계산 함수 직접 실행 또는 시뮬레이션
- 전날 알림 대상 계산
- 당일 알림 대상 계산
- 런타임 에러 여부
- class_schedules 문자열 참조 전체 검색
- 실제 스케줄 데이터로 요일/시간 계산 정상 여부
- 전날 20시 기본값, 당일 1시간 전 기본값 계산 깨짐 없는지 확인

--------------------------------------------------
3-3. super-sync.ts
--------------------------------------------------
- sync 통계 엔드포인트 호출
- tenant 목록 호출
- 변경분 호출
- 500 여부 확인
- db.execute 구조분해 실수 패턴 전체 검색
- rows 접근 방식 통일 여부 확인

--------------------------------------------------
3-4. super.ts
--------------------------------------------------
- 플랜 목록 API 호출
- price_per_month 기준 정렬 검증
- monthly_price 문자열 참조 검색
- 관련 화면 렌더 정상 여부 확인

--------------------------------------------------
3-5. absences / holidays / extra-classes / pricing
--------------------------------------------------
각각에 대해
A. pool_id 포함 요청
B. pool_id 없이 JWT poolId만 있는 요청
C. JWT에 poolId 없거나 잘못된 경우 요청

각 경우의 결과를 확인:
- 정상 fallback 되는지
- 잘못된 권한 허용되지 않는지
- 400/401/403/200이 의도대로 나오는지

==================================================
4. 코드 전수 점검도 같이 수행
==================================================

이번 버그 유형상 아래 패턴을 전역 검색해서 추가 위험이 있는지 찾아라.

1. 존재하지 않는 컬럼/테이블 참조
- admin_user_id
- class_schedules
- monthly_price
- 기타 실제 스키마와 다른 이름

2. db.execute 반환값 오용
- const [row] = await db.execute(...)
- 배열 구조분해 패턴
- rows를 배열로 착각하는 패턴

3. pool_id 강제 패턴
- query pool_id만 요구하고 JWT fallback 없는 엔드포인트
- 관리자/운영자 테넌트 분리 로직 누락 여부

중요:
이번 5개 파일만 보지 말고,
같은 실수가 다른 파일에도 있는지 추가로 찾아라.

==================================================
5. 실패 시 수정 원칙
==================================================

검증 중 실패하면 즉시 수정하라.

수정 원칙:
- 최소 수정으로 원인 해결
- 테넌트 분리 깨지지 않게
- JWT fallback 추가 시 권한 체크 유지
- 스키마명/컬럼명은 실제 DB 기준으로 통일
- db.execute 반환형 처리 방식 일관성 유지

수정 후 반드시:
- 원래 실패한 테스트 다시 수행
- 관련 회귀 테스트도 다시 수행

==================================================
6. 회귀 테스트 범위
==================================================

각 버그 수정 확인 후 아래도 같이 확인하라.

- 오늘 스케줄 화면/API가 다른 역할에서도 문제 없는지
- 푸시 스케줄러가 학부모/선생님/관리자 푸시 로직을 안 깨뜨렸는지
- super-sync가 슈퍼관리자 대시보드/동기화 화면과 연동 깨지지 않는지
- 플랜 목록이 매출/정산/구독 화면과 충돌 없는지
- pool_id fallback이 다른 수영장 데이터 노출 사고를 만들지 않는지

==================================================
8. 최종 목적
==================================================

이번 작업의 목적은
“이미 수정했다고 한 버그가 진짜 해결됐는지 검증하고,
재발 위험까지 줄이는 것”이다.

즉,
- 보고를 믿는 것이 아니라
- 실제 재현과 재검증으로 확인하고
- 실패하면 즉시 고치고
- 회귀 테스트까지 끝내는 것

이 기준으로 꼼꼼하게 수행하라.
