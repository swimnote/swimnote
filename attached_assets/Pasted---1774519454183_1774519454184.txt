[스윔노트 — 시간표 카드 색상 적용 + 색상 피커 통합 작업 지시]

이번 작업은 단순 스타일 수정이 아니다.  
“반 색상 선택 → 저장 → 시간표 카드 반영”까지 실제 데이터 흐름이 끊김 없이 동작해야 한다.

임시 색상 하드코딩 금지  
화면만 바꾸고 저장 안 되는 작업 금지  
기존 accent bar 제거 금지

━━━━━━━━━━━━━━━━━━
1. 작업 목표
━━━━━━━━━━━━━━━━━━

반별 색상을 지정할 수 있게 하고,
저장된 색상이 관리자/선생님 시간표 카드에 동일하게 반영되도록 수정한다.

핵심 원칙:
- 반 구분은 항상 왼쪽 accent bar로 유지
- 카드 배경은 선택한 파스텔 색상 반영
- 텍스트는 항상 어두운 색으로 가독성 유지
- 색상이 없거나 흰색이면 기본 흰색 배경 유지
- UI 반영만이 아니라 DB 저장까지 확인해야 함

━━━━━━━━━━━━━━━━━━
2. 화면별 적용 규칙
━━━━━━━━━━━━━━━━━━

[1] WeeklySchedule.tsx
대상:
- 요일별 일간 뷰 슬롯 카드

적용 규칙:
- 슬롯 카드 배경색 = g.color
- 왼쪽 컬러 바 = 기존 classColor(id) 유지
- 텍스트는 배경색과 무관하게 가독성 유지
- 흰색/미설정이면 카드 배경 = #FFFFFF

[2] WeeklyTimetable.tsx
대상:
- teacher 주간 그리드

적용 규칙:
- 카드 배경 = g.color || #F1F5F9 로 하지 말고
  최종 기준은 아래처럼 통일한다

우선순위:
1. g.color 값이 있고 흰색이 아니면 그 색 사용
2. 값이 없으면 #FFFFFF 사용
3. 좌측 3px accent bar = classColor(id) 유지
4. 텍스트 색상 = #111827 고정
5. 파스텔 계열만 들어오도록 picker에서 제한

즉,
“배경색으로 반 구분 + accent bar로 2차 구분” 구조로 간다.

[3] classes.tsx
대상:
- admin 주간 그리드

적용 규칙:
- WeeklyTimetable.tsx와 동일 패턴
- 좌측 accent bar 유지
- 카드 배경 = 저장된 반 색상
- 미설정/흰색 = #FFFFFF
- 텍스트 = #111827

━━━━━━━━━━━━━━━━━━

━━━━━━━━━━━━━━━━━━
4. 색상 정책
━━━━━━━━━━━━━━━━━━

허용 색상:
- 파스텔 계열만 허용
- 검정/진한 원색/형광색 금지
- 검정 글씨(#111827) 기준으로 읽기 쉬운 색만 사용

기본값:
- #FFFFFF

중요:
- 흰색도 정상 선택값으로 허용 가능
- 단, 흰색이어도 accent bar는 유지해야 함
- 결국 반 구분은 “accent bar + 텍스트 정보”로 항상 가능해야 함

권장 색상 예시:
- #FFFFFF
- #FCE7F3
- #FDE68A
- #FEF3C7
- #DBEAFE
- #E0E7FF
- #DCFCE7
- #CCFBF1
- #F3E8FF
- #F1F5F9

색상 세트는 관리자/선생님/생성 플로우 전부 동일해야 함

━━━━━━━━━━━━━━━━━━
5. 서버/데이터 기준
━━━━━━━━━━━━━━━━━━

- 반 color 값은 실제 반 데이터에 저장
- PATCH / POST 모두 superAdminDb 기준으로 저장
- pool DB 접근 금지
- 기존 반 조회 API에서 color 필드 내려오는지 확인
- 카드 렌더링은 API 응답값 기준으로 동작해야 함
- 클라이언트 임시 상태로만 색상 반영 금지

반드시 확인:
- 생성 시 color 저장되는지
- 수정 시 color 변경되는지
- 재접속 후에도 유지되는지
- 다른 역할 화면에서도 동일하게 보이는지

━━━━━━━━━━━━━━━━━━
6. UI 상세 규칙
━━━━━━━━━━━━━━━━━━

- 카드 텍스트는 #111827
- 서브 텍스트 필요 시 #374151 또는 #4B5563
- accent bar 두께:
  - teacher 주간 그리드: 3px
  - 나머지는 기존 UI 균형 맞춰 유지
- 카드 전체 라운드/패딩/그림자 기존 시스템 해치지 말 것
- 색상 피커는 너무 커지지 않게
- 선택된 색은 체크 또는 테두리로 명확히 표시

━━━━━━━━━━━━━━━━━━
7. 반드시 테스트할 항목
━━━━━━━━━━━━━━━━━━

[생성 테스트]
1. 새 반 생성
2. Step 4에서 색상 선택
3. 생성 완료
4. DB에 color 저장 확인
5. 관리자 시간표 반영 확인
6. 선생님 시간표 반영 확인

[수정 테스트]
1. 기존 반 상세 진입
2. 색상 변경
3. PATCH 요청 성공 확인
4. DB 값 변경 확인
5. 현재 화면 즉시 반영 확인
6. 화면 재진입 후 유지 확인

[예외 테스트]
1. 색상 없는 기존 반 표시 확인
2. 흰색 반 표시 확인
3. null / undefined / 빈문자열 대응 확인
4. 저장 실패 시 롤백 또는 에러 표시 확인

━━━━━━━━━━━━━━━━━━
8. 제출 보고서 형식
━━━━━━━━━━━━━━━━━━

반드시 아래 항목 포함

- 수정 파일 경로
- 생성 시 color 전달 request payload
- PATCH 시 color 전달 request payload
- DB 저장 결과
- 관리자 화면 반영 결과
- 선생님 화면 반영 결과
- 색상 미설정 반 표시 결과
- 흰색 반 표시 결과
- 실패/예외 테스트 결과

절대 금지:
- “적용 완료”
- “정상 작동”
- “수정했습니다”

증거 없는 종료 금지

━━━━━━━━━━━━━━━━━━
