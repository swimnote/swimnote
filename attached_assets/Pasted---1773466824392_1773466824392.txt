이번 작업은 비용 절감을 위해 한 번에 정리해서 진행해줘.
중복 수정 없이 전체 구조를 먼저 분석한 뒤 필요한 변경만 한 번에 반영해줘.

이번 작업 범위:
"학부모/선생님 관리 구조를 목록형에서 승인형 구조로 변경 + 초대링크/승인/거절 흐름 추가"까지만 진행해줘.
그 외 UI 전체 리디자인, 불필요한 리팩토링, 성능 최적화, 다른 메뉴 수정은 하지 마.

목표:
현재 학부모/선생님 관리가 단순 목록 구조인데,
이를 실제 운영에 맞게 "승인형 계정 관리 구조"로 바꿔줘.

핵심 방향:
1. 학부모는 앱 다운로드 후 가입하고 수영장 이름을 검색해서 승인 요청 가능
2. 수영장 관리자는 학부모 승인/거절 가능
3. 선생님은 수영장 관리자가 기본정보를 먼저 입력하고 초대링크를 발송
4. 선생님이 초대링크로 접속 후 가입/연결 요청하면 수영장 관리자가 승인/거절 가능
5. 기존 단순 목록 구조가 아니라 승인 상태 기반 구조로 변경

필수 기능:

1. 학부모 승인 구조로 변경
- 현재 학부모 관리 메뉴를 "학부모 승인/관리" 구조로 바꿔줘
- 학부모는 앱을 설치하고 가입 후 수영장 이름을 검색할 수 있게 해줘
- 학부모가 특정 수영장을 선택하고 승인 요청 버튼을 누를 수 있게 해줘
- 요청 시 상태값 예시:
  - pending
  - approved
  - rejected
- 수영장 관리자는 학부모 요청 목록을 보고 승인/거절 가능하게 해줘

2. 학부모 요청 데이터
학부모 요청 데이터에는 최소 아래 정보가 들어가게 해줘:
- parentUserId
- parentName
- phoneNumber 또는 가입 계정 정보
- poolId
- requestStatus
- requestedAt
- processedAt
- processedBy

3. 학부모 승인 후 권한
- 승인된 학부모만 해당 수영장 자녀 정보, 사진첩, 수영일지, 공지, 알림에 접근 가능
- 승인 전에는 제한된 상태로 두고, 승인 대기 화면 또는 안내 문구를 보여줘
- rejected 상태면 접근 불가 및 적절한 메시지 표시

4. 선생님 승인 구조로 변경
- 수영장 관리자가 선생님 기본정보를 먼저 입력하고 초대할 수 있게 해줘
- 선생님 기본정보 입력 항목:
  - 이름
  - 연락처(전화번호)
  - 직급
- 관리자가 저장 후 초대링크를 생성할 수 있게 해줘
- 전화번호를 기준으로 초대링크를 보낼 수 있는 구조로 만들어줘
- 실제 문자 발송 API 연동은 지금 안 해도 되고,
  우선은 링크 생성/복사 가능한 구조면 충분
- 이후 문자 발송 연동 가능하게 확장 가능한 구조로 설계해줘

5. 선생님 초대/승인 상태
선생님 상태값 예시:
- invited
- joinedPendingApproval
- approved
- rejected
- inactive

동작 방식:
- 관리자가 선생님 기본정보 입력
- 초대링크 생성
- 선생님이 링크로 접속 후 가입 또는 계정 연결
- 상태를 joinedPendingApproval 로 변경
- 수영장 관리자가 승인/거절
- approved 상태일 때만 선생님 권한 활성화

6. 선생님 데이터 구조
선생님 데이터에는 최소 아래 정보가 들어가게 해줘:
- teacherId
- name
- phoneNumber
- position
- poolId
- inviteToken 또는 inviteLink
- inviteStatus
- requestedAt
- approvedAt
- approvedBy

7. 수영장 관리자 화면 변경
학부모/선생님 관리 화면을 아래 구조로 정리해줘:

학부모 관리:
- 승인 대기
- 승인 완료
- 거절됨

선생님 관리:
- 초대 전/기본정보 등록
- 초대 보냄
- 승인 대기
- 승인 완료
- 거절됨

각 항목에서 상태별로 필터 가능하게 해줘

8. 초대링크 구조
- 선생님용 초대링크는 poolId와 초대 토큰을 포함해서 생성
- 링크를 통해 접속한 사람만 선생님 승인 절차를 진행할 수 있게 해줘
- 임의 가입으로 선생님이 되는 구조는 막아줘

9. 학부모 승인 요청 구조
- 학부모는 초대링크 없이도 앱에서 수영장 검색 후 승인 요청 가능
- 단, 승인 전까지는 제한 상태
- 승인 후에만 해당 수영장 소속 학부모로 연결되게 해줘

10. 멀티테넌트 유지
- 학부모/선생님 승인 요청 및 승인 데이터는 모두 poolId 기준으로 저장
- 다른 수영장 관리자에게는 절대 보이지 않게 해줘
- 현재 멀티테넌트 구조와 충돌 없이 유지해줘

11. 권한 구조 유지
- 읽기 전용 플랫폼 관리자나 일반 플랫폼 관리자가 필요한 경우 해당 수영장 상태를 볼 수는 있어도
  실제 학부모/선생님 승인/거절은 권한 있는 수영장 관리자 또는 상위 권한 관리자만 가능하게 해줘
- 수영장 관리자 권한 범위 안에서만 승인 처리되게 해줘

12. JSON 응답 유지
- 모든 관련 API 응답은 JSON 형식 유지
- 에러 발생 시에도 JSON 반환
- 예:
  { "success": true, "data": ... }
  { "success": false, "message": "error" }

13. UI 방향
- 현재 관리 화면 스타일을 크게 해치지 않는 선에서 확장
- 복잡한 디자인보다 상태 중심 구조로 정리
- 목록만 바꾸는 수준이 아니라 실제 승인 플로우가 동작하게 해줘

14. 테스트 조건
아래 테스트가 가능하게 해줘:
- 학부모가 앱 가입 후 수영장 검색 → 승인 요청
- 수영장 관리자가 학부모 승인/거절
- 승인된 학부모만 해당 수영장 데이터 접근 가능
- 관리자가 선생님 이름/연락처/직급 입력 → 초대링크 생성
- 선생님이 링크 접속 후 가입/연결 요청
- 관리자가 선생님 승인/거절
- 승인된 선생님만 선생님 메뉴 권한 활성화

작업 방식:
1. 먼저 현재 학부모 관리, 선생님 관리, auth, 승인 상태, poolId 구조를 분석
2. 같은 파일은 가능하면 한 번만 수정
3. UI 전체 변경, 스타일 정리, 성능 최적화는 하지 마
4. 이번 턴에서는 오직 학부모/선생님 승인형 구조 전환만 진행

마지막 결과 보고 형식:
1. 수정한 파일 목록
2. 추가한 상태값 목록
3. 학부모 승인 플로우 설명
4. 선생님 초대/승인 플로우 설명
5. 추가된 데이터 필드 목록
6. 테스트 방법
7. 남은 후속 작업 목록
