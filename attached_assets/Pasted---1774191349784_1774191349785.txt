프로젝트: 스윔노트 토이키즈 관리자 모드 전면 수정
최우선 목표: 안정성, 데이터 보호, 버그 제거, 경로 단일화

절대 원칙:
- 기능 추가보다 안정성 우선
- 기존에 보이는 버그(Invalid Date, undefined, 버튼 불응, 경로 꼬임) 먼저 수정
- 학부모/선생님 초대는 플랫폼 SMS 발송이 아니라 관리자 휴대폰 기본 문자 앱 사용
- 앱 링크는 수영장별 분기 없이 공용 링크 사용
- 학부모 승인 자동화는 전화번호 + 생년월일 기준
- 백업/복구는 MVP 핵심 기능으로 강화
- 스냅샷 삭제 기능도 넣되 강한 경고문 필수

[1. 초대 방식 전면 변경]
==================================================

기존의 SMS 크레딧 / SMS 과금 / SMS 발송 구조는 관리자 모드에서 제거한다.
운영 관리 안의 "SMS 크레딧" 메뉴 삭제.
플랫폼이 문자를 직접 보내는 구조 사용 금지.

새 방식:
- 관리자 또는 선생님이 학생 정보 등록 후
- "학부모 초대" 또는 "선생님 초대" 버튼 클릭 시
- 휴대폰 기본 문자 앱을 연다
- 수신자 번호는 등록된 번호 자동 입력
- 본문은 자동 생성
- 관리자가 확인 후 직접 발송

공용 링크:
- iPhone용 앱스토어 공용 URL
- Android용 플레이스토어 공용 URL
- 수영장별 코드 없음
- 가입 후 수영장 검색 방식 사용

문자 본문 기본 템플릿:
안녕하세요. [수영장명]입니다.
스윔노트 앱을 설치한 뒤 회원가입 후 [학부모로 이용하기] 또는 [선생님으로 참여하기]를 선택해주세요.
가입 후 수영장을 검색해 [수영장명]을 선택하고,
등록된 회원 정보와 동일하게 자녀 이름/생년월일/보호자 휴대폰 번호를 입력해야 승인됩니다.
iPhone 설치: [앱스토어 공용 URL]
Android 설치: [플레이스토어 공용 URL]

선생님 초대 기본 템플릿:
안녕하세요. [수영장명]입니다.
스윔노트 앱을 설치한 뒤 회원가입 후 [선생님으로 참여하기]를 선택해주세요.
가입 후 수영장을 검색해 [수영장명]을 선택하고, 등록된 휴대폰 번호로 가입해주세요.
iPhone 설치: [앱스토어 공용 URL]
Android 설치: [플레이스토어 공용 URL]

필수 기능:
- 문자 앱 열기 버튼
- 초대 문구 미리보기
- 문구 복사 버튼
- 발송 시도 기록 저장
- 발송 완료 여부는 시스템이 추적하지 않고 "문자 앱 호출 기록"만 남김

초대 내역 메뉴 이름 수정:
- "SMS 발송" 느낌 제거
- "초대 안내 기록" 또는 "문자 앱 호출 기록" 중심으로 변경

[2. 학부모 승인 자동화 수정]
==================================================

자동 승인 기준:
- 보호자 휴대폰 번호 일치
- 자녀 생년월일 일치

자녀 이름:
- 자동 승인 보조 검증용으로 저장
- 관리자 화면에서 비교 표시
- 오타 가능성을 고려해 이름 불일치여도 번호+생년월일 일치 시 자동 승인 가능

학부모 가입 흐름:
1. 공통 회원가입 + 휴대폰 인증
2. 역할 선택에서 학부모 선택
3. 수영장 검색/선택
4. 자녀 정보 입력
   - 자녀 이름
   - 생년월일
   - 관계
   - 보호자 휴대폰 번호
   - 최대 자녀 3명 등록 가능
5. 자동 승인 조건 검사
6. 일치하면 자동 승인
7. 불일치하면 승인 대기 상태

자동 승인 실패 시:
- 승인 대기 목록으로 이동
- 실패 이유 로그 저장

실패 로그 필드:
- 입력 자녀 이름
- 입력 생년월일
- 입력 보호자 번호
- 선택 수영장
- 매칭 실패 사유
  - 번호 불일치
  - 생년월일 불일치
  - 학생 미등록
  - 다중 후보 충돌

추가 메뉴:
운영 관리 > 가입 검증 로그

[5. 이벤트 기록 버그 수정 - 최우선]
==================================================

현재 보이는 오류:
- Invalid Date
- NaN:NaN
- undefined
- "선생님 승인 해제 — undefined"

이건 UI 문제가 아니라 데이터 생성 로직 문제다.
이벤트 로그 생성 시 필수값 보정 로직 추가.

모든 이벤트 필수 필드:
- id
- eventType
- actorRole
- actorName
- targetType
- targetName
- summary
- createdAt
- meta

필수 기본값 처리:
- createdAt 없으면 new Date().toISOString()
- summary 없으면 eventType 기반 기본 문장 생성
- targetName 없으면 "대상 없음"
- actorName 없으면 "시스템"

렌더링 시 안전 처리:
- 날짜 파싱 실패 시 "시간 정보 없음"
- summary 없으면 "기록 정보 없음"
- undefined / NaN 그대로 노출 금지

예시:
"선생님 승인 해제 — 대상 없음"
"시간 정보 없음"

절대 금지:
- Invalid Date 출력
- NaN:NaN 출력
- undefined 문자열 출력

[6. 백업·복구 구조 강화]
==================================================

현재 백업 없음 상태는 위험하므로 보강 필요.

백업 구조:
1. 이벤트 로그는 저장 시마다 즉시 기록
2. 스냅샷은 1시간마다 자동 생성
3. 위험 작업 전 즉시 스냅샷 생성
   - 복구 실행 전
   - 대량 삭제 전
   - 정책 변경 전
   - 권한 대량 변경 전
4. 수동 백업 버튼 유지

스냅샷 유형:
- auto
- manual
- before_restore
- before_delete
- before_policy_change
- before_bulk_permission_change

백업·복구 화면 필수:
- 현재 상태 카드
- 자동 백업 주기 안내
- 스냅샷 목록
- 스냅샷 생성 버튼
- 복구 버튼
- 복구 이력
- 스냅샷 삭제 버튼

==================================================
[7. 스냅샷 보관 정책]
==================================================

보관 기간은 데이터 부담을 고려해 아래처럼 설정한다.

추천 기본 정책:
- 최근 7일: 시간 단위 스냅샷 보관
- 최근 30일: 일 단위 대표 스냅샷 보관
- 최근 90일: 주 단위 대표 스냅샷 보관

즉:
- 0~7일: 1시간 간격
- 8~30일: 하루 1개
- 31~90일: 주 1개

이 정책으로 가면:
- 최근 복구는 정밀하게 가능
- 오래된 백업은 용량 부담 줄임
- MVP 단계에서 현실적

추가:
- 수동 스냅샷은 90일 우선 보관
- before_restore / before_delete는 90일 우선 보관

[8. 스냅샷 삭제 기능 추가]
==================================================

관리자가 백업 데이터 자체를 삭제할 수 있게 한다.
단, 매우 강한 경고문과 2단계 확인 필요.

추가 위치:
데이터 관리 > 백업·복구 > 스냅샷 상세 > 삭제

삭제 전 경고문 정확히 표시:
"데이터 유지를 위하여 백업 데이터를 삭제할 경우 이전 데이터는 영원히 복구할 수 없습니다."

추가 경고문:
- 삭제된 스냅샷은 되돌릴 수 없습니다
- 스냅샷 삭제 후 해당 시점으로는 복구할 수 없습니다
- 현재 운영 데이터에는 즉시 영향이 없지만 과거 복구 가능 범위가 줄어듭니다

삭제 확인 절차:
1. 경고 팝업
2. "삭제 내용을 이해했습니다" 체크
3. 삭제 버튼 활성화

로그 기록:
- snapshot_delete 이벤트 저장
- 삭제자, 삭제 시각, 스냅샷 id, 생성일 기록
