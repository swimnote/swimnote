이번 작업은 비용 절감을 위해 한 번에 정리해서 진행해줘.
중복 수정 없이 전체 구조를 먼저 분석한 뒤 필요한 변경만 한 번에 반영해줘.

이번 작업 범위:
"수영장 가입 신청 + 플랫폼 승인 + 수영장 관리자 계정 생성 + 상태 관리"까지만 진행해줘.
그 외 UI 리디자인, 성능 최적화, 불필요한 리팩토링은 하지 마.

목표:
전국 수영장 플랫폼 SwimNote에서
새 수영장이 가입 신청을 하면 플랫폼 운영자가 승인하고,
승인 후 해당 수영장 관리자 계정을 생성해서 로그인 가능한 구조를 만들어줘.

필수 기능:

1. 수영장 가입 신청 기능
- 새 수영장이 가입 신청할 수 있는 화면 또는 폼을 만들어줘
- 신청 항목:
  - 수영장 이름
  - 대표자명
  - 관리자 이름
  - 관리자 연락처
  - 관리자 이메일 또는 로그인 아이디
  - 비밀번호
  - 주소
  - 프랜차이즈 여부
  - 프랜차이즈명(선택)
  - 예상 회원 수
- 신청 시 기본 상태는 "approvalPending"으로 저장해줘

2. 수영장 데이터 생성 구조
- 가입 신청이 들어오면 새로운 pool 데이터가 생성되거나,
  승인 전 임시 신청 데이터로 저장되게 해줘
- 승인 후 실제 활성 poolId가 생성되거나 연결되게 해줘
- 모든 수영장은 고유한 poolId를 가져야 해

3. 플랫폼 운영자 승인 기능
- 플랫폼 운영자만 수영장 신청 목록을 볼 수 있게 해줘
- 신청 상태 구분:
  - approvalPending
  - approvedFree
  - approvedPaid
  - rejected
  - suspended
- 플랫폼 운영자는 승인 / 반려 / 정지 처리가 가능하게 해줘

4. 승인 시 관리자 계정 생성
- 승인되면 해당 수영장 관리자 계정이 활성화되게 해줘
- 승인 전에는 로그인 불가
- 승인 후에는 해당 관리자 계정이 자신의 poolId로 로그인 가능해야 해
- 관리자 계정 role은 "poolAdmin"으로 저장해줘

5. 반려 / 정지 처리
- rejected 상태면 로그인 불가
- suspended 상태면 로그인 가능 여부를 선택적으로 제한하거나,
  최소한 주요 기능 사용을 제한할 수 있게 구조를 잡아줘
- 상태값은 이후 구독 기능과 연결 가능하게 설계해줘

6. 플랫폼 운영자 관리 화면
- 플랫폼 운영자 화면에서 아래 정보가 보이게 해줘
  - 수영장명
  - 대표자명
  - 관리자명
  - 연락처
  - 프랜차이즈 여부
  - 예상 회원 수
  - 현재 상태
  - 가입 신청일
- 승인 / 반려 / 정지 버튼이 보이게 해줘

7. 로그인 제한
- 승인되지 않은 수영장 관리자 계정은 로그인 불가 처리
- 로그인 시 상태를 확인해서
  approvalPending / rejected / suspended 상태는 적절한 메시지를 보여줘
- approvedFree / approvedPaid 상태만 정상 로그인 가능하게 해줘

8. 데이터 연결
- 승인된 수영장은 이후 생성되는 회원, 반, 사진, 출결, 수영일지, 공지 데이터가
  해당 poolId에 자동 연결되도록 유지해줘
- 현재 멀티테넌트 구조와 충돌 없이 연결해줘

9. JSON 응답 유지
- 모든 API 응답은 JSON 형식 유지
- 에러 발생 시에도 JSON 반환
- 예:
  { "success": true, "data": ... }
  { "success": false, "message": "error" }

10. 브랜드명 유지
- 브랜드 표기는 계속 SwimNote / 스윔노트로 유지
- 기존 브랜드명 변경 작업과 충돌 없게 해줘

작업 방식:
1. 먼저 관련 파일과 기존 인증 구조를 분석
2. 같은 파일은 가능하면 한 번만 수정
3. 불필요한 코드 스타일 변경, UI 전체 수정, 테스트 자동화, 최적화 작업은 하지 마
4. 이번 턴에서는 수영장 가입 신청, 승인, 상태 관리, 관리자 계정 활성화까지만 진행

마지막 결과 보고 형식:
1. 수정한 파일 목록
2. 추가된 데이터 모델 또는 상태값 목록
3. 추가된 화면 목록
4. 승인 플로우 설명
5. 로그인 제한 로직 설명
6. 테스트 방법
7. 남은 후속 작업 목록
