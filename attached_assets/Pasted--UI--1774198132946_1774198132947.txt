[작업 지시: 스윔노트 슈퍼관리자 콘솔 구현]

중요: 이번 작업은 UI만 만드는 작업이 아니다.
반드시 실제 데이터 연결, 상태 변화, 저장, 조회, 필터, 상세 이동, 로그 기록까지 전부 작동해야 한다.
버튼만 보이게 만들고 미작동 상태로 두는 것을 금지한다.
기능 구현 후에는 각 기능이 실제로 작동하는지 직접 점검하고, 점검 결과를 작업 완료 보고에 항목별로 적어라.
거짓 완료 보고 금지.
작동 확인 없이 “완료”라고 하지 마라.

[개발 원칙]
- Replit 환경에서 실제 작동하는 구조로 구현
- UI 목업 금지
- 모든 버튼/탭/팝업/상세페이지/필터 실제 동작 필수
- 이벤트 로그 기록 필수
- 상태값은 문자열 남발하지 말고 enum 또는 상수 체계로 관리
- 날짜/시간 값에서 Invalid Date, NaN, undefined 저장 금지
- 데이터 흐름 꼬이지 않게 단방향 상태 흐름 유지
- 고위험 작업은 재인증 구조 포함
- 부분 복구 금지, 전체 시점 복구만 허용
- “문제 해결”보다 “이전 상태 복구” 철학 유지
- 물리적으로 말이 되게 설계: 데이터 원인-결과 관계, 로그 순서, 결제 집계 시점, 복구 덮어쓰기 순서가 논리적으로 맞아야 한다

==================================================
1. 구현 범위
==================================================

이번 작업은 “슈퍼관리자 콘솔”을 꼼꼼하게 구현하는 작업이다.

핵심 목적:
- 플랫폼 전체 상태 감시
- 리스크 선제 감지
- 로그 추적
- 백업/복구 관리
- 매출/지출 통합 확인
- 공지/광고/보안/고객센터 운영 통제

슈퍼관리자는 평소 실무를 대신 처리하는 사람이 아니다.
하지만 리스크가 보이면 먼저 조치할 수 있어야 하고,
복구 실패나 보안 문제처럼 비상 상황에서는 즉시 개입할 수 있어야 한다.

==================================================
2. 첫 화면 대시보드 구현
==================================================

슈퍼관리자 콘솔 첫 화면은 아래 구조로 구현한다.

[A. 상단 요약 카드]
최소 다음 카드들을 상단에 배치하고 실데이터 연동:
- 전체 운영자 수
- 활성 운영자 수
- 승인 대기 수
- 결제 이슈 수
- 저장 위험 수
- 삭제 예정 수

각 카드 클릭 시 관련 상세 페이지로 이동 가능하게 구현.

[B. 오늘 처리할 일]
이 영역은 “지금 슈퍼관리자가 눌러서 조치 가능한 항목만” 표시한다.
단순 통계가 아니라 actionable item 이어야 한다.

표시 항목:
- 결제 실패
- 저장공간 초과 또는 임계치 초과
- 자동삭제 예정
- 정책 미확인
- 보안 이벤트
- 고객센터 미처리
- 로그인 이상 감지
- 외부 서비스 오류 (PG / 이메일 / 푸시 / 기타 연동)
- 데이터 불일치 (회원 수/매출/로그 누락 등)

각 항목 우측 내림 버튼 또는 상세 버튼 구현:
- 클릭 시 해당 목록 상세 페이지로 이동
- 목록 페이지에서 개별 항목 클릭 시 상세 보기 가능

[C. 리스크 요약]
리스크는 현재 사고뿐 아니라 “곧 문제 될 가능성”도 보여줘야 한다.

표시 항목:
- 결제 리스크
- 저장공간 리스크
- 삭제 예정
- 정책 미확인
- SLA 초과
- 보안 이벤트
- 시스템 리스크 (외부 API/PG/이메일 장애)
- 데이터 리스크 (불일치/백업 이상/복구 실패 가능성)
- 운영 리스크 (장기 미접속 운영자/비활성 운영자)

각 항목은 심각도 배지(낮음/보통/높음/긴급) 표시.

[D. 최근 감사 로그]
슈퍼관리자가 실행한 작업 + 시스템 자동 작업 이력을 표시.
예시:
- 승인 처리
- 결제 실패 처리
- 스냅샷 생성
- 복구 시도
- 자동삭제 예약
- 기능 플래그 변경
- 보안 정책 변경

로그 표시 필드:
- createdAt
- actor
- target
- summary
- severity
- category

actor 구분:
- system
- super_admin

최근 로그 클릭 시 상세 보기 가능.

==================================================
3. 좌측/하단/메인 메뉴 구조 재편
==================================================

기존 기능 나열식 메뉴를 운영 흐름 기준으로 재정리한다.

메뉴 순서:
1) 운영 관리
2) 매출 분석
3) 비용 / 지출
4) 데이터 관리
5) 고객센터
6) 정책 / 컴플라이언스
7) 보안 / 통제
8) 시스템 상태
9) 광고 관리

각 메뉴는 실제 페이지와 연결하고 빈 화면 금지.

==================================================
4. 운영 관리 페이지
==================================================

포함 항목:
- 운영자 관리
- 승인 관리
- 운영 상태
- 수영장 관리자 로그
- 운영자별 상태 확인

[승인 관리]
승인은 하나의 공통 승인 개념이 아니라 각 버튼/기능별 승인 상태값으로 관리.
상태값:
- pending
- approved
- rejected

반드시 구현:
- 승인 대기 목록
- 승인 처리
- 거절 처리
- 거절 후 재승인 가능
- 거절 후 다시 거절 가능
- 모든 승인/거절/재승인은 이벤트 로그 기록

[거절 팝업]
거절 버튼 누르면 사유 입력창 표시.
- 직접 타이핑 가능
- 비워둬도 거절 가능
저장값:
- status = rejected
- rejectedAt
- rejectedBy
- rejectReason(nullable)

==================================================
5. 매출 분석 페이지
==================================================

중요:
슈퍼관리자 매출은 “수영장별 상세 매출 화면”이 아니라
플랫폼 전체 매출 집계 화면이다.

[필터 탭]
상단 탭 3개 구현:
- 주간
- 월간
- 연간

[집계 기준]
반드시 실제 결제 완료 로그 기반으로만 계산.
포함:
- 신규 결제
- 구독 갱신
- 업그레이드 결제

제외:
- 단순 가입
- 미결제
- 실패 결제

절대 금지:
- 회원 수 기반 추정 매출
- 수기 입력 매출

[표시 항목]
- 해당 기간 누적 매출
- 결제 성공 건수
- 결제 실패 건수
- 신규 구독 수
- 갱신 구독 수
- 해지 수
- 환불 수
- 환불 금액
- 이전 기간 대비 증감률
- 예상 매출 추이

[기간 동작]
- 주간: 해당 주 시작일부터 현재까지
- 월간: 해당 월 1일부터 현재까지
- 연간: 해당 연도 1월 1일부터 현재까지

==================================================
6. 비용 / 지출 페이지
==================================================

매출과 별도 메뉴로 구현.
플랫폼 운영 지출을 확인할 수 있어야 한다.

표시 항목:
- 데이터베이스 비용
- 스토리지 비용
- 트래픽 비용
- PG 수수료
- 세금 추정
- 기타 운영비
- 월 지출 총합
- 주간/월간/연간 필터
- 순이익 = 매출 - 지출

가능하면 실데이터 기반 계산, 불가하면 명확한 mock source 분리 후 추후 교체 가능한 구조로 구현.

==================================================
7. 데이터 관리 페이지
==================================================

포함:
- 저장공간 현황
- 사용량
- 백업 / 스냅샷
- 복구
- 삭제 정책
- 이벤트 로그
- 백업 메타

[백업 규칙]
- 자동 백업 1시간 단위
- 위험 작업 전 즉시 백업
- 수동 백업 버튼 제공

[스냅샷 정책]
보관 규칙:
- 최근 7일: 1시간 단위
- 최근 30일: 하루 1개
- 최근 90일: 주 1개

스냅샷 이름 규칙:
수영장이름_스냅샷_YYYY-MM-DD_HH-mm

[복구 원칙]
- 부분 복구 금지
- 전체 시점 복구만 허용
- 복구 전 자동 백업 필수
- 현재 데이터는 덮어쓰기 됨

[복구 미리보기 화면]
스냅샷 선택 후 복구 실행 전에 미리보기 화면 필수.
표시 항목:
- 스냅샷 이름
- 생성 시각
- 수영장명
- 복구 대상 데이터 종류
- 현재 데이터 덮어쓰기 경고
- 복구 전 자동 백업 생성 안내

복구 대상 데이터 종류 예시:
- 회원 정보
- 반 배정
- 수업 스케줄
- 출결 데이터
- 일지 데이터
- 설정값
- 권한 관련 데이터

[복구 실패 처리]
복구 실패 시 팝업 표시:
- 실패 안내 문구
- 실패 사유
- 관리자 문의 버튼
- 실패 로그 자동 저장

실패 안내 문구:
“복구에 실패했습니다. 관리자에게 문의해 주세요.”

[관리자 문의 버튼]
- 고객센터 문의 작성 화면으로 바로 연결
- 문의 유형은 자동으로 “복구 문의” 선택
- 중요도는 최고 우선순위
- 고객센터 목록에서 비상/긴급 표시

[복구 실패 개입 원칙]
- 수영장 관리자 스스로 복구 시도 실패 시
- 슈퍼관리자가 개입 가능
- 복구 실패 관련 감사 로그, 에러 로그, 문의 티켓 모두 연결

==================================================
8. 고객센터 페이지
==================================================

일반 문의만 두지 말고 운영 분류 필요.

문의 분류:
- 일반 문의
- 결제 문의
- 환불 문의
- 운영 문의
- 복구 문의
- 보안 문의

특히 “복구 문의”는 일반 문의와 구분해서 표시:
- 긴급/비상 뱃지
- 가장 높은 우선순위
- 별도 정렬 상단 노출
- 슈퍼관리자가 빠르게 확인 가능하게 처리

복구 실패 후 문의 작성으로 진입하면
- 제목/내용 자동 기본값 일부 채우기 가능
- 실패 사유 로그와 연결 가능하면 연결

슈퍼관리자 계정이 털리면 끝이라는 전제로 구현.

포함:

- OTP / 2차 인증
- 비밀번호 재입력 정책
- 세션 관리
- 로그인 실패 기록
- IP 기록
- 기기 기록
- 이상 로그인 감지
- 외부 서비스 연결 상태
- 기능 플래그
- 읽기 전용 제어
- 킬스위치

[고위험 작업 재인증]
다음 작업 전에는 반드시 재인증:
- 복구 실행
- 삭제 실행
- 권한 변경
- 기능 플래그 변경
- 킬스위치 실행
- 민감 정책 변경

재인증 방식:
- OTP
- 비밀번호 재입력
- 세션 검증

[기능 플래그]
- 전체 기능 ON/OFF
- 운영자별 예외 적용 가능
- 위험도 표시
- 변경 사유 입력
- 감사 로그 기록

[읽기전용 제어]
3단계:
- 플랫폼 전체
- 운영자별
- 기능별

==================================================
12. 광고 관리 페이지
==================================================

현재는 “준비 상태”로 만들고 실제 학부모 화면 광고 노출은 아직 하지 않는다.

슈퍼관리자 페이지에는 구현:
- 광고 등록
- 이미지 업로드
- 제목 입력
- 설명 입력
- 링크 설정
- 노출 기간 설정
- 상태값 관리

상태값 예시:
- scheduled
- inactive
- active

중요:
- 지금은 슈퍼관리자 콘솔에서만 관리 기능 구현
- 학부모 화면에는 광고 슬롯 노출하지 말 것
- 향후 광고 사용 예정 구조만 마련

==================================================
13. 공지 팝업 시스템 구현
==================================================

슈퍼관리자가 대상별 공지를 등록할 수 있어야 한다.

대상:
- 수영장 관리자
- 선생님
- 학부모
- 전체

[동작 방식]
앱 실행 후 로그인 흐름에서 최신 공지 1개를 팝업으로 먼저 띄운다.

버튼:
- 확인
- 다시 보지 않기

규칙:
- 공지는 강제 확인 구조
- 팝업이 떠 있으면 다음 화면 진입 불가
- 닫기만 하고 넘기는 구조 금지
- 반드시 확인 또는 다시 보지 않기 처리 후 진행
- 최신 공지 1개만 노출
- 새 공지 등록 시 이전 공지를 덮어쓰기
- 다시 보지 않기를 누른 공지는 다시 노출 안 함
- 이후 새 공지를 등록하면 그 새 공지는 다시 노출

공지 등록 필드:
- 제목
- 내용
- 대상
- 강제 여부(기본 true 가능)
- 생성일시

==================================================
14. SMS 정책 반영
==================================================

중요:
- SMS 판매 기능 제거
- SMS 크레딧 구조 제거

하지만 반드시 유지:
- 수영장 관리자가 자기 휴대폰 메시지 앱으로 발송한 문자 기록 로그 수집/보관

우리는 문자 서비스 사업자가 아니지만
복구/감사/추적을 위해 문자 발송 기록 로그는 저장해야 한다.

이 로그는 데이터 관리, 감사 로그, 복구 추적과 연결할 수 있게 구조화.

==================================================
15. 삭제 / 보존 정책
==================================================

핵심:
삭제는 고위험 작업이다.

삭제 허용 조건:
- 해지 확정
- 정책 동의
- 유예 완료

자동삭제 금지 조건:
- 결제 실패만으로 자동삭제 금지
- 저장공간 초과만으로 자동삭제 금지

삭제 방식:
- 전체 데이터 삭제
- 기간 지정 삭제
- 항목별 삭제
- 이벤트 로그 필수
- 재인증 필수

기존 킬스위치 및 영구삭제 정책과 논리 충돌 없게 구현.

작업 후 반드시 직접 점검하고 결과를 보고하라.
“완료” 한 줄로 끝내지 마라.

- 고위험 작업에서 재인증 걸리는지 여부
- 광고 등록/수정/상태 변경 여부
- 문자 발송 기록 로그 저장 여부
- 이벤트 로그에 필수 필드 누락 없는지 여부
- Invalid Date / NaN / undefined 저장 여부 확인
- 빈 데이터일 때 깨지지 않는지 여부
- 모바일 레이아웃 깨짐 여부

19. 최종 목적
==================================================

이번 작업의 목적은 예쁜 관리자 화면이 아니라,
“운영이 망가지지 않는 슈퍼관리자 콘솔”을 만드는 것이다.

- 강한 보안
- 공지 통제
- 긴급 복구 문의 우선 처리

겉모양보다 데이터 논리와 운영 철학을 우선해서 정리하라.
