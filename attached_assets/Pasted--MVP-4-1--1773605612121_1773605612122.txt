지금부터 스윔노트 MVP 관리자/운영앱을 기능 연결 중심으로 전면 수정한다.

가장 중요한 원칙은 아래 4가지다.

1. 화면만 만들지 말고 모든 버튼, 이름 클릭, 상태 변경, 연결 버튼이 실제 데이터와 연결되어 작동해야 한다.
2. 각 작업 후 반드시 직접 테스트하고, 테스트 결과를 체크리스트와 로그 형태로 기록해서 보여줘야 한다.
3. 테스트하지 않은 기능은 완료로 표시하지 마라.
4. 구현 후 “완료”라고 말하려면 반드시 체크 기록, 테스트 경로, 결과를 함께 제출해야 한다.

--------------------------------------------------
[개발 최우선 원칙]
--------------------------------------------------

- 앱 안에 보이는 모든 버튼은 실제 기능과 연결할 것
- 이름 클릭 시 상세정보가 열릴 것
- 상세정보 종료 시 반드시 진입했던 원래 목록으로 복귀할 것
- 어느 화면에서 수정/연결/상태변경을 하든 동일한 실제 데이터에 반영할 것
- 관련 목록/상세/다른 탭까지 반영되도록 연결할 것
- 삭제는 즉시 완전삭제 금지
- 자동 로그인은 로그아웃 또는 앱 삭제 전까지 유지
- 역할 전환은 재로그인 없이 같은 계정 세션 안에서 처리
- 작업마다 반드시 테스트 후 체크 기록 남길 것

--------------------------------------------------
[필수 구현 항목]
--------------------------------------------------

1. 수업 화면 전면 수정
- 기존 반 카드 목록형 폐기
- 주간 스케줄보드형으로 변경
- 기본 진입: 주간 스케줄보드
- 흐름:
  주간 스케줄보드
  -> 요일+시간 선택
  -> 선생님 선택 화면
  -> 반 학생 수업관리 상세
- 선생님 선택 화면에는 "스케줄보드로 이동" 버튼 필수
- 반 상세에는 "스케줄보드로 이동", "선생님 선택" 버튼 필수
- 출결은 반 관리 안에서만 진입
- 레인 구분 없음
- 선생님 색상 구분은 추후

2. 사람 탭 구조 통일
- 학생 / 학부모 / 선생님 / 승인 탭
- 공통 흐름:
  목록 -> 이름 클릭 -> 상세정보 -> 종료 -> 원래 목록 복귀
- 학생 탭: 이름 검색 가능
- 학부모 탭: 이름 검색 가능
- 승인 탭: 이름 검색 가능
- 승인 목록은 학생 이름 우선 표시

3. 승인 관리
- 학부모 승인 / 선생님 승인 구분
- 대기 / 승인 / 거절 상태별 목록
- 이름 클릭 시 상세정보 표시
- 상세에서 상태 변경 가능:
  대기 / 승인 / 거절 버튼 3개 모두 항상 표시
- 상태 변경 후 해당 상태 목록으로 자동 이동
- 상세 종료 시 현재 상태 목록으로 복귀
- 승인 시 학생/학부모 자동 생성 및 연결
- 반배정은 승인 단계가 아니라 회원등록 완료 후 진행

4. 이름 클릭 전역 통일
- 학생 / 학부모 / 선생님 이름이 보이는 모든 곳에서 이름 클릭 시 상세정보 표시
- 적용 화면:
  사람 탭, 승인 관리, 반 관리, 미배정 목록, 학부모 관리, 선생님 관리, 삭제/복구 센터 등
- 상세 종료 시 원래 목록 복귀

5. 학부모 관리
- 목록에서 이름 클릭 시 학부모 상세정보
- 학부모 상세에서 바로 학생 연결 가능
- 연결 기능은 공통 로직으로 구현
- 연결 완료 후 안내 문구:
  "이제부터 스윔노트를 이용할 수 있습니다"

6. 선생님 관리
- 계정 목록에서 이름 클릭 시 선생님 상세정보
- 상세 종료 시 계정 목록 복귀
- 활성 상태 선생님 계정은 선생님 모드와 연결되는 기준 계정으로 유지

7. 관리자/선생님 역할 전환
- 한 계정이 관리자+선생님 역할 동시 가능
- 관리자 화면: 선생님 전환 버튼 + 로그아웃
- 선생님 화면: 관리자 이동 버튼 + 선생님 설정
- 버튼은 숨기지 말 것
- 권한 없는 경우:
  "이동 권한이 없습니다" 알림만 표시
- 재로그인 없이 역할 전환

8. 자동 로그인
- 로그아웃 또는 앱 삭제 전까지 로그인 유지
- 앱 종료 후 다시 실행해도 로그인 유지

9. 설정/로그
- 관리자 설정 추가
- 선생님 설정 추가
- 활동 로그는 관리자만 조회
- 관리자/선생님 설정 변경 이력 모두 관리자 로그에 기록
- 기록 항목:
  수정자명, 권한, 메뉴, 항목, 수정 전, 수정 후, 수정 시각

10. 삭제/복구 구조
- 어떤 삭제 버튼도 즉시 완전삭제 금지
- 1차 삭제:
  삭제 로그 기록
  복구 가능한 상태로 보관
  휴지통처럼 마지막 위치 정보 저장
- 삭제/복구 센터는 복잡한 카테고리 분리보다 휴지통 구조로 만들 것
- 검색 기능 추가
- 원래 자리 복구 가능
- 최종 삭제 권한은 관리자만
- 관리자 로그기록 삭제는 비밀번호 재입력 필요

11. 학생 상세
- 직접 반배정 변경 버튼보다
  "수업 상태" 같은 버튼으로 반배정 페이지 이동

12. 키패드 전역 처리
- 모든 입력창/검색창/폼에 공통 적용
- 취소 버튼 -> 키패드 닫힘
- 빈 영역 터치 -> 키패드 닫힘
- 아래로 드래그 -> 키패드 닫힘
- 포커스 해제 시 자동 닫힘
- 개별 화면별이 아니라 공통 로직으로 처리

13. 학부모 연결 요청 시트
- 검색 가능
- 취소 시 키패드 닫힘
- 드래그 시 키패드 닫힘
- 실제 연결 데이터와 반영 연동
- 연결 완료 시 목록/상세/학생 정보에 즉시 반영

14. 버튼 검수 원칙
- 버튼이 보이면 반드시 클릭 가능해야 함
- 클릭 시 실제 기능 실행
- 데이터 수정 반영
- 관련 화면까지 연동
- 종료 시 원래 화면 복귀

테스트하지 않은 것은 절대 완료로 표시하지 마라.

각 기능 구현 후 아래 형식으로 반드시 기록해라.

[체크 기록 형식]

구현 여부: 완료 / 미완료 / 부분완료
테스트 경로:
예) 사람 탭 > 회원 목록 > 서태웅 클릭 > 학생 상세 열림 > 닫기 > 회원 목록 복귀

입력 데이터:
예) 테스트 계정, 테스트 회원명, 테스트 상태값

예) 이름 클릭 시 학생 상세가 열리고 닫으면 회원 목록으로 돌아와야 함

실제 결과:
예) 정상 동작 / 오류 발생 / 일부만 동작

예) 상세에서 수정한 보호자 연락처가 목록/상세/연결 화면에 모두 반영되는지 여부

로그 기록:
예) 설정 수정 시 관리자 활동 로그에 남았는지 여부

판정:
통과 / 실패

--------------------------------------------------

최종 보고할 때는 반드시 아래 순서로 제출해라.

1. 이번 작업에서 구현한 기능 목록

--------------------------------------------------
[추가 강제 규칙]
--------------------------------------------------

- 테스트 없이 완료라고 말하지 마라
- 체크 기록 없이 완료라고 말하지 마라
- 버튼 연결 누락이 있으면 바로 목록으로 보고해라
- 기능이 완전하지 않으면 "부분완료"로 표시해라
- UI보다 데이터 연결과 작동 검수를 우선해라
- 가능한 경우 테스트용 더미데이터를 넣고 직접 눌러가며 검수해라
- 복귀 동선, 상태 변경, 연결 반영, 삭제/복구, 권한 차단, 자동 로그인 유지 여부를 반드시 각각 확인해라

이제 위 명령을 기준으로 바로 개발을 시작해라.
한 번에 전부 끝냈다고 하지 말고,
작업 단위를 나눠서 구현 -> 테스트 -> 체크 기록 제출 순서로 진행해라.
첫 번째 작업 묶음부터 시작하고 체크 기록을 먼저 보여줘.
