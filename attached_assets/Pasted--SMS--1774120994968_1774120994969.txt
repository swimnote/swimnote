현재 목표:
기능 추가를 무작정 늘리지 말고,
수익 플랫폼 기준으로 필요한 항목을 재분류하고,
보안/인증/SMS/과금유도 흐름을 보강하되,
앱이 무한루프/크래시 없이 안정적으로 작동하도록 구현한다.

절대 원칙:
- 데이터 보호 우선
- 과금 유도형 우선
- 삭제는 마지막 단계
- 결제 실패만으로 자동삭제 금지
- 기능 추가보다 안정성 우선
- 테스트 없이 완료했다고 하지 말 것

==================================================
[1. 메뉴 재분류]
==================================================

슈퍼관리자 메뉴를 4그룹으로 재구성:

1) 운영 관리
- 운영자 관리
- 구독·결제 관리
- 구독 상품 설정
- 저장공간 관리
- 저장공간 정책

2) 보호/통제
- 데이터·킬스위치
- 백업/복구/스냅샷
- 읽기전용 제어
- 기능 플래그

3) 정책/지원
- 정책·컴플라이언스
- 고객센터
- 초대/SMS 관리

4) 보안/감사
- 운영 로그·감사
- 장애·리스크 센터
- 슈퍼관리자 보안관리

아이콘이 많아도 괜찮다.
대신 그룹 헤더와 구역 분리로 복잡도를 낮춘다.

==================================================
[2. 수익/과금 흐름 재정리]
==================================================

현재부터 아래 흐름으로 통일:

플랜 사용량 접근
-> 초과 예상 안내
-> 추가 과금/업그레이드 유도
-> 경고
-> 업로드 차단 예정
-> 업로드 차단
-> 필요 시 read-only
-> 해지 확정 후에만 삭제 예약 가능

중요:
- 결제 실패만으로 자동삭제 예약 금지
- 저장공간 초과만으로 자동삭제 예약 금지
- 자동삭제는 해지 확정 + 정책 동의 + 유예 완료 후에만

[3. 저장공간 정책 수정]
==================================================

현재:
95% 초과 즉시 업로드 차단 느낌이 강함

수정:
- 80%: 경고
- 95%: 차단 예정 + 추가용량 구매 유도
- 100%: 업로드 차단

추가:

- "추가 용량 구매" CTA
- "상위 플랜 업그레이드" CTA
- 예상 초과 비용 표시
- 관리자 override 기능 추가

==================================================
[4. 자동삭제 정책 수정]
==================================================

자동삭제는 아래 조건일 때만 허용:

- 운영자가 해지 확정
- 삭제 정책 동의 완료
- 삭제 예정 고지 발송 완료
- 유예 시간 경과
- 필요 시 슈퍼관리자 승인

금지:
- 결제 실패만으로 삭제
- 단순 미결제로 삭제
- 저장공간 초과만으로 삭제

추가:
- 삭제 사유 구분값
- 해지 사유 / 미결제 사유 / 수동 삭제 사유 분리
- 삭제 실행 전 강제 스냅샷 생성
- 삭제 실행 전 2단계 확인

==================================================
[12. 앱 안정성 규칙]
==================================================

매우 중요:
- 계산값은 useMemo로만 처리
- effect 안에서 setState/store update 반복 금지
- derived state를 다시 store에 저장 금지
- 위험 작업은 모두 확인 모달 + 사유 입력
- 모든 리스트는 loading/empty/error 분리
- 모든 날짜 formatDateSafe 사용
- 모든 텍스트 safeText 사용

절대:
- Maximum update depth 다시 발생하면 안 됨
- undefined / Invalid Date 노출 금지
- 탭 전환 시 state 루프 금지
