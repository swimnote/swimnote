[스윔노트 — 수영장 관리자 자동가입 + 관리자선생님 자동생성 지시문]

이번 작업은 수영장 관리자 가입 흐름을 승인 대기 없이 즉시 가입 구조로 바꾸고,
가입과 동시에 선생님 계정도 자동으로 생성되게 만드는 작업이다.

핵심 목표:
- 수영장 관리자는 슈퍼관리자 승인 없이 즉시 가입
- 가입 즉시 수영장 생성/연결 완료
- 동시에 선생님 계정 자동 생성
- 선생님 모드에 들어가면 관리자 이름 기반의 선생님 계정이 바로 보이게 할 것

━━━━━━━━━━━━━━━━━━
1. 가입 정책 변경
━━━━━━━━━━━━━━━━━━

기존:
- 수영장 관리자 가입
- 승인 대기
- 슈퍼관리자 승인 후 활성화

변경 후:
- 수영장 관리자 가입
- 수영장 정보 입력
- 즉시 가입 완료
- 즉시 로그인 가능
- 즉시 관리자 모드 사용 가능

즉,
👉 수영장 관리자 가입에는 승인 대기 단계가 없어야 한다

금지:
- "승인 대기중" 화면 노출
- 슈퍼관리자 승인 필요 구조
- 가입 후 비활성 상태 유지

━━━━━━━━━━━━━━━━━━
2. 수영장 관리자 가입 완료 시 자동 처리
━━━━━━━━━━━━━━━━━━

수영장 관리자 가입 성공 직후 아래를 자동으로 처리:

1. 관리자 계정 생성
2. 수영장 레코드 생성
3. 관리자 계정과 수영장 연결
4. 관리자 계정 roles 설정
   - ["pool_admin", "teacher"]
5. 선생님 엔티티/선생님 프로필 자동 생성
6. 관리자 모드/선생님 모드 모두 진입 가능 상태로 완료

즉,
👉 관리자 가입 = 관리자 생성 + 선생님 생성까지 한 번에 끝

━━━━━━━━━━━━━━━━━━
3. 승인 상태 처리
━━━━━━━━━━━━━━━━━━

수영장 관리자 계정은 생성 시점부터 바로 활성 상태로 만든다.

예:
- is_activated = true
- status = active

수영장도 즉시 운영 가능 상태로 생성

금지:
- pending
- waiting_approval
- requested
- inactive

단,
일반 선생님 가입 신청/학부모 가입 승인 흐름은 기존 정책 유지 가능
이번 작업은 "수영장 관리자 가입"만 즉시 승인으로 변경

━━━━━━━━━━━━━━━━━━
4. 관리자 동시 선생님 등록 규칙
━━━━━━━━━━━━━━━━━━

수영장 관리자 가입 시 선생님 계정 또는 선생님 프로필을 동시에 만든다.

표시명 규칙:
- 관리자 모드 표시명: 수영장이름 + 관리자
- 선생님 모드 표시명: 관리자선생님

예:
- 관리자 모드: 토이키즈 관리자
- 선생님 모드: 관리자선생님

중요:
- 실명선생님이 아니라 이번 정책에서는 관리자선생님으로 단순화
- 선생님 목록/선생님 홈/선생님 프로필에 즉시 보여야 함

━━━━━━━━━━━━━━━━━━
5. 선생님 모드 연동
━━━━━━━━━━━━━━━━━━

관리자 가입 후 선생님 모드에 진입하면:

- 별도 가입 신청 없음
- 별도 승인 없음
- 이미 생성된 관리자선생님 계정/프로필이 보여야 함
- 선생님 홈, 스케줄, 학생처리 화면 진입 가능해야 함

즉,
👉 관리자는 가입 즉시 선생님 모드까지 쓸 수 있어야 한다

━━━━━━━━━━━━━━━━━━
6. 역할 구조
━━━━━━━━━━━━━━━━━━

수영장 관리자 가입 계정 roles:

- pool_admin
- teacher

일반 선생님 가입 계정 roles:

- teacher

차이:
- 관리자 가입자는 관리자↔선생님 전환 가능
- 일반 선생님은 선생님만 가능

기존 관리자/선생님 전환 버튼 규칙 유지:
- pool_admin 역할 있으면 전환 버튼 표시
- 일반 teacher만 있으면 관리자 전환 버튼 없음

━━━━━━━━━━━━━━━━━━
7. 가입 API 수정
━━━━━━━━━━━━━━━━━━

수영장 관리자 가입 API 또는 register 로직에서 아래 분기 추가:

관리자 가입인 경우:
- 즉시 activated 처리
- 수영장 생성
- roles = ["pool_admin", "teacher"]
- teacher profile/entity 자동 생성
- 승인대기 응답 금지

응답 예시 방향:
- success: true
- activated: true
- roles: ["pool_admin", "teacher"]
- pool_created: true
- teacher_profile_created: true

금지:
- success인데 waiting approval 반환
- 가입은 됐지만 teacher 생성 안 됨

━━━━━━━━━━━━━━━━━━
8. DB 저장 규칙
━━━━━━━━━━━━━━━━━━

반드시 superAdminDb에만 저장

저장 대상 예:
- users
- swimming_pools
- user_pools 또는 연결 테이블
- teacher 관련 테이블 / teacher profile
- role/permission 관련 테이블

중요:
- pool DB 저장 금지
- 승인대기 레코드 남기지 말 것
- 관리자 가입과 teacher 자동생성이 트랜잭션 또는 원자적 흐름으로 처리되게 할 것

권장:
- 중간 실패 시 롤백
- 관리자만 생성되고 teacher 안 생기는 반쪽 상태 금지

━━━━━━━━━━━━━━━━━━
9. 가입 후 화면 흐름
━━━━━━━━━━━━━━━━━━

가입 완료 후 흐름:

1. 수영장 관리자 회원가입 완료
2. 자동 로그인 또는 역할 선택 화면 진입
3. 관리자 카드 + 선생님 카드 둘 다 보임
4. 관리자 모드 정상 진입
5. 선생님 모드 정상 진입
6. 선생님 화면에 관리자선생님 표시

승인대기 화면으로 가면 실패

━━━━━━━━━━━━━━━━━━
10. 기존 승인 로직 분리
━━━━━━━━━━━━━━━━━━

이번 작업에서 분리할 것:

- 수영장 관리자 가입 = 자동 승인
- 일반 선생님 가입 신청 = 승인 흐름 유지 가능
- 학부모 가입 = 기존 정책 유지

즉,
승인 시스템을 전부 없애는 게 아니라
👉 수영장 관리자 가입만 승인 예외 처리

━━━━━━━━━━━━━━━━━━
11. 테스트
━━━━━━━━━━━━━━━━━━

반드시 실제로 확인:

1. 수영장 관리자 가입
2. 수영장 정보 입력
3. 승인대기 화면이 안 뜨는지 확인
4. 즉시 활성 계정 생성 확인
5. swimming_pools 생성 확인
6. roles = ["pool_admin", "teacher"] 확인
7. 선생님 엔티티/프로필 자동 생성 확인
8. 관리자 모드 진입 확인
9. 선생님 모드 진입 확인
10. 선생님 화면에 관리자선생님 표시 확인
11. 일반 선생님 가입은 기존 승인 구조 유지되는지 확인
12. 모든 데이터가 superAdminDb에만 저장되는지 확인

━━━━━━━━━━━━━━━━━━
12. 보고 형식
━━━━━━━━━━━━━━━━━━

반드시 포함:

- 수정 파일 경로
- 가입 API 수정 내용
- 승인대기 제거 방식
- 관리자 자동 활성화 결과
- 수영장 자동 생성 결과
- teacher 자동 생성 결과
- roles 저장 결과
- 관리자/선생님 화면 진입 결과
- superAdminDb 저장 확인 결과

“가입 되게 수정함”만 쓰지 말고
가입 → 활성화 → 수영장 생성 → teacher 생성 → 관리자/선생님 진입까지 실제 결과를 같이 제출할 것

━━━━━━━━━━━━━━━━━━
